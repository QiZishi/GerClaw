# 0029 — 五大处方与用药审查生产闭环

> 创建：2026-07-16 | 优先级：P0 | 状态：待医学内容确认 | 前置：0021 Runtime Permission/HITL、0023 CGA、0025 真实身份/RBAC

## 1. 目标

将输入框的“五大处方”和医生端“用药审查”从当前安全降级入口，升级为服务端持久化、规则可追溯、受 Runtime Harness 治理的临床建议 workflow。系统只生成“待医生审核的 AI 辅助建议”，绝不生成可自行执行的电子处方或确定性诊断。

## 2. 已审计事实与阻断

- 当前 `apps/mvp` 在入口被 `ChatArea` 显式安全降级；不可达旧本地 LLM 分支仍存在，见 `TD-006`，不得重新接线或恢复为前端模拟报告。
- `输入输出/五大处方报告模板.md` 不是空白/通用报告 schema，而是心脏康复个案原文，含具体患者生理数据和处方药名、剂量、频次（例如降压药、他汀、抗血小板药、镇静抗抑郁药）。不得将它作为任何其他患者的默认内容、prompt few-shot 示例或自动填充模板。
- `TD-003` 记录 DDI/Beers/剂量数据尚待医学团队审核；当前后端没有受版本控制、带来源与审核记录的规则集，因此不得对用户宣称已完成 DDI、Beers 或剂量结论。
- 患者生成草案后的医生编辑、批准、拒绝与患者授权依赖 0025 的真实账号/RBAC/授权链路。访客模式可体验低风险信息收集，但不能把药物修改标记为可执行。

## 3. 需要确认的医学内容边界

在启用任何真实处方/用药审查结果前，必须由产品负责人或医学审核人提供并确认：

1. 去标识、通用且版本化的五大处方 JSON schema 与示例（不得带真实个案、具体处方或剂量）。
2. DDI、Beers、剂量规则的来源、许可、版本、适用人群、审核人和复审日期；未审核规则只能用于内部测试，不能对用户输出临床结论。
3. 医生批准的角色、患者授权、可编辑字段、药物建议的高风险审批语义和审计保留期限。

在以上确认前，不得以“完成”“已审查”“可执行处方”或静态案例结果展示任何能力。

## 4. 可安全推进的阶段

### 阶段 A — 结构和治理（不输出临床建议）

- 建立独立 `modules/prescription/` 与 `modules/medication_review/` 的 Protocol、Pydantic/Zod 契约、版本字段、权限、预算、trace 和错误模型。
- 实现最小化患者信息收集与缺失字段状态；所有用户输入经 PHI/安全策略处理，不直接写入日志或向量库。
- 处方草案只可为结构校验后的空/占位章节，清晰标为“等待医学规则与医生审核”，不得编造循证来源或药物建议。
- [x] 删除 `ChatArea` 的旧本地 LLM 处方/审查分支，防止未来误接线。

#### 阶段 A 已完成的安全收口（2026-07-16）

- 已从 `apps/mvp/src/components/layout/ChatArea.tsx` 物理删除旧本地 LLM 请求、五大处方/用药审查 prompt、前端报告生成与本地信息抽取（606 行）。普通聊天仅调用受 Runtime Harness 保护的 `streamAgentChat`。
- 入口继续明确安全降级：临床 workflow 尚未接入真实后端时，只提示用户可继续进行一般健康咨询；不生成报告、不宣称审查完成，也不发送旧 `/api/llm/chat` 请求。
- 变更后实际执行：`npm run lint`、`npm run build`、`npm run test:audio`（6/6）、`npm run test:document`（5/5）、`npm run test:search`（4/4）。
- 生产构建浏览器复验（`http://127.0.0.1:3048`）：患者端点击“五大处方生成”、医生端点击“用药审查”均仅显示“生产工作流正在接入”的状态提示；未出现处方/审查报告或右侧产物。CGA 的“保存并休息”后也正常回到欢迎页。

### 阶段 B — 经审核的确定性用药规则

- 载入经审核、可追溯的规则集后，按名称归一化、DDI、Beers、重复用药、剂量校验的确定性顺序产生版本化风险事实；LLM 只能解释事实，不能改写命中结果。
- 规则命中和无规则覆盖均需显式展示；高风险命中 fail closed 并进入 Runtime HITL。

### 阶段 C — 受控生成与批准

- RAG 引用和结构化模型输出须通过格式、内容、循证与安全校验；失败不产出报告。
- 仅在真实登录、患者授权和医生批准完成后可向患者发布；编辑、批准、拒绝、导出与版本差异均写审计 Trace。

## 5. 验收标准

- [x] 不存在可从前端触发的本地模拟处方/用药审查报告分支。
- [ ] 每个 API、模型输出、规则输入和导出都由严格 schema 与版本绑定保护。
- [ ] 未审核规则或缺失医生批准时，系统不输出药物调整或可执行处方。
- [ ] 已审核规则后，DDI/Beers/剂量/重复用药的命中、来源、版本和测试用例可追溯；LLM 不得覆盖确定性结果。
- [ ] 处方 JSON 严格匹配经确认的通用模板，所有医学建议有可追溯证据与统一免责声明。
- [ ] 真正的患者/医生授权、HITL、撤销、审计、浏览器 E2E、≤10 并发和独立审阅完成后，才可标记为已交付。

## 6. 下一步

等待第 3 节的医学内容与审批边界确认；确认后从阶段 A 开始，按小变更集实现并验证。当前继续维持前端安全降级，不以旧个案模板或未审核规则生成报告。
