# 0029 — 五大处方与用药审查生产闭环

> 创建：2026-07-16 | 优先级：P0 | 状态：阶段 A 部分完成，待医学内容确认 | 前置：0021 Runtime Permission/HITL、0023 CGA、0025 真实身份/RBAC

## 1. 目标

将输入框的“五大处方”和医生端“用药审查”从当前安全降级入口，升级为服务端持久化、规则可追溯、受 Runtime Harness 治理的临床建议 workflow。系统只生成“待医生审核的 AI 辅助建议”，绝不生成可自行执行的电子处方或确定性诊断。

## 2. 已审计事实与阻断

- 当前 `apps/mvp` 在入口被 `ChatArea` 显式安全降级；不可达旧本地 LLM 分支仍存在，见 `TD-006`，不得重新接线或恢复为前端模拟报告。
- `docs/references/五大处方报告模板.md` 已提供五类处方和康复字段的通用结构，同时保留心脏康复原始案例说明。后端仅可采用其类别/字段结构；具体患者生理数据、药名、剂量、频次不得作为其他患者默认内容、prompt few-shot 示例或自动填充模板。
- `TD-003` 仍记录 Beers 和广覆盖 DDI/剂量数据尚待医学团队审核。2026-07-17 已基于本地可校验指南加入受版本控制的**有限** DDI/剂量规则集；它只能输出“来源可追溯、待临床治理批准”的规则命中，不能宣称完成 Beers 或全量用药审查。
- 患者生成草案后的医生编辑、批准、拒绝与患者授权依赖 0025 的真实账号/RBAC/授权链路。访客模式可体验低风险信息收集，但不能把药物修改标记为可执行。

## 3. 需要确认的医学内容边界

将草案升级为正式患者发布或可执行临床结论前，必须由产品负责人或医学审核人提供并确认：

1. 去标识、通用且版本化的五大处方 JSON schema 与示例（不得带真实个案、具体处方或剂量）。
2. DDI、Beers、剂量规则的来源、许可、版本、适用人群、审核人和复审日期；在此之前，系统只能输出明确标为有限范围、来源可追溯、待临床复核的规则结果，不能声称完整覆盖或正式发布。
3. 医生批准的角色、患者授权、可编辑字段、药物建议的高风险审批语义和审计保留期限。

在以上确认前，不得以“完整审查”“医生已批准”或“可执行处方”表述产物。模型提出的加药、停药、替换或剂量调整候选只有在可解析的证据 ID 绑定于结构化建议字段时才可显示；患者端在整份内容末尾统一提示风险，医生端不以机械拦截取代专业判断。

## 4. 可安全推进的阶段

### 阶段 A — 结构和治理（不输出临床建议）

- 建立独立 `modules/prescription/` 与 `modules/medication_review/` 的 Protocol、Pydantic/Zod 契约、版本字段、权限、预算、trace 和错误模型。
- 实现最小化患者信息收集与缺失字段状态；所有用户输入经 PHI/安全策略处理，不直接写入日志或向量库。
- 处方草案只可为结构校验后的空/占位章节，清晰标为“等待医学规则与医生审核”，不得编造循证来源或药物建议。
- [x] 删除 `ChatArea` 的旧本地 LLM 处方/审查分支，防止未来误接线。

### 阶段 B — 经审核的确定性用药规则

- 载入经审核、可追溯的规则集后，按名称归一化、DDI、Beers、重复用药、剂量校验的确定性顺序产生版本化风险事实；LLM 只能解释事实，不能改写命中结果。
- 规则命中和无规则覆盖均需显式展示；高风险命中 fail closed 并进入 Runtime HITL。

#### 阶段 B 当前落地（2026-07-17，有限规则集，待临床治理批准）

- `modules/medication_review/rules/core-v1.json` 已成为服务端加载、Pydantic 校验的 `medication-rules-v3` 产物。它记录来源标题、定位、本地语料相对路径、SHA-256 和 `source_traceable_pending_clinician_approval` 状态；当前有四份本地来源，包含 30 条精确 DDI、4 条日剂量和 1 条限定 Beers 信号。产物仅保存规则摘要，不复制受版权保护的正文。
- `POST /api/v1/clinical-intakes/{id}/medication-review-draft` 在既有 tenant/actor/session 归属、限流、严格 Pydantic/Zod 契约和 Trace 下，生成确定性的 DDI、瑞舒伐他汀日剂量、显式通用名重复和多重用药核对产物。药物文本不进入 Trace、模型、RAG 或外部服务；Trace 只保留规则版本、发现数量和不可逆请求指纹。
- 禁忌或严重级的确定性命中会与审查产物在同一事务创建 `medication_review` 本人安全提醒；提醒使用固定、可行动但不含药名、剂量、规则文本或来源定位的措辞，不能据此自行改药。风险账本按 active critical、active high、已了解的优先级返回，避免立即复核事项被较新的随访提醒遮蔽。
- 医生端已显示规则版本、具体命中结论、来源定位与语料指纹，并明确未识别药物数。≥75 岁只会把已命中的中等风险升级为严重风险；不会凭年龄推断药物或剂量。
- Beers 不声称已安装完整授权规则表。当前仅有一个本地、来源可追溯且待临床治理批准的老年苯二氮卓—失眠核对信号：只有年龄≥65 岁且命中明确别名时才提示核对适应证，绝不从药物清单推定失眠或禁忌。有限规则未命中明确不等于安全。产物固定要求医师/药师复核，并禁止用户自行开始、停用、替换或调整剂量。
- 定向验证：规则引擎及路由契约测试 12/12、API Ruff/Mypy、MVP lint、受控代理测试 11/11、production build 通过；浏览器医生端已验证表单和禁用态。仍待 Compose 真正请求路径、医生批准/HITL 和医学审核签字后，才可把规则标为可面向患者发布。

#### 五大处方 Runtime 治理收口（2026-07-17）

- `POST /clinical-intakes/{id}/prescription-draft` 已在模型调用前解析 server-owned `prescription@1.0.0` workflow 及 `security.workflow.prescription` 风险档案。该 workflow 接受既有 owner/session 限定的上传资料，但拒绝 Skills；风险档案绑定输入/输出 schema、预算、未可信资料隔离、患者归属、证据 provenance 与模型外发脱敏控制。
- 新增 `GERCLAW_PRESCRIPTION_GENERATION_TIMEOUT_SECONDS=600` 整流程硬预算；它独立于三模型候选各自 180 秒 timeout，允许受限回退而不允许无限运行。Trace 只记录 workflow/version 和原有 PHI-free 元数据。
- 当 `current_medications` 存在时，生成器在模型内容完成后附加独立 `medication-rules-v3` 产物，模型不能创建、覆盖或解释命中。该有限、来源可追溯规则结果会进入五大处方报告，但不等同于完整 DDI/Beers/剂量审查。2026-07-17 合成 Compose 浏览器生成实际命中阿托伐他汀—地高辛 DDI，并展示来源定位与 SHA。初次生成暴露了把输入剂量误判为模型开药的安全过滤缺陷，现已允许如实记录用户提供剂量，并允许模型在结构化、带 evidence ID 的 `recommendations` 中提出开始、停用、加用、减量、增量或替换候选；无证据的自由文本候选会被丢弃并降级为待复核基线。患者端仅在整份内容末尾统一提示风险，医生端不机械拦截有证据候选；定向 API 回归为 `48 passed`。
- 2026-07-18：聊天式五大处方生成复用低频、支持 reduced-motion 的 Codex 三点状态和精确已执行时间，并提供“停止生成”。客户端以预先绑定的 Trace ID 请求受限 BFF/API 取消入口；服务端在模型调用前持久化 running Trace、使用 Redis 身份范围取消协调、在取消终态写入 PHI-free 事件并拒绝创建草案。真实浏览器访客链路验证停止后回到可重试状态，数据库最新 `prescription.generate` Trace 为 `cancelled / prescription_generation_cancelled`。该能力只终止待复核草案生成，不改变医生批准、患者授权或正式发布门禁。

### 阶段 C 增量（2026-07-18）

- `prescription_draft_review` 现是患者可授予指定活跃医生的独立、可撤回且到期失效的资源权限。有效授权加医生账户/`clinical_intake:read` scope 后，医生可读取该患者最多 20 份加密五大处方草案，并仅看到自己对各草案的复核历史。
- 医生可追加 `approved` 或 `returned` 复核意见。意见以加密字段存储，记录草案内容的 SHA-256 指纹与该医生单调 revision；患者、其他医生、过期/撤回授权和无效草案均不能伪造或读取该记录。该操作记录的是对 AI 草案的临床复核，不验证医生资质、不签发电子处方、不执行任何治疗变更。
- 患者本人读取草案历史时会获得该草案的全部复核意见；聊天式五大处方重新进入同一会话后在已恢复草案下方直接显示最新一条“临床复核意见”，不强迫在两个页面之间切换。前端仍不将“通过”表述为可自行执行，且实时轮询/通知与医生工作台尚未实现。
