# 0034 — 前后端模块完备性审计

> 创建：2026-07-17 | 优先级：P0 | 状态：功能链路复核中 | 依据：`goal-objective.md`、`gerclaw设计要求.md`、`新任务.md`

## 范围与结论

本次审计覆盖 `apps/api/src/gerclaw_api/modules` 的 21 个模块、`apps/mvp`、活跃执行计划、需求矩阵、真实 Compose 回归及已有 MinerU 病例 PDF 验证。`AGENTS.md` 与 `README.md` 只代表模块约束和说明存在，**不是功能完成证据**。每一行必须同时核对可调用代码、API/Runtime 接线、前端或明确的非 UI 消费方、测试和真实运行证据；任何一项缺失都只能标为部分完成。

前端第一阶段可按既有浏览器验收结论继续进入后端阶段：布局、加载、语音控制和账户入口不应再作为阻断后端开发的理由。它仍不能被表述为完整产品交付，因为临床 workflow 与授权工作流尚未有真实后端终态可呈现。

## 目标要求但尚无独立模块边界的能力

`validation` 已新增独立模块，并实际接管 `public-chat-sse-v1`：Harness 发出事件和 FastAPI 入队浏览器 SSE 前均严格校验；浏览器 Zod 只接收声明的事件字段与 `timestamp`，未知字段/事件 fail closed。`local-rag-evidence-v1` 也已在 Hybrid RAG 产出、AgentScope adapter、引用投影和 RAG Eval 复用，缺少文档、章节、chunk 或来源 provenance 的结果不能以默认位置补写为引用。HTTP、模型、工具、memory、voice 和 export 尚未统一迁移。`orchestration` 已实际接管 Chat 的 Trace 建立/回放、会话租约、取消/失败终态和指标，Chat Service 通过 feature-owned 回调保留上下文、模型与领域工作；它不创建第二个 Agent/模型调用，也不替代临床副作用恢复、补偿与批准后执行。`observability_feedback` 已独立承接管理员的 Bad Case 聚合契约：数据库只按来源/严重度/状态计数，模块和 BFF/UI 不接触快照、反馈正文、图片或用户标识；它不替代 Trace 持久化，也不自动将真实病例提升为 Eval。`workflows` 已新增独立 Registry，实际约束 Chat 可选 workflow 的版本、owner、允许上下文和 workflow 风险档案，并将版本绑定写入 Trace；它不重复实现现有会话/Trace/checkpoint 层，也不代表临床 workflow 恢复/补偿已完成。`security_evaluation` 已有独立严格契约，并实际接入 Chat Toolkit、workflow registry 与核心资产构造：三种真实工具在注册和构建时均需要版本绑定的风险档案，外网工具缺服务端脱敏证明会 fail closed；四条 workflow 还会核验输入/输出、预算、非信任数据、患者归属、外发脱敏和检索溯源等适用控制；两个 Agent、Memory 与本地 RAG corpus 也在构造前核验适用 profile。它只是一段可执行的生产收敛，不能表述为应用范围的完整风险评估或发布门禁。Bad Case 授权去标识化晋升与 Skill 资产档案仍未实现。后续应在复用现有代码而非复制第二套架构的前提下，逐步收敛这些边界。

## 不可绕过的产品/合规输入

以下不是工程人员可安全猜测的空白，确认前保持 fail closed：

1. `docs/references/五大处方报告模板.md` 已提供五类报告结构和病例示例，现有 `five-prescription-report-v1` 契约也覆盖其共同字段；但模板不是 DDI/Beers/剂量/重复用药或个体化目标的可追溯医学规则来源。仍需这些规则的来源、许可、版本、审核人与复审日期，才能生成可供医生审核的临床草案。
2. 医生资质激活、患者授权资源粒度、有效期/撤回、紧急访问和审计保留策略。
3. 账号验证方式、找回/MFA/风控，以及访客数据迁移是否允许和如何确认。
4. 若要把 Mini-Cog/MMSE 的绘图、动作、观察条目升级为专业人员核验结果，仍需授权审核人、证据记录与患者/医生可见性策略；当前链路仅提供明确标注的本人作答筛查。

## 开发顺序

1. 先完成核心实用链路：Memory 的可用闭环、Skill 自然语言生成/版本/评测闭环、RAG 的本地知识库检索与无证据拒答评测；上传文档保持为受控会话输入而非知识库语料。它们的现有基础实现均已核验，但不能因已有接口而停止补齐。
2. 随每个真实后端终态接入患者/医生前端，并继续逐模块核对实际代码、API、消费方和测试；慢病账本已完成这一最低可用接线，仍不添加临床解释。
3. 在收到第 1 项医学资料后，按 0029 实现五大处方的确定性规则、RAG 证据、HITL 和报告闭环；病例 PDF 仅作为受控输入/关键证据，不作为公共知识库。
4. 最后处理隐私保护等外围防护模块的剩余功能，并在所有功能闭环后执行跨端 E2E、空卷 Docker 与≤10并发验收。
