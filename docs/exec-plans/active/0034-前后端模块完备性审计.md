# 0034 — 前后端模块完备性审计

> 创建：2026-07-17 | 优先级：P0 | 状态：功能链路复核中 | 依据：`goal-objective.md`、`gerclaw设计要求.md`、`新任务.md`

## 范围与结论

本次审计覆盖 `apps/api/src/gerclaw_api/modules` 的 19 个模块、`apps/mvp`、活跃执行计划、需求矩阵、真实 Compose 回归及已有 MinerU 病例 PDF 验证。`AGENTS.md` 与 `README.md` 只代表模块约束和说明存在，**不是功能完成证据**。每一行必须同时核对可调用代码、API/Runtime 接线、前端或明确的非 UI 消费方、测试和真实运行证据；任何一项缺失都只能标为部分完成。

前端第一阶段可按既有浏览器验收结论继续进入后端阶段：布局、加载、语音控制和账户入口不应再作为阻断后端开发的理由。它仍不能被表述为完整产品交付，因为临床 workflow 与授权工作流尚未有真实后端终态可呈现。

## 不可绕过的产品/合规输入

以下不是工程人员可安全猜测的空白，确认前保持 fail closed：

1. 五大处方通用、去标识 JSON schema；DDI/Beers/剂量/重复用药规则的来源、许可、版本、审核人与复审日期。
2. 医生资质激活、患者授权资源粒度、有效期/撤回、紧急访问和审计保留策略。
3. 账号验证方式、找回/MFA/风控，以及访客数据迁移是否允许和如何确认。
4. Mini-Cog/MMSE 绘图、动作、观察条目的授权人工审核人、证据记录与患者/医生可见性。

## 开发顺序

1. 先完成核心实用链路：Memory 的可用闭环、Skill 自然语言生成/版本/评测闭环、RAG 的本地知识库检索与无证据拒答评测；上传文档保持为受控会话输入而非知识库语料。它们的现有基础实现均已核验，但不能因已有接口而停止补齐。
2. 随每个真实后端终态接入患者/医生前端，并继续逐模块核对实际代码、API、消费方和测试；慢病账本已完成这一最低可用接线，仍不添加临床解释。
3. 在收到第 1 项医学资料后，按 0029 实现五大处方的确定性规则、RAG 证据、HITL 和报告闭环；病例 PDF 仅作为受控输入/关键证据，不作为公共知识库。
4. 最后处理隐私保护等外围防护模块的剩余功能，并在所有功能闭环后执行跨端 E2E、空卷 Docker 与≤10并发验收。

## 验收纪律

- 每项“已实现”必须有对应代码、契约测试与实际运行证据。
- 临床规则和患者数据不写入日志、Trace、评测集或回复；病例资料不得被复制为默认模板或 few-shot 示例。
- 模块需求必须同时更新 `docs/REQUIREMENTS_MATRIX.md`、长期规划和对应模块文档。
