# 0034 — 前后端模块完备性审计

> 创建：2026-07-17 | 优先级：P0 | 状态：功能链路复核中 | 依据：`goal-objective.md`、`gerclaw设计要求.md`、`新任务.md`

## 范围与结论

本次审计覆盖 `apps/api/src/gerclaw_api/modules` 的 19 个模块、`apps/mvp`、活跃执行计划、需求矩阵、真实 Compose 回归及已有 MinerU 病例 PDF 验证。`AGENTS.md` 与 `README.md` 只代表模块约束和说明存在，**不是功能完成证据**。每一行必须同时核对可调用代码、API/Runtime 接线、前端或明确的非 UI 消费方、测试和真实运行证据；任何一项缺失都只能标为部分完成。

前端第一阶段可按既有浏览器验收结论继续进入后端阶段：布局、加载、语音控制和账户入口不应再作为阻断后端开发的理由。它仍不能被表述为完整产品交付，因为临床 workflow 与授权工作流尚未有真实后端终态可呈现。

## 目标要求但尚无独立模块边界的能力

`goal-objective.md` 要求的 `validation`、`orchestration` 和 `observability-feedback` 仍分别散落在 Pydantic/route/service、`chat_service` 与 Trace service 中；没有与其他核心模块同等的独立模块目录、对外契约和完整运行时闭环。`workflows` 已新增独立 Registry，实际约束 Chat 可选 workflow 的版本、owner、允许上下文和 workflow 风险档案，并将版本绑定写入 Trace；它不重复实现现有会话/Trace/checkpoint 层，也不代表临床 workflow 恢复/补偿已完成。2026-07-17 新增的 `security_evaluation` 已有独立严格契约，并实际接入 Chat Toolkit：三种真实工具在注册和构建时均需要版本绑定的风险档案，外网工具缺服务端脱敏证明会 fail closed。但它只是一段可执行的生产收敛，不能表述为应用范围的完整风险评估或发布门禁。Bad Case 授权去标识化晋升及其余资产的风险档案仍未实现。后续应在复用现有代码而非复制第二套架构的前提下，逐步收敛这些边界。

## 不可绕过的产品/合规输入

以下不是工程人员可安全猜测的空白，确认前保持 fail closed：

1. `docs/references/五大处方报告模板.md` 已提供五类报告结构和病例示例，现有 `five-prescription-report-v1` 契约也覆盖其共同字段；但模板不是 DDI/Beers/剂量/重复用药或个体化目标的可追溯医学规则来源。仍需这些规则的来源、许可、版本、审核人与复审日期，才能生成可供医生审核的临床草案。
2. 医生资质激活、患者授权资源粒度、有效期/撤回、紧急访问和审计保留策略。
3. 账号验证方式、找回/MFA/风控，以及访客数据迁移是否允许和如何确认。
4. 若要把 Mini-Cog/MMSE 的绘图、动作、观察条目升级为专业人员核验结果，仍需授权审核人、证据记录与患者/医生可见性策略；当前链路仅提供明确标注的本人作答筛查。

## 开发顺序

1. 先完成核心实用链路：Memory 的可用闭环、Skill 自然语言生成/版本/评测闭环、RAG 的本地知识库检索与无证据拒答评测；上传文档保持为受控会话输入而非知识库语料。它们的现有基础实现均已核验，但不能因已有接口而停止补齐。

3. 在收到第 1 项医学资料后，按 0029 实现五大处方的确定性规则、RAG 证据、HITL 和报告闭环；病例 PDF 仅作为受控输入/关键证据，不作为公共知识库。
4. 最后处理隐私保护等外围防护模块的剩余功能，并在所有功能闭环后执行跨端 E2E、空卷 Docker 与≤10并发验收。
