# 0033 — 统一隐私脱敏与数据分类

> 创建：2026-07-17 | 优先级：P0 | 状态：阶段 A 进行中 | 前置：0014 基础设施、0018 外部搜索

## 目标

把分散的脱敏逻辑逐步收敛为可版本化、可测试、按外发目的执行的隐私边界；绝不把“文本替换”误表述为完整的数据生命周期、用户透明度或受控再识别能力。

## 阶段 A：外部搜索 query egress

- `modules/privacy_redaction/` 提供 `1.0.0` 的严格 Pydantic 契约和
  `redact_external_search_query()`。返回可外发文本与 PHI-free 分类计数，
  绝不返回匹配原文、位置或可逆映射。
- `ProductionSearchModule` 在所有 AnySearch/Tavily 调用前经过该策略；旧的
  `sanitize_search_query()` 仅保留兼容入口并委派给统一模块。
- 当前策略识别带标签的中英文姓名、手机号、身份证、邮箱、常见凭证赋值和控制
  字符；空或超过 4,000 字符的外发 query fail closed。
- 已运行 `33 passed`（隐私 canary 与 Search 回归）、Ruff、Mypy；重建 Compose
  API 后容器内 canary 输出 `1.0.0`、去标识文本与仅类别计数。

## 不在本阶段范围

- 不持久化原文或可逆映射，不新增未经授权的再识别能力。
- MinerU、ASR/TTS、模型 prompt、导出、评测集、Bad Case、用户数据用途页面、
  保留/删除/备份清除和供应商处理台账仍未接入本模块，不能声称已统一覆盖。
- 字段级分类注册表需要为每个 schema 定义存储、日志、Trace、vector、export 与
  provider 处理规则；当前 Runtime `DataClass` 不能替代这份注册表。

## 下一步

1. 为模型、MinerU、ASR/TTS 建立独立的目的限制 adapter，先以最小化数据而非
   “通用全量脱敏”外发。
2. 建立字段分类注册表、版本、canary/golden 评测和发布门槛。
3. 在真实账号/RBAC 后实现用户可见的数据用途、导出、删除和受控再识别审计。
