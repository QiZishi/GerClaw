# 0022 — 前端全接入与适老化设计

> 创建：2026-07-15 | 优先级：P0 | 状态：进行中 | 唯一客户端：`apps/mvp`

## 1. 目标

把已有 FastAPI/AgentScope 能力做成可真实操作、可理解、适合老年人的 Web 界面。`apps/web` 为空，不另建第二套客户端；`apps/mvp` 继续作为 Next.js BFF 前端，并以服务层连接后端。

## 2. 设计决定

- **对象与单一任务**：老年患者在 30 秒内发现语音或文字入口并开始一次安全健康咨询；医生在同一工作台迅速辨认会话、证据和待处理事项。
- **视觉方向**：专业冷蓝（医生）/亲和暖蓝（患者）的“诊间台 + 家庭健康状态带”。状态带以文字、图标和颜色同时表达联网、语音、审批和安全状态，是唯一强调性的视觉签名；其余区域克制、可读、避免模板化装饰。
- **排版与响应式**：继续使用系统中文字体；桌面三栏、1024px 收起右栏、768px 抽屉侧栏、手机单列；患者老年模式正文不小于 18px、操作目标不小于 48px。

## 3. 范围

1. 审计并修复 `apps/mvp` 的布局、按钮响应、加载/空/错误/离线状态与键盘可达性。
2. 打通现有 FastAPI 聊天/SSE、会话、Skill 与 0021 请求方审批 API；审批请求必须有可见状态、刷新、取消与正确的无权限提示，不能假装已执行。授权人查看详情、批准/拒绝必须建立在真实账号/RBAC 上，随 0026 账号计划交付，禁止把本地“医生模式”当成授权身份。
3. 整合患者/医生入口，统一三栏和移动端行为；保留所有外部访问于 `services/` + BFF allowlist，边界用 Zod 校验。
4. 清除 UI 中把尚未实现的 CGA、处方、用药审查、文档/语音能力伪装成成功的路径；无后端实现时给出明确“暂未接入/可继续对话”的安全降级。
5. 使用真实 headless browser 覆盖访客→患者、核心按钮、错误/离线和不同断点；每轮前端改动跑 lint/build。

## 4. 非目标

- 不在本计划虚构 CGA、五大处方、用药审查、Voice 或 MinerU 的后端业务能力；它们留给 0023+ 对应后端计划。
- 不迁移到空的 `apps/web`，也不复制一套前端。
- 不绕过 BFF、访客身份或后端的 HITL 权限判断。

## 5. 验收标准

- [ ] 首页、侧栏、聊天输入和右侧面板在四个断点均没有遮挡，键盘焦点和文字标签可用。
- [ ] 患者模式满足 ≥18px 正文、≥48px 可操作目标、高对比度及语音优先入口；医生模式保持高信息密度但不牺牲可读性。
- [ ] 每个可见按钮有真实效果或明确禁用/说明；无后端、网络失败、空会话、流式中断和高风险提示都有可恢复状态。
- [ ] 已接入功能（聊天、Skill、审批）只经 `services/` 与 BFF allowlist；响应按 Zod schema 校验，不展示技术错误或伪成功。
- [ ] 0021 审批请求可在聊天流中呈现，请求人能够查看状态并取消；普通访客不能决定审批或读取敏感参数/执行 token。授权人工作台在 0026 真实账号/RBAC 建立后验收。
- [ ] `npm run lint`、`npm run build` 与浏览器审阅通过；独立审阅无 P0/P1 后再归档。

## 6. 预定证据

- eslint、Next production build 与现有前端必要测试。
- Playwright snapshot/screenshot：桌面、平板、手机，及访客/患者状态。
- BFF/API 负向演练：未配置后端、网络失败、审批无权限和可恢复提示。
