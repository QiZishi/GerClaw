# 0025 — 真实身份、RBAC 与患者授权

> 创建：2026-07-16 | 优先级：P0 | 状态：阶段 B/C 进行中 | 前置：0014 基础设施、0021 Runtime Permission/HITL

## 1. 目标

把登录页作为系统的强制入口，并在该入口提供“以游客身份进入患者端”选项；同时提供真实患者与医生账号的服务端身份链路。账号模式必须固定角色、最小权限、支持短期/刷新会话与密码变更；医生只能在患者已授权的最小资源范围内访问数据，授权到期或撤回后立即失效。游客仅可使用患者端，本次会话历史不在下次进入时恢复，且不能借 UI 角色切换获得账号权限；其运行数据仍作为受控 bad-case/Trace 数据留存。

## 2. 已确认的需求与事实

- `docs/PRD.md` 的 `IAM-01` 要求访客、患者、医生账号的注册、登录、退出、密码变更和短期/刷新会话；密码须使用强哈希。
- `IAM-02`、`IAM-05`、`IAM-06` 要求 tenant/actor/user/role/ownership 全链路隔离，患者授权可授予、到期、撤回，撤回后缓存、下载链接和后续查询立即失效；不得存在万能 scope、前端声明角色或仅 UI 隐藏的控制。
- 当前已实现本地患者/医生账号、可撤销刷新会话及由 BFF 保护的 cookie/CSRF；账号角色由服务端 JWT 传入 Runtime。版本化、到期和 revision-fenced 的患者授权实体已经真实存在，当前范围为 `health_profile_read`、`cga_report_read` 与 `prescription_draft_review`。医生资质仍未实现。`RuntimePrincipal` 的 `patient_id` 与 `patient_access_verified` 仅代表访客/患者对其自身会话主体的服务端证明；医生账号没有该证明，不能将该布尔值视为前端可声明的权限。
- 现有医生模式只是本地展示状态，不能作为审批人、跨患者读取者或临床动作执行者。

## 3. 不可违反的安全约束

- 所有身份、角色、scope、患者归属、授权有效期和撤回状态只由服务端验证；浏览器不可传入或覆盖任何一项。
- 密码仅保存强哈希；刷新 token 仅保存有轮换标识的不可逆摘要，短期 access token、会话 cookie、CSRF 与登出必须可撤销。
- 医生访问必须同时满足已验证账号角色、tenant、有效授权、目标资源类型和患者 ID；所有缺失或过期条件均 fail closed。
- 角色升级、医生资质、患者授权授予/撤回、登录异常、token 重放和跨患者拒绝都写 PHI-free 审计事件；不得把密码、token、联系方式或病历写入 Trace/日志。
- 授权撤回时必须原子失效后续 API 查询、缓存键、预签名下载链接与活跃刷新会话；已经合法导出的离线副本不得虚假承诺远程撤回。
- 访客数据向账号迁移必须有明确、一次性、可审计的确认动作；未确认前不得自动合并或把其他浏览器访客数据绑定到账号。

## 4. 待产品/合规确认的决策

以下项目未在权威需求中给出，决定了数据模型与安全边界，确认前不得实现为生产行为：

1. **账号标识与验证**：使用邮箱、手机号码、医院工号/SSO，还是多种方式；其验证、找回、MFA 与风控要求。
2. **医生入驻与资质**：谁可创建/激活医生账号，是否需要机构管理员审核、执业信息校验和复审周期。
3. **患者授权粒度**：按患者—医生、患者—机构、会话、文档、CGA、处方/用药审查等何种资源类型授予；默认期限、最大期限、续期与紧急访问政策。
4. **法律与数据策略**：身份信息/授权审计/刷新会话的保留期、删除与 legal hold；对已下载文件的撤回告知文案。
5. **访客迁移**：是否允许把现有访客会话、CGA、文档和健康画像迁入账号；允许时的用户确认、冲突处理与撤销窗口。

## 5. 实施阶段

### 阶段 A — 身份与会话基础

- 定义版本化 Pydantic/Zod 契约、账号/凭据/刷新会话实体、Alembic 迁移、密码哈希与认证 service。
- 提供注册、验证、登录、刷新轮换、登出和密码变更 API；建立 CSRF、限流、枚举防护、token 重放与安全审计测试。
- Next BFF 仅以 HttpOnly/Secure/SameSite cookie 持有会话，不向前端暴露 refresh token；访客与账号 JWT claim、scope 和 actor 解析清晰分离。

### 阶段 B — 固定角色与患者授权

- 服务端创建不可由客户端自升的患者/医生角色与机构归属；经确认的医生资质流程才可激活临床 scope。
- 建立版本化、可到期/撤回的 patient access grant，含资源范围、理由、授予者、审批/版本和审计；Runtime PermissionEngine 在每次受保护访问重新校验。
- 接入 CGA 历史、健康画像、文档、审批与后续处方/用药审查；分别覆盖跨患者、跨医生、过期、撤回、缓存和下载链接失效。

### 阶段 C — 前端与受控迁移

- 账号入口、会话状态、角色可见性、授权授予/撤回和过期状态均从真实后端读取；不以本地模式切换冒充权限。
- 若第 4 节确认允许，提供一次性访客数据迁移向导；否则维持访客数据隔离与明确说明。
- 完成患者、医生、审核/运维角色的浏览器 E2E、≤10 并发会话、审计查询与独立审阅。

## 6. 验收标准

- [ ] 账号注册、验证、登录、刷新、登出、密码变更均由严格 schema、强哈希、服务端限流和可撤销会话支撑。
- [ ] 前端角色切换不改变 API 身份；访客 JWT 不能访问患者/医生账号数据或临床 scope。
- [ ] 数据库、缓存、向量、文件、报告、Trace 和 API 均按 tenant/actor/user/role/patient ownership 隔离。
- [ ] 医生仅在患者有效最小授权内访问对应资源；到期/撤回立即阻断 API、缓存、下载链接和后续刷新会话。
- [ ] 授权、角色、登录和撤回都有 PHI-free 审计；负向、竞争、重放、过期、跨租户/跨患者与浏览器 E2E 覆盖。
- [ ] CGA、文档、审批、处方和用药审查使用同一授权事实源，不复制本地权限判断。
