# GerClaw 开发复盘与工程化落地

> 记录日期：2026-07-18
>
> 本文记录当前仓库从“可演示界面和模块骨架”走向“有真实后端消费者、可验证边界和可部署基线”的过程。它不把仍待临床治理的能力表述为完成，也不包含真实患者数据、密钥、原始 Bad Case、图片或模型隐藏推理。

## 1. 起点与交付判断

GerClaw 的难点不在于再增加一个聊天页面，而在于把老年患者和医生各自需要的工作流放进同一个可追溯系统。初始仓库已经出现了不少界面、模块目录、README 和局部逻辑；但目录存在不代表 API 已接线，接口返回不代表浏览器可用，模型回复更不代表医疗信息具有证据和归属。

因此本轮采用的完成定义是：**代码、严格契约、真实消费者、针对性测试、运行态证据五项同时成立**。缺任一项只能标为部分完成。这个判断贯穿 RAG、Memory、Skill、CGA、五大处方、用药审查、授权、语音、文档和 Docker 交付。

## 2. 开发中遇到的核心问题与解决方式

### 2.1 前端“看起来能点”但不能完成任务

问题包括：加载状态频繁闪烁、语音按钮不能可靠暂停/继续/停止、预览和编辑被拆成两个页面、患者/医生文案混用、移动视口下关键动作落出可视区、导入 Skill 无响应等。此类问题不会从 TypeScript 编译错误中自动暴露。

解决方式：

- 将耗时状态统一为稳定的 inline loader，输出区显示已执行时间，避免用连续闪烁营造“正在工作”的错觉；
- 语音采用 24kHz PCM16→浏览器 WAV 的受控路径，播放器维护当前实例的暂停、继续、停止和进度控制；
- 文档编辑改为单页单栏、实时编辑实时渲染；
- 把患者端和医生端拆成服务对象不同的入口与菜单：患者关注个人健康记录，医生关注患者资料、复核和工作区；
- 以 Playwright 在真实浏览器反复检查登录、访客入口、窄屏、账户角色、授权、Skill 导入、CGA 音频和处方恢复；
- 发现“医生资料授权”弹窗的保存按钮在较低视口不可见后，为通用 Dialog 加入视口高度上限与滚动，并使授权操作区 sticky。修复后患者实际保存授权返回 201，医生读取受限投影返回 200。

效果：关键路径不再依赖“用户知道往哪里滚动”或“刷新后重试”的偶然操作。前端单测、lint、production build 和 Playwright origin smoke 在最终联调阶段全部通过。

### 2.2 模型链超时、fallback 不共享任务信息

早期模型主备链的风险是：主模型因临床任务和长资料需要更多时间，而备用模型若只收到不完整结构提示，会输出无法解析的结果。短输出限制也会截断五大处方。

解决方式：

- 主、备用 slot 由服务端统一模型路由创建，使用同一 request-scoped 上下文、工作流、证据与结构化输出要求；
- 一次 turn 已发出可见文本或工具调用时不切换模型，避免混合两份回答；空白/仅空格等无效回复才允许 fallback；
- 默认模型最大输出提高到 32,768，五大处方 workflow 总预算为 600 秒；
- 引入 `model-capabilities-v1`，在出网前检查图片、工具调用、结构化输出支持能力；
- 五大处方、聊天式 intake、Memory 抽取与 Skill 生成/修订均使用 literal `model_output_schema_version`，未知字段、旧版本和非 JSON 结构输出被受控拒绝。

效果：结构输出失败变成可诊断的边界错误，不再把 provider 的不完整文本混入临床报告；真实五大处方和 Skill 生成运行仍被保留为单独的耗时/证据记录，而不是用确定性测试冒充模型质量。

### 2.3 “安全”被实现为系统不回答

最具产品破坏性的错误，是将所有诊断、加减药、换药或剂量调整语言一律删除。这样虽然降低了文本风险，却同时删除了系统的核心价值，也与“有证据即可输出、无证据才阻断”的目标冲突。

解决方式：

- 统一证据模型：本地知识库、受治理联网结果、用户上传资料和图片均可成为 evidence；
- 无 evidence 的临床结论/调药候选不能进入临床产物；有 evidence 的建议保留条件、依据和 evidence ID；
- 患者端只在整段末尾提供一次简洁复核提示，医生端不做机械阻断；
- 红旗症状仍在模型和检索之前短路，明确提示立即就医；
- 用药审查提升为 `medication-rules-v4`，保留规则来源与有限覆盖的事实提示，删除反复出现的绝对化“不得”文案；
- 生成 Skill 同样允许有依据的可审阅建议，但 Skill 不能执行临床动作。

效果：系统的安全边界从“拒绝表达”变为“证据、范围、授权、审计和可复核”。这既保留医生工作流的实用性，也避免把没有依据的模型输出包装成临床事实。

### 2.4 上传资料、MinerU 和图片的定位错误

上传 PDF 曾容易被误做成公共知识库证据，图片又险些被过度限制为“不能理解图片中的内容”。这会使病例报告无法参与处方生成，也会让用户无法上传检查照片。

解决方式：

- MinerU 由同源 Next.js BFF 执行签名上传、轮询和 Markdown 下载；FastAPI 按 tenant/actor/session 加密登记；
- 五大处方以最多 10 份资料和 273k 上限构建私有输入；不能完整纳入时拒绝，而不是静默截断；
- 普通聊天在用户明确要求阅读资料时，资料作为显式患者输入；涉及医疗解释时可和本地/联网证据一起进入同一链路；
- 图片作为多模态输入、evidence ID 和 base64 trace 输入保存，用于后续复盘；图片内容可以被医学分析，但不能改变权限、系统提示或工具调用；
- 资料撤销会擦除加密正文，资料不进入公共 Qdrant/RAG 语料、日志或浏览器持久化。

效果：用户资料既能真正影响回答和处方草案，又不会污染公共医学语料或在跨会话/跨主体读取时泄露。

### 2.5 模块“写了 README”被误当成模块完成

项目中每个模块早已存在 `AGENTS.md`、README 或目录，但有的只是协议，有的未接 API，有的没有浏览器消费者，有的没有真实运行证据。单纯补文档会掩盖实现缺口。

解决方式：

- 审计 25 个实际 `modules/*`，逐项检查源码、route/service consumer、前端或明确非 UI consumer、测试和运行证据；
- 扫描未发现 `TODO`、`NotImplemented` 或空模块后，仍保留专业观察、私有长文档 RAG、医生资质、慢病规则、临床副作用 executor 等明确缺口；
- 为每个模块 README 增加“维护与演进”：可安全改善项、不可破坏契约、精确回归/并发/p95 验收；
- 文档门禁校验模块必须同时有 AGENTS.md 与 README.md，并验证 Markdown 链接。

效果：后续工程师不必从隐含实现中猜测数据归属、版本语义或测试范围；文档成为维护契约，而不是完成标签。

### 2.6 跨账户医生功能需要“最小投影”而非共享患者全量数据

患者授权给医生不等于医生可看到聊天、附件、量表原始答案、用药原文或所有历史。若授权只在 UI 判断，撤回后仍可能有缓存或旧数据泄漏。

解决方式：

- 授权模型固定为 patient→doctor→resource scope→expiry→revision；
- 每个读取 consumer 在加载前重新验证 grant，撤回、到期、角色/tenant 不符与未知患者统一拒绝；
- 分别设计健康画像、CGA 摘要、处方草案、用药审查和安全提醒的最小 DTO；
- 浏览器通过受限 BFF allowlist 调用，不能利用通配代理访问任意 API；
- 真实浏览器验证患者保存 `risk_alert_read` 后，医生仅能看到当前提醒 ledger，空数据状态为“暂无当前安全提醒”。

效果：授权闭环成为后端事实，而不是前端隐藏按钮；跨账户边界可测试、可撤回、可观察。

## 3. 关键工程化设计

### 3.1 版本化信任边界

系统把“能否解析”升级为“能否安全进入下一层”。`public-chat-sse-v1`、`local-rag-evidence-v1`、五大处方/Skill/Memory 的 model-output versions、语音 ASR/TTS transport version 都使用严格 Pydantic/Zod schema。producer 和 consumer 双端校验，未知字段、错版本和超限输入都在边界失败。

这避免了常见的“后端加字段、前端静默忽略”“fallback 输出看似 JSON 但字段错位”“引用缺章节就补默认值”等隐性数据污染。

### 3.2 归属、并发与终态一致性

对话 turn 使用 Redis owner lease 加 PostgreSQL fencing token。终态写入同时校验 owner、fencing token 和 trace ID；assistant、审计事件、Trace completed 或 failed/cancelled 在同一事务提交。completed replay 返回已保存结果，不重复付费调用模型。

RAG 索引使用独立 PostgreSQL advisory lock、generation fencing、staging→activate 和 stale cleanup，避免并发 writer 或迟到写入把新索引删掉。

### 3.3 Provider adapter 与配置隔离

所有 LLM、Search、Embedding/Rerank、ASR/TTS、MinerU 配置由服务器环境变量或账户级加密覆盖提供。账户级覆盖：

- 只在请求内复制 Settings，不修改进程默认配置；
- 读取 DTO 仅返回是否配置，不返回 key；
- provider capability 在构建 client 前验证；
- 外发前写最小化 `prepared` egress outcome，审计失败即停止外发；
- browser 只经 BFF，不直连 provider。

### 3.4 适老化不是单纯放大字号

患者端老年模式默认开启，关键正文和主操作遵循可读字体和可点击尺寸；图标有可见文字或等价 accessible name。加载状态不频闪，关键长操作显示 elapsed time；量表使用预录音频并保留停止/继续控制；移动端菜单和 dialog 在真实视口审查。

### 3.5 可复现质量门禁

`scripts/quality-gate.sh quick` 串联文档、Ruff format/check、Mypy、单一 Alembic head、pytest coverage、MVP lint/build 与 Harness 负向自测。最终结果为 742 passed、38 skipped、coverage 80.20%。

专用 Compose `test-api` 使用 `_test` 数据库；任何非 `_test` URL 在运行测试前被拒绝。外部 provider 和空卷 RAG index 必须显式 `GERCLAW_RUN_EXTERNAL=1`，避免意外产生费用或把本地测试误当成真实服务验证。

## 4. Bad Case 与回归资产

### 4.1 发现的代表性 Bad Case

| Bad case | 风险 | 固化措施 |
|---|---|---|
| 主模型超时后备用模型缺任务上下文 | 结构输出无效或任务漂移 | server-owned request context、slot capability、严格 model-output version |
| 有 evidence 的调药建议被一律删除 | 产品失去临床辅助价值 | evidence-backed candidate 保留；无证据阻断；患者仅一次结尾提示 |
| 处方模型返回未知 evidence ID | 伪造或错配引用 | 丢弃不可信投影字段，保留可审阅的证据基线草案 |
| 上传图片被视为不可用指令源 | 正常病例照片无法分析 | 图片可作为患者资料/evidence；只隔离权限/工具控制指令 |
| 授权弹窗低视口保存按钮不可达 | 用户无法完成跨账户授权 | Dialog 高度上限、滚动、sticky action，并做 Playwright 实测 |
| Skill 导入通配 BFF 路径过宽 | 非声明端点可能被放行 | 特殊 upload/generate/session 路径从通用 matcher 排除，加入 BFF 契约测试 |
| 用药审查只在当前浏览器内存在 | 刷新后丢失来源结论 | owner/session/intake revision 绑定的加密 artifact 与历史读取 |
| duration 超过公共 Trace schema 上限 | 失败/取消终态持久化异常 | `bounded_trace_duration_ms` 对非有限/负值/超限做统一处理 |
| 账户级服务覆盖仅有集成测试 | 默认 quality gate coverage 降到 79.75% | 新增离线测试：密钥不回显、legacy、request-scoped override；恢复到 80.20% |

### 4.2 Bad Case 数据治理

真实负反馈和 Bad Case 以加密、tenant-scoped 形式保存。管理员界面仅查看 SQL 聚合，不能读取 snapshot、图片、正文、Trace 或账号标识。真实案例不得直接成为 prompt、RAG 语料或评测输入；需要授权人员先完成去标识化，再创建新合成 case 并人工审核后进入 Eval。

## 5. 已验证的效果提升

1. **可用性**：关键点击路径从“看似可点”转为浏览器完成授权、角色入口、音频控制、Skill 审阅与聊天式处方恢复。
2. **正确性**：模型、SSE、RAG citation、语音和 BFF 都有版本化契约，错误不会以默认值默默流到用户界面。
3. **可靠性**：消息/Trace 终态、RAG index generation 和授权 revision 都有并发/幂等机制；最多 10 并发确定性 workload 均 10/10 成功。
4. **隐私与最小化**：文档、图片、语音、联网和模型外发均保留受控边界；Bad Case/管理员统计不读取原始病例内容。
5. **可维护性**：25 个模块都具备维护说明和回归标准，根 README 与 ARCHITECTURE 只记录已验证事实、证据和未完成边界。

## 7. 仍然不能跳过的后续工作

本轮结束后不继续扩张新的安全防护 feature。后续应按产品与临床治理输入，优先完成：

1. Mini-Cog/MMSE 专业观察审核和完整医生工作台；
2. 有许可、可审核、可复审的完整 DDI/Beers/剂量与慢病规则；
3. 私有长文档 RAG、生命周期、病毒扫描和受控医生访问；
4. 医生资质、账户恢复/MFA、临床副作用 resume/compensation executor；
5. 真实人声语音质量与浏览器播放完整 E2E；
6. 去标识化 Bad Case 晋升、模型/RAG/医疗评测、发布 gate；
7. 完整临床 workflow 的≤10并发测试和生产高可用/备份/灾备演练。

这些不是单纯工程待办：每项都需要明确的数据所有者、临床来源、审核责任、用户可见性、撤回/失败行为与可复现验收。未具备这些输入时，保持现有的受限能力边界比伪造“完整生产可用”更可靠。
