## 1. 产品定位

GerClaw 是一款面向老年患者与老年科医生的 Web 端 AI 双向诊疗平台，以老年专科医生智能体为核心，通过语音优先的适老化交互和专业CGA评估/五大处方能力，为老年人提供便捷、专业、安全的智能医疗健康服务。

## 2. 核心价值

| 价值维度 | 说明 |
|---------|------|
| 医疗可及性 | 通过语音交互和适老化设计，降低老年人使用AI医疗服务的门槛，解决老年人数字鸿沟问题 |
| 专业规范化 | 基于循证医学的老年综合评估(CGA)和五大处方体系，输出结构化、可溯源的医疗建议 |
| 双向诊疗 | 同时服务患者端（语音对话采集、健康咨询）和医生端（辅助诊断、处方生成、评估工作区），形成医患闭环 |
| 医疗安全 | 通过权限分级、免责声明、循证引用、双智能体复核等机制，确保AI医疗输出的安全性和可解释性 |
| 模块化可演进 | MVP纯前端快速验证，二阶段平滑迁移到前后端分离+AgentScope多智能体架构，支持Docker容器化生产部署 |

## 3. 目标用户

- **核心用户 - 老年患者**：65岁以上老年人，特征为视力/听力/操作能力下降，需要大字体、大按钮、语音优先的交互方式；核心场景是健康咨询、用药提醒、慢病管理、CGA自我评估；痛点是传统医疗APP操作复杂、不会打字、看不清小字。
- **核心用户 - 老年科医生**：老年科/全科临床医生，特征为工作繁忙、需要快速获取患者综合健康信息；核心场景是辅助诊断、CGA评估工作区、处方开具与审核、患者管理；痛点是CGA评估耗时长、老年患者用药复杂易出错、缺乏系统化工具。
- **利益相关方**：老年患者家属（协助查看报告）、医疗机构管理者（质量管控）、医疗AI安全审核人员。

## 4. 功能清单

### 4.1 分阶段规划

| 阶段 | 部署平台 | 架构 | 目标 |
|------|---------|------|------|
| **MVP（一阶段）** | IGA Pages | 纯前端Next.js静态导出，所有API前端直连，API Key通过部署平台环境变量注入 | 核心功能闭环，可演示可体验，所有模型API服务可真实调用 |
| **二阶段（生产）** | ModelScope (Docker) | 前后端分离：Next.js前端 + FastAPI后端 + AgentScope多智能体 + PostgreSQL/Redis/Qdrant，MVP代码归档到apps/mvp/，API调用迁移至Python后端 | 企业级生产系统，日均万次DAU，完整权限引擎+HITL+RAG+账号系统 |

> **重要架构决策**：MVP阶段API服务嵌入前端是因为IGA Pages不支持Python后端运行。二阶段开发时需将所有API调用服务从前端移除，以Python代码写入FastAPI后端，最终Docker部署解决该问题。

## 7. 协作模式确认

- **模式**：双智能体对抗模式（开发者写+跑，审阅者验+判）
- **开发者职责**：编码、自测（npm run build/lint检查）、提交变更汇报、启动前端服务
- **审阅者职责**：独立审查代码、独立测试运行、铁律检查、判定通过/不通过、回写长期规划
- **角色分离**：禁止同一智能体既开发又验收
- **前端测试铁律**：涉及前端的任务，必须在审阅通过后启动前端服务让用户手动测试，记录后台日志，用户反馈+日志修复后重新进入对抗循环，审阅者通过且用户测试通过才能写入长期规划并进入下一阶段

## 8. 铁律确认

### 通用铁律（3条，不可删减）

1. **真实执行优先**：禁止空谈"应该可以"，必须实际运行npm run build/npm run dev并附完整输出
2. **项目视角优先**：禁止破坏现有逻辑，改完必须跑全量构建和关键路径测试
3. **禁止虚伪成功**：禁止模糊表述"已完成"，失败必须如实报告错误信息和复现步骤

### 项目特定铁律

- **医疗安全底线**：禁止给出确定性诊断结论（如"您患有XX病"），所有医疗输出必须附带免责声明，高风险症状（胸痛/呼吸困难/意识障碍等）必须提示立即就医
- **循证可溯源**：所有医学建议必须标注来源依据（联网搜索结果/指南引用），禁止编造医学知识或引用不存在的文献；未启用联网搜索时对不确定的医学问题必须明确告知
- **适老化无障碍**：患者端老年模式下禁止使用<16px字体、<44px按钮、低对比度配色；禁止纯图标无文字标签按钮；禁止自动播放声音（除用户主动触发的TTS）
- **配置不可硬编码**：所有API Key、模型名、URL、协议必须通过环境变量配置，禁止在代码中硬编码任何API Key或特定模型名称；参考设计要求4.5节配置管理规范
- **技术栈优先级**：严格遵循设计要求2.1.1节三层优先级准则（设计要求>AgentScope参考>技术选型推荐），二阶段开发时智能体相关能力优先使用AgentScope，禁止跳过AgentScope直接使用其他框架
- **禁止读取origin目录**：严禁阅读gerclaw-main-origin和gerclaw-design-origin目录下的任何文件作为参考，必须从设计要求文档和参考资料独立开发

## 10. 约束与风险

| 类型 | 内容 | 缓解措施 |
|------|------|---------|
| 已知约束 | IGA Pages不支持Python后端运行，MVP必须纯前端静态导出 | API调用全部前端直连，API Key通过部署平台环境变量注入；二阶段迁移FastAPI后端 |
| 已知约束 | 浏览器端调用API会暴露API Key在网络请求中（虽然不在源码中） | MVP阶段可接受（部署平台环境变量不硬编码到仓库），二阶段通过FastAPI后端代理解决 |
| 技术风险 | GPT-4o API调用存在数据出境风险，可能受网络影响 | 模型配置完全环境变量化，支持一键切换到DashScope qwen-plus等国内模型作为备份 |
| 技术风险 | Mimo ASR/TTS的PCM16流式播放在浏览器端实现较复杂（24kHz PCM16LE单声道，需逐chunk解码播放） | 使用Web Audio API的AudioBufferSourceNode进行流式播放，参考Mimo官方文档实现 |
| 技术风险 | MinerU云API的具体endpoint/认证方式尚未提供 | PRD标注待定，开发前需用户确认；先用图片多模态作为替代方案 |
| 技术风险 | 纯前端localStorage存储容量有限（约5-10MB），大量会话历史可能溢出 | 实现会话数量上限和自动清理策略，提示用户重要内容及时导出 |
| 依赖风险 | 所有功能依赖外部API服务（OpenAI/Mimo/AnySearch/Tavily/MinerU），任何一个服务不可用都会影响体验 | 实现主备模型切换、AnySearch/Tavily双搜索兜底、API超时重试、友好错误提示 |
| 医疗风险 | AI生成医疗建议存在错误风险，可能误导老年用户 | 系统prompt严格约束不做确定性诊断，所有输出带免责声明，高风险症状提示立即就医，循证来源可追溯 |
| UX风险 | 适老化设计需要在"信息完整性"和"简洁易用"之间取得平衡 | 患者端默认展示关键信息和大按钮，详细信息折叠在"查看详情"中；参考设计要求13.7节完整适老化规范 |
