# 工具调用可视化 — 设计文档

> 模块：工具调用可视化 | 关联产品规格：[product-specs/工具调用可视化.md](../product-specs/工具调用可视化.md)

---

## 1. 设计目标

基于React组件实现AI工具调用过程的可视化展示，对齐Trae Work交互体验，满足以下设计目标：

1. **三态清晰感知**：运行中（旋转+实时计时）、完成（绿勾+耗时）、失败（红叉+错误+重试）三种状态视觉明确，用户一眼可知工具执行情况
2. **实时流式反馈**：工具调用随SSE流实时更新状态，开始调用立即显示卡片，结果返回立即更新状态，计时实时跳动，减少用户等待焦虑
3. **详情按需展开**：默认折叠不占用空间，点击展开查看完整参数、返回结果（JSON格式化高亮）、耗时详情，满足医生用户验证需求
4. **工具类型可识别**：不同工具（搜索/文件/语音/处方/评估/用药）使用不同图标和中文名称，用户快速识别AI在做什么
5. **失败可重试**：工具调用失败提供一键重试按钮，重新执行该调用，提升可靠性
6. **体验对齐Trae Work**：卡片布局、颜色、交互、动画风格参考Trae Work，保持产品体验一致性
7. **多工具并行支持**：支持并行/串行多个工具调用，每个工具独立卡片、独立状态、独立计时，ID关联不混乱
8. **适老化无障碍**：卡片点击区域足够大，状态不仅靠颜色区分（同时有形状图标），支持键盘操作

约束：
- 作为通用对话模块的子组件，集成到ContentBlockRenderer体系
- Vercel AI SDK处理tool_call/tool_result的SSE解析，本组件负责UI渲染
- 纯前端展示组件，无后端依赖
- 工具类型映射表可扩展，新工具添加只需更新配置
- **§4.2.3 七项可视化统筹**：本模块除工具调用（ToolCallBlock）外，还负责统筹子智能体加载（SubAgentTree）、决策过程（DecisionTimeline）、流式文本输出（StreamingText）三项可视化组件；思维链见 `思维链展示.md`、联网搜索结果见 `联网搜索.md`、文档解析状态见 `文档解析.md`

## 2. 架构设计

### 2.1 模块位置

在整体架构中的位置：**前端UI组件层**，位于`components/chat/blocks/tool-call-block.tsx`，是通用对话模块ContentBlockRenderer的子组件，负责`type: 'tool_call'`内容块的渲染。

- **上层依赖（使用者）**：
  - ContentBlockRenderer：消息内容块分发器，遇到`type: 'tool_call'`的block渲染本组件
  - Message组件：消息容器，提供流式状态上下文
  - 各功能模块（联网搜索、五大处方、CGA评估、用药审查、文档解析）：这些模块定义和注册工具，但工具调用的UI展示统一由本组件负责
- **本模块内部**：ToolCallBlock主组件、三态子视图、计时器Hook、JSON高亮组件、工具图标映射
- **下层依赖**：
  - Vercel AI SDK：提供tool_call/tool_result流式delta
  - StreamingChatClient：解析SSE delta，构建ToolCallBlock数据，处理重试逻辑
  - ShadCN/UI基础组件：Button、Collapsible、Badge
  - Lucide React图标库
  - 代码高亮库（用于JSON高亮）

### 2.3 数据流

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    StreamingChatClient (流式客户端)                      │
│                                                                         │
│  SSE chunk接收：                                                        │
│  ├─ delta 包含 tool_calls 数组？                                        │
│  │   └─ 每个tool_call：                                                │
│  │       ├─ 新toolCallId → 创建新ToolCallBlock，status='running'       │
│  │       └─ 增量args（流式参数）→ 追加到params                         │
│  ├─ delta 包含 tool_result？                                           │
│  │   └─ 按toolCallId查找：                                             │
│  │       ├─ 有result字段 → status='completed', result=..., 计算耗时    │
│  │       └─ 有error字段/错误状态 → status='failed', error=..., 计算耗时│
│  └─ delta 包含 finish_reason → 标记所有运行中工具完成（或失败）         │
│                                                                         │
│  工具重试处理：                                                          │
│  - retryToolCall(toolCallId) → 从消息中提取原始参数 → 重新发送工具调用  │
│  - 重置该ToolCallBlock为running状态，新的结果通过SSE流更新              │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                        ContentBlockRenderer                             │
│  遍历消息blocks数组，按顺序渲染：                                        │
│  ThinkingBlock → ToolCallBlock(s) → TextBlock(s) → ImageBlock...        │
│  block.type === 'tool_call' → <ToolCallBlock block={block} />           │
│  多个ToolCallBlock按数组顺序排列                                        │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                         ToolCallBlock 组件                              │
│                                                                         │
│  Props: block: ToolCallBlock                                            │
│  内部状态：isCollapsed（默认true）                                       │
│                                                                         │
│  useToolTimer(block)：                                                 │
│  ├─ status === 'running' → setInterval每100ms更新elapsedMs             │
│  └─ status !== 'running' → 清除interval，elapsedMs = durationMs        │
│                                                                         │
│  渲染逻辑：                                                             │
│  ├─ status === 'running' → <ToolRunningState ... />                    │
│  ├─ status === 'completed' → <ToolCompletedState ... />                │
│  └─ status === 'failed' → <ToolFailedState ... onRetry={retry} />      │
│                                                                         │
│  点击卡片头部 → toggle isCollapsed                                      │
│  isCollapsed === false → <Collapsible><ToolDetails ... /></Collapsible>│
│                                                                         │
│  重试：调用 retryToolCall(block.toolCallId)                             │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
        ┌───────────────────────┼───────────────────────┐
        ▼                       ▼                       ▼
┌───────────────┐     ┌─────────────────────┐   ┌─────────────────┐
│ 复制操作      │     │ 导出操作            │   │ 正常展示        │
│ copyMessage() │     │ sanitizeExport()    │   │ 用户可见        │
│ 过滤工具块    │     │ 移除.tool-call-block│   │ 可展开/重试     │
│ 只复制正文    │     │ 不出现在导出文件    │   │ 查看参数结果    │
└───────────────┘     └─────────────────────┘   └─────────────────┘
```

## 3. 接口设计


## 4. 数据设计

### 4.1 数据库表/集合（如适用）

ToolCallBlock数据作为Message的一部分，随消息一起存储到localStorage中：
- 工具调用完成/失败后，完整的params、result、error、durationMs都随消息持久化
- 刷新页面后工具卡片显示最终状态（完成/失败），不显示"运行中"状态
- `isStreaming`序列化时始终为false（刷新后无流式状态）
- `isCollapsed`不持久化，每次加载默认折叠

### 4.2 状态机

**ToolCallBlock状态机**：

```
idle（不渲染）→ 收到tool_call delta → running（运行中）
  │                                  │
  │                                  ├─ 实时计时（每100ms更新elapsed）
  │                                  ├─ 流式追加params（如为流式参数）
  │                                  ├─ 用户点击重试（失败后）→ running（重置）
  │                                  └─ 收到tool_result
  │                                      ├─ 成功结果 → completed（完成）
  │                                      │   └─ 用户点击展开 → expanded ↔ collapsed
  │                                      └─ 错误/超时 → failed（失败）
  │                                          ├─ 用户点击重试 → running
  │                                          └─ 用户点击展开 → expanded ↔ collapsed
  └─ 流结束仍running → marked as failed（超时中断）→ failed
```

**展开/折叠状态**：
```
collapsed（默认）→ 用户点击卡片 → expanding → expanded
      ↑                                               ↓
      └─────────────── collapsing ← 用户点击卡片 ───────┘
```

**计时器状态**：
```
idle → status=running → ticking（setInterval每100ms更新elapsedMs）
      ↑                      │
      └──── status≠running ──┘ → stopped（清除interval，elapsedMs=durationMs）
```

## 5. 错误处理

| 错误类型 | 处理方式 | 用户反馈 | 日志级别 |
|---------|---------|---------|---------|
| tool_result返回错误（API错误、参数错误、业务错误） | 将status标记为failed，error字段存储错误消息，停止计时，显示重试按钮 | 卡片红叉，显示错误摘要文字，重试按钮可点击，展开可见完整错误信息 | warn（含toolName、error message） |
| 工具调用超时（30s无tool_result返回） | 流结束时检查所有running状态工具，标记为failed，error="工具调用超时" | 红叉+"调用超时，请重试"+重试按钮 | warn（含toolName、超时时间） |
| toolCallId不匹配（收到result找不到对应call） | 忽略该result或创建unknown工具卡片（兜底） | 显示"未知工具"卡片，记录日志 | error（含orphan toolCallId） |
| 工具返回结果不是有效JSON | result存储为原始字符串，JsonHighlight降级显示为纯文本块 | 卡片正常显示完成状态，结果区域显示原始文本（标注"返回格式非JSON"） | info |
| 重试调用本身失败（网络错误） | 卡片保持failed状态，允许再次重试 | 同普通失败，可继续点击重试 | error（含重试错误信息） |
| 并行多个工具时一个失败 | 其他工具状态不受影响，独立更新 | 失败卡片显示错误，其他卡片正常运行/完成，用户仍可看到成功工具的结果 | -（设计如此） |
| 浏览器不支持setInterval（极端情况） | 降级为仅显示状态图标，不显示实时计时数字，完成后显示总耗时 | 运行中不显示跳动的计时数字（只显示spinner），完成后显示总耗时 | warn（记录userAgent） |
| JSON高亮库加载失败 | 降级为<pre>标签显示格式化缩进的纯文本JSON，无颜色高亮 | JSON内容缩进正确可读，只是没有语法颜色 | warn |

## 6. 测试策略

- **单元测试覆盖**：
  - `lib/ai/tool-display-map.ts`：已知工具名映射正确、未知工具名回退到unknown配置、工具名大小写兼容
  - `components/chat/blocks/tool-call-block/use-tool-timer.ts`：计时开始/停止正确、elapsedMs准确、interval清除无内存泄漏
  - JsonHighlight组件：JSON格式化缩进正确、不同类型着色正确、复制按钮工作
  - ToolCallBlock组件：三态渲染正确、点击展开折叠切换、重试按钮触发回调

- **集成测试覆盖**：
  - 流式工具调用流程：收到tool_call→显示运行中卡片+实时计时→收到tool_result→更新为完成/失败状态
  - 多工具并行：两个工具先后调用→两个独立卡片→各自独立计时/状态→一个失败不影响另一个
  - 重试流程：失败卡片→点击重试→卡片回到running→新结果返回→更新状态
  - 复制过滤：复制消息内容验证不包含工具调用文本
  - 导出过滤：导出DOM验证移除了tool-call-block元素

- **测试场景**：
  - **主流程**：发送需要搜索的消息→看到"联网搜索"卡片运行中（转圈+计时）→搜索完成绿勾显示耗时→点击展开→看到搜索参数和返回JSON→点击折叠
  - **失败路径1**：模拟工具调用失败→红叉+错误信息→点击重试→工具重新运行→成功完成
  - **失败路径2**：工具超时（30s无响应）→标记为失败→提示超时→可重试
  - **边界情况**：
    - 快速连续多个工具调用（并行3个以上）→验证卡片顺序和独立状态
    - 工具返回超大JSON（>50KB）→展开不卡顿，滚动正常
    - 流式过程中点击展开→params实时追加正确
    - 老年模式下卡片高度≥44px、图标足够大
    - 键盘Tab聚焦→Enter展开/折叠→Enter点击重试
    - 未知工具名→正确显示通用扳手图标+"工具调用"名称
  - **工具类型覆盖**：联网搜索、文件解析、处方生成、CGA评估、用药审查各类型图标正确

## 7. 安全考虑

- **XSS防护**：
  - 工具参数和结果JSON通过React渲染（文本转义），不使用dangerouslySetInnerHTML
  - JsonHighlight组件自行处理转义，不直接插入原始HTML
  - 错误信息文本同样转义显示，避免工具返回的恶意脚本注入

- **医疗安全**：
  - 工具卡片明确标注为"工具调用"，不与AI正文混淆
  - 联网搜索工具的结果展示（搜索结果卡片）由联网搜索模块负责，本组件只展示调用过程，不直接展示医疗内容
  - 失败状态明确提示，不让用户误以为工具成功执行
  - 工具卡片不导出到正式文档，避免中间过程信息干扰最终医疗建议

- **内容过滤**：
  - 工具参数和结果中可能包含敏感信息（API Key等不应出现在参数中，由调用方保证）
  - MVP阶段前端工具调用参数确实会暴露在浏览器中（PRD已记录此风险，二阶段后端代理解决）
  - 复制/导出时工具卡片被过滤，不会被用户意外分享出去

- **重试安全**：
  - 重试使用完全相同的参数重新调用，不修改参数，避免意外行为
  - 重试次数不做限制（用户可多次重试），但每次重试都是独立的API调用
  - 幂等性由工具本身保证（搜索、查询类天然幂等，副作用工具MVP阶段谨慎使用）

## 8. 可观测性

- **日志（console）**：
  - `debug`：工具调用开始（toolCallId、toolName）、工具结果返回（toolCallId、耗时ms、success/failure）
  - `info`：工具调用重试（toolCallId、重试次数）、未知工具名映射到unknown
  - `warn`：工具调用失败（toolName、error）、工具超时、orphan toolCallId、JSON解析降级
  - `error`：重试请求本身失败、SDK工具调用解析异常
  - 日志前缀：`[GerClaw][ToolCall]`

- **指标（可选，前端内存统计）**：
  - 各工具调用次数统计
  - 各工具平均耗时/成功率
  - 工具重试率
  - 用户展开工具详情的比例

- **追踪**：
  - 每个ToolCallBlock的toolCallId贯穿调用→结果→重试全流程，日志携带该ID
  - 失败日志包含完整错误栈和工具参数（便于排查API问题）

- **告警**：
  - MVP纯前端无服务端告警；工具失败通过UI红叉和重试按钮告知用户，用户自行重试

## 9. 技术选型说明

| 技术/库 | 用途 | 选型理由 | 替代方案 |
|---------|------|---------|---------|
| ShadCN/UI Collapsible | 展开折叠容器 | 项目已集成，基于Radix UI无障碍好，动画支持 | 纯CSS max-height transition（更轻量但需自行处理ARIA） |
| Lucide React图标 | 工具/状态图标 | 项目统一图标库，Search/FileText/Mic/Pill/Clipboard/Check/X/Loader等图标齐全 | 自定义SVG图标（增加维护成本）、Emoji（跨平台显示不一致） |

## 10. 开放问题

- **工具结果富渲染**：某些工具（如联网搜索）的结果不仅是JSON，还需要渲染为专门的UI卡片（搜索结果列表），本组件负责通用工具过程展示，结果内容是渲染JSON还是交给上层模块渲染富卡片？当前方案：本组件默认显示JSON详情，各工具模块可在消息流中额外插入自己的富渲染Block（如联网搜索的SearchResultsBlock），ToolCallBlock只展示"调用了搜索"这一过程
- **工具调用耗时阈值告警**：是否对执行特别慢的工具（>10s）显示特殊提示（如"仍在运行中，请稍候..."）？MVP暂不添加，因为运行中已有计时跳动表示还在工作
- **并行工具调用的顺序**：多个工具并行调用时，卡片按什么顺序排列？按SDK返回tool_call的顺序排列即可，结果返回时就地更新不重排序
- **工具调用取消**：长耗时工具是否支持取消？Vercel AI SDK和大多数API不支持中止已发出的工具调用请求，MVP不做取消功能
- **工具收藏/固定**：是否允许用户收藏常用工具或固定某些工具卡片？MVP不做，二阶段考虑
- **工具调用瀑布图/时间线视图**：多工具复杂调用时是否展示时间线视图？MVP每个工具独立卡片，多智能体二阶段后考虑树状/瀑布图可视化
