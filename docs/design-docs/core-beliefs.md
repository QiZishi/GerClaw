# Core Beliefs — 核心理念

> 项目的设计哲学和核心信念

---

## 1. 我们相信

- **医疗安全是不可逾越的底线**：宁可功能少做或提醒"不确定请就医"，也绝不能输出可能误导老年用户的错误医疗建议
- **适老化不是放大版普通UI**：老年人需要的是大字体、大按钮、语音优先、步骤提示、确认对话框，不是把年轻人的UI放大字号
- **简单优于复杂**：能用一个大麦克风按钮解决的，不要放三个小按钮；能用自然语言对话完成的，不要做表单
- **专业可信胜过花哨**：医疗产品的可信度来自循证引用、结构化输出、免责声明，不是酷炫动画
- **语音是患者端的第一入口**：老年人打字困难，语音识别+语音合成必须作为一等公民支持
- **流式输出是刚需不是加分项**：老人等不了30秒看一大段文字，逐字流式输出让他们边看边读
- **配置可变硬编码可耻**：API Key、模型名、URL、音色一切可配置，代码里不出现任何特定服务的硬编码
- **模块化先行快速验证**：MVP纯前端快速验证核心价值，二阶段平滑迁移到全栈架构，不做过度设计
- **参考资料是准绳**：严格遵循设计要求文档定义的功能和规范，不脱离需求自行发挥
- **工具调用透明可见**：AI搜索了什么、用了什么工具、思考过程是什么，用户都能看到，不做黑盒

## 2. 我们不做

- **不做确定性诊断**：AI不替代医生，不说"您得了XX病"，只说"建议关注/建议就医检查"
- **不编造医学知识或文献**：没有联网搜索时对不确定的问题说"我不确定"，不编造指南名称和文献引用
- **不硬编码API Key或模型名**：所有外部服务配置走环境变量，.env.example提供模板，真实Key不入Git
- **不读取gerclaw-main-origin和gerclaw-design-origin**：这两个目录禁止阅读和参考，必须从设计要求和官方参考资料独立开发
- **不做YOLO式提交**：每次代码变更必须build通过、核心路径测试通过，不能"应该能跑"就提交
- **不在患者端用小字体/低对比度/纯图标**：老年模式下≥18px正文、≥48px按钮、WCAG AA对比度、图标必有文字标签
- **不吞错误**：API失败、解析失败、存储失败都要如实记录和提示用户，不静默失败
- **不做前后端耦合**：MVP前端API调用层抽象为统一接口，为二阶段迁移Python后端预留适配层
- **不跳过AgentScope能力**：二阶段开发时，智能体相关能力优先查AgentScope参考是否已实现，已有的不重复造轮子
- **不引入不必要的依赖**：能用浏览器原生API(MediaRecorder/Web Audio/fetch)解决的，不引入额外npm包

## 3. 技术决策原则

| 原则 | 说明 |
|------|------|
| 医疗安全第一 | 任何技术决策如果可能影响医疗建议准确性，优先安全性 |
| 枯燥稳定技术优先 | 选择Next.js/React/Tailwind/ShadCN这些成熟方案，不为新而新 |
| 组合优于继承 | React组件用组合模式，hooks抽离逻辑，避免深层嵌套 |
| 显式优于隐式 | 配置显式声明在.env中，类型显式定义(TypeScript)，不做魔法自动推导 |
| 失败快优于静默失败 | 错误尽早抛出并展示给用户，不吞异常不假装成功 |
| 渐进增强 | 核心功能（文本对话）在最基础环境可用，语音/流式/动画是增强 |
| API层抽象 | 所有外部LLM/ASR/TTS/搜索调用封装在统一的API client层，切换服务不影响业务组件 |
| 类型安全 | TypeScript严格模式，API请求响应定义interface，any类型最小化 |
| 移动端优先思考 | 虽然MVP主要面向桌面/平板，但布局响应式适配手机 |
| 可访问性(a11y) | 语义化HTML、ARIA标签、键盘导航、屏幕阅读器支持 |

## 4. 代码品味

- 小组件，单一职责：一个组件只做一件事，超过200行考虑拆分
- 有意义的命名：组件名PascalCase，函数camelCase，常量UPPER_SNAKE，不用单字母变量（除循环i/j/k）
- 注释解释"为什么"，不是"做什么"：代码本身应该清楚做了什么，注释解释这样做的原因
- 错误处理是正常路径：try-catch是业务逻辑的一部分，不是事后补丁
- 样式用Tailwind原子类：不写内联style，不写独立CSS文件（除非全局样式）
- 状态管理最小化：useState/useContext够用就不引入Zustand/Redux
- 环境变量类型安全：用NEXT_PUBLIC_前缀暴露给前端的变量，统一在config模块读取和校验
- 不使用console.log生产代码：开发调试用console.debug，生产环境移除或用结构化日志
- 文件组织按功能模块：components/chat/、components/voice/、components/prescription/，不按类型放(components/buttons/等)
- 导入路径整洁：使用@/别名指向src/，不写深层相对路径(../../..)
