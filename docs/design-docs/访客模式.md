# 访客模式 — 设计文档

> 模块：访客模式 | 关联产品规格：[product-specs/访客模式.md](../product-specs/访客模式.md)

---

## 1. 设计目标

基于localStorage实现零门槛访客模式，满足以下设计目标：

1. **零 friction 入门**：无任何注册/登录流程，访问即使用，欢迎页仅需一次勾选即可进入，最大程度降低用户使用门槛
2. **纯本地数据存储**：所有数据（会话、消息、技能、设置）仅存浏览器localStorage，不上传服务器，符合隐私保护要求
3. **优雅降级**：localStorage不可用（隐私模式）或配额满时，自动降级为内存存储，保证核心功能可用，同时明确告知用户数据状态
4. **存储容量保护**：实时监控localStorage使用量，超过阈值主动提示并自动清理旧会话，避免QuotaExceededError导致功能不可用
5. **数据安全可清除**：提供一键清除所有数据功能，二次确认防止误操作，清除后状态干净可重新开始
6. **医疗合规前置**：首次访问必须主动勾选同意医疗免责声明才能使用，留存同意记录（本地时间戳）
7. **适老化设计**：欢迎页按钮≥48px，文字清晰，二次确认操作简单明确，适合老年用户使用

约束：
- MVP阶段不引入账号系统、不做云端同步，完全依赖localStorage
- localStorage单一key存储所有数据（便于原子读写、版本迁移）
- 二阶段迁移账号系统时，需考虑访客数据导入到账号的迁移路径
- 严格遵循PRD第4.2节P0功能清单和第10节约束风险

## 2. 架构设计

### 2.1 模块位置

在整体架构中的位置：**前端基础层**，位于`lib/storage/`和`components/visitor/`，在应用最初始化阶段加载，为所有上层模块提供本地存储能力。

- **上层依赖（被所有模块使用）**：
  - 通用对话模块：会话/消息持久化依赖本模块的localStorage存储
  - 技能管理模块：自定义技能持久化
  - 设置模块：主题/角色/字号等设置持久化
  - 所有功能模块：都依赖本模块的存储降级能力
- **本模块内部**：欢迎页组件、存储Provider、容量监控、设置面板、清除数据对话框、内存存储降级实现
- **下层依赖**：仅依赖浏览器原生localStorage API，无其他npm包依赖

## 3. 接口设计

### 3.1 对外接口

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `<VisitorStorageProvider>` | React Provider | `{ children: ReactNode }` | JSX.Element | 应用根组件包裹，初始化存储层，处理欢迎页显示 |
| `useVisitorStorage()` | React Hook | 无 | `VisitorStorageContextValue` | 获取存储实例、状态、操作方法 |
| `<WelcomeScreen>` | React组件 | `{ onAccept: () => void }` | JSX.Element | 欢迎页组件（一般由Provider自动渲染，无需手动调用） |
| `<SettingsPanel>` | React组件 | `{ open: boolean, onOpenChange: (open) => void }` | JSX.Element | 设置面板，可在侧边栏按钮触发 |
| `<StorageWarningBar>` | React组件 | 无 | JSX.Element \| null | 顶部警告条，自动根据storageStatus显示/隐藏 |
| `storage` 对象 | Adapter实例 | - | StorageAdapter | 直接操作存储的get/set/delete/clear，供各模块持久化使用 |

**StorageAdapter统一接口**：
```typescript
interface StorageAdapter {
  get<T>(key: string): Promise<T | null>;
  set<T>(key: string, value: T): Promise<void>;
  delete(key: string): Promise<void>;
  clear(): Promise<void>;
  getUsage(): Promise<StorageUsage>;
  readonly type: 'localStorage' | 'memory';
}
```

### 3.2 依赖接口

| 依赖 | 来源 | 用途 | 失败处理 |
|------|------|------|---------|
| `window.localStorage` | 浏览器原生 | 持久化存储 | 不可用时自动切换到MemoryStorageAdapter，显示顶部黄色警告条 |
| `JSON.parse/stringify` | JS原生 | 序列化/反序列化 | JSON解析失败（数据损坏）时重置为空数据，记录error日志，显示toast |
| ShadCN/UI Dialog/Checkbox/Button/Input | `components/ui/` | 欢迎页、设置面板、确认对话框UI组件 | 基础UI组件全局可用，无降级问题 |
| Sonner toast | `sonner`（ShadCN内置） | toast提示（存储警告、清理提示、错误提示） | toast组件全局可用 |

## 4. 数据设计

### 4.1 localStorage存储结构

使用单一key存储所有结构化数据，便于原子读写和版本迁移：

| Key | 类型 | 说明 |
|-----|------|------|
| `gerclaw_v1` | `GerClawLocalStorage` | 所有访客数据的根对象，JSON格式存储 |

**根对象字段说明**：
- `version`: 数据版本号，当前固定为`"1"`，未来结构变更时递增并编写迁移函数
- `disclaimerAccepted`: boolean，用户是否已同意免责声明
- `firstVisitAt`: number（时间戳），首次访问时间
- `lastActiveAt`: number，最后活跃时间（每次写入更新）
- `settings`: UserSettings对象，用户偏好设置
- `chat`: 通用对话模块的会话/消息数据（由对话模块管理，本模块不关心内部结构）
- `skills`: 技能管理模块的自定义技能数据（由技能模块管理）
- 其他模块可在根对象下扩展自己的命名空间，如`prescriptionDraft`、`cgaResults`等

**存储设计原则**：
- 单一key：避免多个key之间数据不一致，读取一次即可拿到所有数据
- 版本化：根level的version字段用于迁移
- 模块自治：各模块自己管理根对象下自己命名空间的数据结构，本模块只负责整体读写和容量管理

### 4.2 状态机

**存储层状态机**：

```
initializing → 检测localStorage → normal (LocalStorage正常)
     │              └─ localStorage不可用 → degraded (内存模式)
     ├─ normal状态下写入失败(QuotaExceeded) → 尝试AutoCleaner
     │   ├─ 清理成功 → 回到normal
     │   └─ 清理仍失败 → degraded (切换到内存)
     ├─ 用户点击clearAllData() → clearing → cleared → 刷新页面回到initializing
     └─ 数据损坏(JSON解析失败) → reset → normal (空数据)
```

**欢迎页状态机**：

```
检查disclaimerAccepted → false → showing_welcome (显示欢迎页遮罩)
     │     └─ true → hidden (不显示，直接进应用)
     └─ 用户勾选复选框并点击"开始使用" → writing_consent (写入标记) → hidden
```

## 5. 错误处理

| 错误类型 | 处理方式 | 用户反馈 | 日志级别 |
|---------|---------|---------|---------|
| localStorage完全不可用（隐私模式） | 立即切换到MemoryStorageAdapter，所有功能正常可用 | 顶部黄色警告条："当前为隐私浏览模式，数据仅临时保存，刷新页面后将丢失"，警告条带关闭按钮 | warn（记录userAgent） |
| localStorage写入抛出QuotaExceededError | 1. 调用AutoCleaner删除最早非固定会话；2. 重新尝试写入；3. 仍失败则切换到内存模式 | 清理成功：toast提示"已清理最早对话记录释放空间"；切换内存：顶部红色警告条 + toast"存储空间已满，已切换为临时模式" | error（记录已用字节数、已删除会话数） |
| localStorage读取JSON解析失败（数据损坏） | 捕获异常，重置为空数据（保留损坏数据到内存供调试，不写入） | Toast："本地数据格式错误，已重置" | error（记录原始损坏数据前缀，便于排查） |
| 数据版本号高于当前代码支持版本 | 不做强制迁移（避免数据丢失），降级为内存模式，提示用户 | Toast："检测到新版数据格式，当前版本无法读取，请升级应用" | warn（记录版本号） |
| 自动清理后仍空间不足 | 放弃清理，切换到内存模式 | 红色警告条："存储空间不足，新数据将不会被保存" | error |
| 用户输入"确认清除"错误 | 阻止清除，输入框标红 | 对话框内红色提示："输入有误，请输入'确认清除'" | -（用户输入错误） |
| 防抖写入期间页面卸载 | 使用beforeunload事件尝试同步写入（navigator.sendBeacon或直接同步setItem） | 无提示（尽力而为，大部分情况已被防抖写入覆盖） | warn |

## 6. 测试策略

- **单元测试覆盖**：
  - `lib/storage/local-storage-adapter.ts`：get/set/delete/clear、序列化/反序列化、容量计算
  - `lib/storage/memory-storage-adapter.ts`：内存读写正确性
  - `lib/storage/auto-cleaner.ts`：清理策略（按时间排序、删除最早非固定、目标大小达成）
  - `lib/storage/storage-monitor.ts`：阈值判断、警告触发、清理触发
  - 版本迁移函数（当前v1无需迁移，但框架需测试）

- **集成测试覆盖**：
  - 首次访问流程：访问→显示欢迎页→勾选禁用→勾选后按钮可用→点击进入→标记已同意→刷新不显示欢迎页
  - 清除数据流程：设置→点击清除→弹出对话框→输入错误→无法清除→输入正确→清除→刷新→回到欢迎页
  - 存储降级流程：模拟localStorage禁用→验证内存模式工作→验证警告条显示→刷新数据丢失
  - 容量管理：填充数据到>4MB→验证警告toast→填充到>4.5MB→验证自动清理→验证大小回到<4MB
  - 数据持久化：写入数据→刷新页面→数据仍存在

- **测试场景**：
  - **主流程**：首次访问→勾选免责→进入应用→发送几条消息→创建自定义技能→刷新页面→所有数据保留
  - **失败路径1**：在浏览器隐私模式下打开应用→验证警告条显示→发送消息→刷新→消息丢失（符合预期）
  - **失败路径2**：通过DevTools填满localStorage→验证自动清理触发→验证旧会话被删除→新消息可保存
  - **边界情况**：
    - 快速连续发送消息（验证防抖写入不丢失数据）
    - 写入过程中刷新页面（验证最后一条消息是否保存）
    - 手动修改localStorage为非法JSON→验证数据重置
    - 清除数据后所有状态重置（包括disclaimerAccepted标记）
  - **适老化**：欢迎页按钮高度≥48px，文字清晰可读

## 7. 安全考虑

- **医疗免责合规**：
  - 欢迎页必须展示完整医疗免责声明文本，用户必须主动勾选复选框才能点击"开始使用"
  - 记录同意时间戳（firstVisitAt）到localStorage，作为用户已阅读免责声明的凭证（本地留存）
  - 禁止跳过欢迎页直接进入应用（除非disclaimerAccepted为true）
  - 清除数据后必须重新同意免责声明

- **隐私保护**：
  - 不生成任何用户标识（无UUID、无设备指纹、无匿名ID）
  - 除用户主动发起的模型API调用外，不向任何第三方发送数据
  - 隐私说明文字在欢迎页和设置页面都清晰可见，明确告知用户数据仅存本地
  - localStorage中的数据不加密（MVP阶段），但不存储任何真实身份信息（姓名、身份证、手机号等）

- **数据清除安全**：
  - "清除所有数据"按钮使用红色警示样式
  - 二次确认对话框要求用户输入"确认清除"文字，而非简单点击确定，防止误触
  - 清除操作不仅清空gerclaw_v1，还清理所有gerclaw_前缀的key（防止未来多key遗留）
  - 清除后强制刷新页面，确保内存中状态也被重置

- **XSS/数据注入**：
  - localStorage读取的JSON内容在反序列化后不直接作为HTML渲染，所有内容通过React转义
  - 用户输入的会话标题、技能名称等在展示时经过React的XSS防护
  - 不使用eval或new Function解析localStorage内容，只用JSON.parse（带try-catch）

## 8. 可观测性

- **日志（console）**：
  - `info`：应用初始化（存储类型：localStorage/memory）、免责声明同意、数据清除成功、自动清理完成（删除会话数、释放空间）
  - `warn`：localStorage不可用降级内存、存储超过4MB警告、数据版本不兼容、页面卸载时数据可能未保存
  - `error`：QuotaExceededError写入失败、JSON解析失败（数据损坏）、自动清理后仍空间不足
  - 日志前缀：`[GerClaw][Storage]`

- **指标**：
  - 存储使用率百分比（每次写入后更新）
  - 会话数量统计
  - 自动清理触发次数/频率
  - 降级到内存模式的用户比例（MVP纯前端无法上报，仅在console可见）

- **追踪**：
  - 存储操作trace：每次set/delete记录key和数据大小（debug级别）
  - 自动清理trace：清理前大小、清理了哪些会话、清理后大小

- **告警**：
  - MVP纯前端无服务端告警，通过页面内警告条和toast实现用户侧"告警"
  - 红色警告条表示功能受限（内存模式），黄色为警告（隐私模式/空间不足）

## 9. 技术选型说明

| 技术/库 | 用途 | 选型理由 | 替代方案 |
|---------|------|---------|---------|
| localStorage原生API | 持久化存储 | PRD明确指定，浏览器原生无需依赖，简单键值对足够MVP需求，5-10MB容量满足文本会话存储 | IndexedDB（API复杂，异步编程成本高，MVP无需大容量/结构化查询；二阶段迁移账号系统时可引入Dexie.js封装） |
| React Context | 存储状态Provider | 存储状态是全局状态，所有模块都需要访问，Context API足够（存储状态变化不频繁：仅初始化/降级/清除时变化，写入不触发Context重渲染） | Zustand（如未来存储状态更新频繁可考虑切换，当前Context足够简单） |
| 单一key存储策略 | localStorage数据组织 | 便于原子读写（一次get/set所有数据）、便于版本迁移、避免多key不一致问题；写入时使用防抖（300ms）合并频繁更新，无性能问题 | 多key分模块存储（写入灵活但读取需多次IO，数据一致性难保证，迁移复杂） |
| 防抖写入（debounce 300ms） | 写入优化 | 发送消息等操作可能触发多次快速写入（消息更新、会话updatedAt更新），防抖合并减少setItem调用次数，避免性能问题和读写竞争 | 即时同步写入（每次状态变化立刻写入，性能差且可能导致SSE流式更新时频繁IO） |
| MemoryStorageAdapter降级 | 异常容错 | localStorage不可用时保证功能可用，用户虽然会丢失数据但能完成当前会话；优雅降级而非直接崩溃 | 直接显示错误页阻止使用（用户体验差，隐私模式是常见场景） |
| 输入文字确认清除 | 二次确认机制 | 比单纯的"确定/取消"对话框更能防止误操作，特别是针对老年用户点击不准确的情况；参考GitHub删除仓库的交互 | 普通确认对话框（容易误点确定）、邮箱/验证码确认（过重） |

## 10. 开放问题

- **localStorage可用容量探测**：不同浏览器localStorage实际容量不同（5-10MB不等），是否需要在初始化时做一次容量探测（写入大块数据测试）？还是保守使用4MB阈值？当前方案采用保守4MB阈值，避免探测带来的写入开销和副作用
- **二阶段访客数据迁移**：二阶段账号系统上线后，访客localStorage数据如何引导用户迁移到登录账号？当前暂不处理，待账号系统设计时明确迁移方案（如登录后提示"发现本地访客数据，是否导入到账号？"）
- **localStorage数据加密**：是否需要对存储的医疗对话内容进行对称加密（用户设置密码）？MVP阶段为了零门槛不引入密码，但考虑到医疗隐私敏感性，二阶段可考虑可选的本地加密选项
- **多标签页同步**：用户打开多个标签页时，localStorage变化可通过storage事件监听同步；是否需要实现多标签页状态同步？MVP暂不处理，多标签页可能导致数据覆盖（最后写入获胜），后续可通过storage事件实现跨标签页同步
- **数据损坏自动恢复**：JSON解析失败时当前直接重置，是否可以尝试从历史版本恢复（写入前保留一份bak）？可考虑写入前先备份到gerclaw_v1_bak，主数据损坏时尝试从bak恢复，提高可靠性
