# 适老化与角色切换 — 设计文档

> 模块：适老化与角色切换 | 关联产品规格：[product-specs/适老化与角色切换.md](../product-specs/适老化与角色切换.md)

---

## 1. 设计目标

为GerClaw MVP实现患者端/医生端双角色切换和患者端老年适老化UI系统，核心目标：
- **适老化强制默认**：患者端首次访问即开启老年模式，大字体≥18px、大按钮≥48px、WCAG AAA高对比度≥7:1，零配置可用
- **语音优先视觉引导**：老年模式下麦克风按钮为界面最大最醒目元素，视觉权重最高，引导老年用户语音交互
- **纯前端CSS变量驱动**：通过CSS自定义属性（CSS Variables）统一管控字号、按钮尺寸、间距、颜色、动画速度，切换老年模式无需重新渲染组件，仅切换根元素class
- **医生端专业模式锁定**：医生端始终标准模式（14px字号、高信息密度），老年模式开关在医生端完全隐藏且不生效
- **访客模式零门槛**：无登录页面，首次访问直接进入患者端老年模式，所有UI偏好持久化到localStorage
- **无障碍基础合规**：支持键盘导航、屏幕阅读器ARIA标签、尊重prefers-reduced-motion系统设置、焦点环高对比度可见
- **关键操作防误触**：老年模式下删除、切换角色等不可逆操作弹出大尺寸二次确认对话框
- **角色切换刷新式切换**：MVP阶段角色切换采用页面刷新方案（简单可靠），二阶段账号模式再做无刷新切换

**依赖关系**：

### 2.3 数据流

**应用初始化流程：**

```
应用加载 (app/layout.tsx)
    ↓
AccessibilityProvider初始化
    ├─ accessibilityStorage.load() 从localStorage读取状态
    │   ├─ 读取失败/无数据 → 使用默认值: {role: 'patient', elderMode: true, autoTTS: true, theme: 'light'}
    │   └─ 读取成功 → 使用保存的状态（医生端时强制elderMode=false）
    ├─ 检测window.matchMedia('(prefers-reduced-motion: reduce)') → 设置respectReducedMotion
    ├─ 检测window.matchMedia('(prefers-color-scheme: dark)') → 若未强制浅色则跟随系统
    └─ 根据当前状态设置document.documentElement.classList:
        ├─ role='patient' + elderMode=true → 添加'elder-mode'类，移除'doctor-mode'
        ├─ role='patient' + elderMode=false → 移除'elder-mode'和'doctor-mode'
        └─ role='doctor' → 添加'doctor-mode'类，移除'elder-mode'（老年模式强制关闭）
    ↓
渲染子组件
    ├─ 侧边栏底部根据role渲染RoleSwitcher和（仅患者端）ElderModeToggle
    ├─ 所有组件通过useAccessibility() Hook读取状态
    └─ 所有样式通过Tailwind类消费CSS变量（自动响应根元素class变化）
```

**老年模式开关切换流程（无刷新）：**

```
用户点击ElderModeToggle
    ↓
调用setElderModeEnabled(!elderModeEnabled)
    ↓
AccessibilityProvider状态更新
    ├─ accessibilityStorage.save({elderModeEnabled: newValue})
    ├─ newValue=true → document.documentElement.classList.add('elder-mode')
    │   └─ 强制主题为light（老年模式禁用深色）
    └─ newValue=false → document.documentElement.classList.remove('elder-mode')
    ↓
CSS变量即时生效：
    ├─ --font-size-base: 14px → 18px（或反向）
    ├─ --btn-min-height: 32px → 48px
    ├─ --color-*: 切换为高对比度色值
    └─ 所有使用CSS变量的组件自动重绘（无需组件重新mount）
```

**角色切换流程（刷新式）：**

```
用户点击RoleSwitcher
    ↓
ConfirmDialog.confirm({
    title: '切换到医生端',
    message: '切换后将重新加载页面，当前会话将保留。是否继续？',
    confirmText: '确认切换',
    cancelText: '取消'
})
    ↓ 用户确认
accessibilityStorage.save({currentRole: 'doctor'})
    ↓
window.location.reload()
    ↓
页面重新加载 → 初始化流程读取currentRole='doctor'
    ├─ 添加'doctor-mode'类，移除'elder-mode'
    ├─ 侧边栏隐藏ElderModeToggle
    ├─ 显示PatientList入口
    ├─ WelcomePage渲染医生端版本
    └─ 输入框placeholder切换为医生端文案
```

**删除会话二次确认流程：**

```
用户点击删除会话按钮
    ↓
判断当前模式 → 老年模式下必须走ConfirmDialog
    ↓
ConfirmDialog.confirm({
    title: '确认删除对话',
    message: '删除后这个对话的所有内容将无法恢复，您确定要删除吗？',
    confirmText: '确认删除',
    cancelText: '取消',
    danger: true
})
    ↓ 用户点击"确认删除"
执行删除操作
    ↓
Toast反馈（老年模式下Toast文字≥18px，显示时间延长至4秒）
```

## 3. 接口设计

### 3.1 对外接口

**useAccessibility Hook：**

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `currentRole` | state | - | `UserRole` | 当前角色：'patient' \| 'doctor' |
| `isDoctor` | derived | - | `boolean` | 是否医生端（便于条件渲染） |
| `isPatient` | derived | - | `boolean` | 是否患者端 |
| `elderModeEnabled` | state | - | `boolean` | 老年模式是否开启（仅患者端有效，医生端始终false） |
| `autoTTSPlayback` | state | - | `boolean` | AI回复是否自动朗读（老年模式默认true） |
| `themeMode` | state | - | `ThemeMode` | 主题模式（老年模式强制'light'） |
| `respectReducedMotion` | state | - | `boolean` | 是否尊重减少动画偏好 |
| `setElderModeEnabled` | function | `enabled: boolean` | `void` | 切换老年模式开关，自动保存localStorage和更新DOM class |
| `setAutoTTSPlayback` | function | `enabled: boolean` | `void` | 切换自动朗读开关 |
| `switchRole` | function | `role: UserRole` | `Promise<boolean>` | 请求切换角色（弹出确认框，确认后刷新页面），返回是否确认切换 |
| `confirmAction` | function | `options: ConfirmOptions` | `Promise<boolean>` | 弹出确认对话框，返回用户是否确认（老年模式自动使用大尺寸对话框） |

**ConfirmDialog组件调用方式（命令式）：**

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `ConfirmDialog.confirm` | static async function | `options: ConfirmOptions` | `Promise<boolean>` | 弹出确认对话框，用户确认resolve(true)，取消/关闭resolve(false) |

**ConfirmOptions类型：**
```typescript
interface ConfirmOptions {
  title: string;
  message: string;
  confirmText?: string;  // 默认"确认"
  cancelText?: string;   // 默认"取消"
  danger?: boolean;      // 是否危险操作（红色确认按钮）
  size?: 'normal' | 'large';  // 老年模式自动使用large
}
```

### 3.2 依赖接口

| 依赖 | 来源 | 用途 | 失败处理 |
|------|------|------|---------|
| `localStorage` | 浏览器原生API | 持久化角色和偏好设置 | try-catch包裹，失败时使用内存状态，刷新后恢复默认值，console.warn记录 |
| `window.matchMedia` | 浏览器原生API | 检测prefers-reduced-motion和prefers-color-scheme | 不支持时默认respectReducedMotion=true（安全降级，禁用动画），主题默认light |
| `window.location.reload` | 浏览器原生API | 角色切换后刷新页面 | 极端情况（如iframe沙箱限制）下提示用户手动刷新 |
| `document.documentElement.classList` | DOM API | 添加/移除'elder-mode'/'doctor-mode'类名切换CSS变量 | SSR Next.js环境下useEffect中执行（避免hydration不匹配） |
| `Toast`组件 | 全局UI组件 | 操作成功/失败提示 | Toast不可用时降级为console.log，关键错误用alert兜底 |
| `ChatContext` | 对话模块Context | 删除会话、切换会话等操作触发确认对话框 | 对话模块依赖本模块的confirmAction接口，而非反向依赖 |
| `useTTSPlayer` | 语音交互模块Hook | 老年模式下AI回复完成后自动触发TTS播放 | 监听AI消息完成事件，autoTTSPlayback为true时调用togglePlay |

## 4. 数据设计

### 4.1 本地存储

**localStorage键值设计：**

| 键名 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `gerclaw_current_role` | `'patient' \| 'doctor'` | `'patient'` | 当前角色 |
| `gerclaw_elder_mode` | `'true' \| 'false'` | `'true'` | 老年模式开关（仅患者端有效） |
| `gerclaw_auto_tts` | `'true' \| 'false'` | `'true'` | AI回复自动朗读开关 |
| `gerclaw_theme_mode` | `'light' \| 'dark' \| 'system'` | `'light'` | 主题偏好（老年模式强制覆盖为'light'） |

**存储注意事项：**
- localStorage只能存字符串，boolean值用'true'/'false'字符串存储
- 读取时做类型校验，非法值回退到默认值
- 所有读写操作try-catch包裹（隐私模式下localStorage可能抛出异常）
- 医生端状态下忽略gerclaw_elder_mode值，始终强制为false

### 4.2 状态机

**老年模式状态流转：**

```
初始加载
    ↓
读取localStorage → 无数据/非法值 → ELDER_ON（默认）
    ↓              有数据 → ELDER_ON / ELDER_OFF
    ↓
用户点击Toggle开关
    ↓
ELDER_ON ──点击──→ ELDER_OFF
ELDER_OFF ──点击──→ ELDER_ON
    ↓                   ↓
更新DOM class ──────→ 更新DOM class
保存localStorage       保存localStorage
```

**角色状态流转（医生↔患者）：**

```
首次访问 → PATIENT_MODE（默认老年模式开启）
    │
    │ 点击"切换到医生端" → 确认对话框
    │                          ↓ 确认
DOCTOR_MODE ←──── 页面刷新 ←───┘
    │
    │ 点击"切换到患者端" → 确认对话框
    │                          ↓ 确认
PATIENT_MODE ←─── 页面刷新 ←───┘
```

**确认对话框状态：**

```
closed ──调用confirm()──→ opening ──动画完成──→ open
                                              │
                                    ┌─────────┴─────────┐
                                    ↓                   ↓
                                点击取消            点击确认
                                    │                   │
                                    └─────────┬─────────┘
                                              ↓
                                         closing ──动画完成──→ closed
                                              ↓
                                    Promise resolve(false/true)
```

## 5. 错误处理

| 错误类型 | 处理方式 | 用户反馈 | 日志级别 |
|---------|---------|---------|---------|
| localStorage读写失败（隐私模式/配额满） | try-catch捕获，使用内存中React state作为fallback，不持久化 | 无用户提示（静默降级），刷新后恢复默认模式 | warn |
| localStorage值非法（如手动篡改为undefined/null） | 类型校验失败时回退到默认值，覆盖写入正确默认值 | 无用户提示，自动恢复默认 | warn |
| matchMedia API不支持（旧浏览器） | 默认为respectReducedMotion=true（安全降级，禁用动画），主题默认light | 无用户提示 | warn |
| 角色切换后刷新被浏览器阻止 | 捕获异常，显示提示引导用户手动刷新 | Alert："页面刷新被阻止，请手动按F5刷新页面以完成切换" | error |
| SSR/hydration不匹配（Next.js服务端渲染时无window对象） | 所有DOM class操作在useEffect客户端侧执行，初始渲染使用与客户端一致的默认状态（患者端老年模式），避免hydration warning | 无用户感知（hydration正确） | info |
| 用户在医生端通过DevTools手动添加.elder-mode类 | CSS通过.doctor-mode选择器优先级覆盖老年模式变量（:root.doctor-mode强制标准变量，特异性高于.elder-mode），确保医生端始终标准 | 无用户提示，医生端UI不受影响 | info |
| ConfirmDialog连续快速调用 | 单例模式，同时只允许一个对话框打开，后续调用加入队列或拒绝（返回false） | 第二个confirm调用立即resolve(false)，不弹出第二个对话框 | warn |
| 自动TTS播放时语音模块未就绪/失败 | try-catch包裹自动播放调用，失败静默，不阻塞消息显示流程 | 无错误弹窗打扰老年用户；用户可手动点击播放按钮重试；控制台日志记录TTS失败原因 | warn |

## 6. 测试策略

- **单元测试覆盖**：
  - `accessibilityStorage.ts`：localStorage读写、默认值兜底、非法值处理、异常捕获
  - `AccessibilityProvider`：状态初始化、状态更新、class切换逻辑（用jsdom模拟document）
  - `ConfirmDialog`：open/close状态流转、Promise resolve正确、Esc/遮罩点击取消
  - CSS变量：验证.elder-mode类下变量值正确（计算样式检查）

- **集成测试覆盖**：
  - 老年模式Toggle切换后字号/按钮尺寸变化（getComputedStyle验证）
  - 医生端不渲染ElderModeToggle组件
  - 角色切换保存localStorage并触发刷新（mock window.location.reload）
  - 键盘导航：Tab可聚焦所有交互元素，Enter触发按钮，Esc关闭对话框

- **手工测试场景**：
  - 主流程1（首次访问）：清空浏览器数据→访问应用→默认进入患者端→检查字号18px+、按钮48px+、麦克风超大按钮
  - 主流程2（关闭老年模式）：点击侧边栏老年模式开关→字号缩小到14px→按钮缩小→主题可切换深色
  - 主流程3（切换医生端）：点击"切换到医生端"→弹出确认框→确认→页面刷新→医生端加载→14px字号→可见患者列表→无老年模式开关
  - 主流程4（切换回患者端）：医生端点击"切换到患者端"→确认→刷新→回到患者端老年模式
  - 主流程5（二次确认）：在老年模式下点击删除对话→弹出大确认框→取消不删除→确认则删除
  - 失败路径1（localStorage禁用）：浏览器隐私模式/禁用localStorage→切换老年模式/角色→功能正常但刷新后恢复默认
  - 失败路径2（prefers-reduced-motion）：系统开启减少动画→应用动画全部禁用
  - 边界情况1（深色模式+老年模式）：系统深色模式→开启老年模式→仍显示浅色高对比度主题
  - 边界情况2（键盘导航）：Tab键遍历所有交互元素→焦点环清晰可见→Enter触发→Esc关闭对话框
  - 边界情况3（对比度验证）：使用对比度检测工具验证正文/背景对比度≥7:1
  - 边界情况4（刷新持久化）：切换医生端→关闭标签页→重新打开→仍为医生端
  - 边界情况5（医生端防老年模式）：医生端通过DevTools添加.elder-mode类→UI仍为标准模式字号
  - 边界情况6（自动TTS）：老年模式下发送消息→AI回复完成→自动开始语音播放

## 7. 安全考虑

- **访客隐私保护**：MVP访客模式不收集任何PII（个人身份信息），localStorage仅存储UI偏好和会话文本，不存储姓名/身份证/手机号等真实身份信息
- **localStorage安全**：不存储敏感数据（如API Key、健康档案），仅存储UI偏好（角色/开关状态）；XSS攻击无法通过本模块获取敏感信息
- **高对比度视力保护**：老年模式颜色组合经过WCAG AAA对比度验证（≥7:1），确保白内障/黄斑变性等视力障碍老年用户可阅读；不使用颜色作为唯一信息传达方式（图标+文字双重编码）
- **防误触机制**：老年模式下关键操作（删除、角色切换）强制二次确认，确认按钮文字明确（不用模糊的"确定"/"是/否"），取消按钮视觉上与确认按钮同等明显（不做灰色弱化，避免老人找不到取消）
- **自动播放音量控制**：老年模式自动TTS播放时设置合理初始音量（Web Audio API gainNode=0.7），不突然播放大音量惊吓用户；浏览器自动播放策略限制下首次需要用户交互，但老年模式默认开启视为用户主动设置
- **DOM class优先级防护**：通过CSS特异性（specificity）确保.doctor-mode的样式优先级高于.elder-mode，防止用户或扩展程序篡改class导致医生端误显示老年模式
- **键盘焦点安全**：所有交互元素有清晰的焦点环（3px高对比度outline），键盘用户不会丢失焦点位置；模态框打开时焦点陷阱（focus trap）在对话框内，Tab不会跑到背景页面
- **无自动外部请求**：角色切换和老年模式切换是纯前端状态变化，不触发任何外部API请求，不泄露用户角色/偏好信息给第三方

## 8. 可观测性

- **日志**：
  - info级别：应用初始化（角色/老年模式/主题状态）、角色切换请求与确认、老年模式开关切换、确认对话框展示/确认/取消
  - warn级别：localStorage读写失败、localStorage值非法、matchMedia不支持、自动TTS播放失败
  - error级别：角色切换刷新失败、关键DOM操作异常
  - 日志格式：`[Accessibility] Initialized: role=patient, elderMode=true, theme=light`

- **指标（前端）**：
  - `accessibility_mode_distribution`：用户角色和模式分布（用于了解患者/医生端使用比例）
  - `elder_mode_toggle_count`：老年模式开关切换次数（了解多少用户关闭老年模式）
  - `role_switch_count`：角色切换次数
  - `confirm_dialog_accept_rate`：确认对话框确认率（确认/取消比例，判断提示是否过度）
  - `css_variable_switch_latency`：切换老年模式到样式生效的耗时（目标<100ms）

- **追踪**：
  - 每个会话初始化时记录初始模式状态，贯穿整个会话
  - 角色切换事件记录from/to角色和是否确认

- **告警（MVP阶段无后端，仅控制台）**：
  - localStorage完全不可用（内存模式）：console.warn提示用户数据不会持久化
  - 检测到不支持CSS变量的极旧浏览器：console.error提示升级浏览器
  - 连续3次角色切换失败：console.error提示检查是否有脚本阻止页面刷新

## 9. 技术选型说明

| 技术/库 | 用途 | 选型理由 | 替代方案 |
|---------|------|---------|---------|
| CSS Custom Properties (CSS Variables) | 字号/颜色/间距/动画统一管控 | 浏览器原生支持（Chrome/Edge/Safari 99%+覆盖），切换class即时生效无需组件重渲染，性能极佳，可在DevTools中直接调试 | CSS-in-JS（styled-components emotion）运行时注入样式性能开销大；Sass变量编译时确定无法动态切换；Tailwind CSS theme扩展基于CSS变量实现，两者配合使用而非替代 |
| React Context + Hooks | 全局状态管理 | 适老化状态是全局低频变更状态（切换角色/模式才变），Context足够，无需引入额外状态管理库；useAccessibility Hook提供类型安全的访问方式 | Zustand/Redux（对于低频UI偏好状态过度设计，增加包体积和复杂度，不采用） |
| Tailwind CSS v4 | 原子化样式 + CSS变量消费 | Tailwind配置可将CSS变量映射为theme token，所有组件用`text-base`/`p-4`/`btn`等原子类自动响应模式切换，无需手写媒体查询或条件类名 | 纯CSS写所有样式（重复代码多，维护困难）；CSS Modules（需每个组件写多套样式适配老年模式，不采用） |
| 命令式ConfirmDialog | 二次确认对话框 | Promise式API（const confirmed = await confirmAction(...)）比声明式状态更易用，关键操作处一行代码即可弹出确认框，避免每个组件维护open/setOpen状态 | 声明式Dialog（每个使用处需useState管理open状态，样板代码多，老年模式/标准模式尺寸判断重复，不采用） |
| window.location.reload | 角色切换 | MVP阶段角色切换采用刷新方案最简单可靠，避免复杂的组件重载/状态重置/内存泄漏问题；刷新后从零初始化保证状态一致性 | 无刷新角色切换（需完整重置所有Context状态、清空对话相关state、重新渲染所有组件，复杂度高且易出bug，MVP阶段不采用，留待二阶段账号模式实现） |
| localStorage | UI偏好持久化 | 浏览器原生，简单可靠，5MB容量足够存储UI偏好；无需IndexedDB等复杂存储 | IndexedDB（对于几个键值对过度设计）；Cookie（每次请求携带，浪费带宽，不采用） |
| matchMedia API | 系统偏好检测 | 浏览器原生，实时响应系统设置变化（prefers-reduced-motion/prefers-color-scheme） | 用户手动设置（增加用户负担，不如自动检测系统偏好） |

## 10. 开放问题

- **自动TTS播放的范围和时机**：老年模式下是所有AI回复都自动播放，还是≥50字的长回复才自动播放？短回复（如"好的"/"明白了"）自动播放可能频繁打扰用户。建议：≥20字才自动播放，短回复仅显示播放按钮不自动播。
- **老年模式开关的位置和文案**：目前设计为侧边栏底部开关，文案"老年模式"+辅助说明"大字体、大按钮、语音优先"。是否需要在欢迎页也提供一个明显的老年模式开关入口，方便老年用户的家属快速调整？
- **确认对话框的老年模式尺寸**：老年模式确认框按钮≥48px已确定，但模态框宽度和文字大小是否需要进一步放大（如宽度90%屏幕宽、标题24px、正文20px）？需实际老年用户测试后调整。
- **医生端是否允许临时开启大字体**：PRD明确"医生端始终标准专业模式"，但不排除有视力不佳的老年科医生需要大字体。是否在医生端设置中提供"字号调节"（而非完整老年模式）？建议MVP严格锁定医生端标准模式，医生反馈需求后二阶段再考虑。
- **模拟患者数据的丰富度**：MVP医生端患者列表使用纯前端硬编码模拟数据，需要多少条？5条/10条？是否需要包含头像、最近CGA评分等更丰富的字段？建议：8条示例数据，覆盖不同风险等级和常见老年疾病。
- **二阶段无刷新角色切换**：MVP采用刷新方案，二阶段账号模式下是否需要实现无刷新切换？如果需要，当前AccessibilityProvider设计是否需要预留接口？当前设计已经将状态集中在Context中，二阶段只需去掉reload()调用改为状态驱动的组件树重置即可。
- **老年模式下的Toast持续时间**：普通模式Toast显示2-3秒，老年模式是否需要延长到4-5秒以便老人读完？建议：老年模式Toast显示4秒，且文字≥18px。
