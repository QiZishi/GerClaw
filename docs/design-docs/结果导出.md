# 结果导出 — 设计文档

> 模块：结果导出 | 关联产品规格：[product-specs/结果导出.md](../product-specs/结果导出.md)

---

## 1. 设计目标

基于纯前端技术栈实现多格式导出能力，满足以下设计目标：

1. **零后端依赖**：所有导出逻辑在浏览器端完成，使用成熟的前端库（jsPDF、docx.js），不需要后端服务支持，符合MVP纯前端静态导出架构
2. **统一导出模板**：三类内容（对话/处方/CGA报告）共享相同的头部模板（标题+日期+免责声明），确保导出文档格式规范、医疗免责声明不遗漏
3. **内容安全过滤**：导出时自动过滤思维链、工具调用卡片、内部调试信息，只导出用户可见的最终内容，避免敏感中间信息泄露
4. **中文兼容优先**：PDF采用html2canvas截图方案规避中文字体嵌入问题，确保所有平台中文显示一致
5. **API简洁可复用**：封装统一的`exportContent`函数，三种格式导出实现为独立策略，供五大处方、CGA评估、通用对话三个模块调用
6. **适老化兼容**：导出按钮尺寸≥48px，点击区域足够大，老年模式下按钮放大，加载状态文字清晰

约束：
- 严格遵循PRD技术栈：Next.js 15、React 18、Tailwind CSS 4、ShadCN/UI
- 导出库版本：jspdf、html2canvas、docx（均为latest稳定版）
- 二阶段迁移FastAPI后端时，导出功能可保留前端实现，也可扩展为后端渲染（当前前端实现足够MVP需求）

## 2. 架构设计

### 2.1 模块位置

在整体架构中的位置：**前端通用工具层**，位于`lib/export/`目录，作为可复用工具服务于上层功能模块。

- **上层依赖（使用者）**：
  - 五大处方模块（右侧预览面板）：`components/prescription/prescription-panel.tsx`
  - CGA评估模块（右侧报告面板）：`components/cga/cga-report-panel.tsx`
  - 通用对话模块（消息操作/会话详情）：`components/chat/message-actions.tsx`
- **本模块内部**：导出入口组件、格式选择菜单、三种格式导出策略、模板生成工具、Blob下载工具
- **下层依赖**：无特殊后端依赖，仅使用浏览器API和npm库

### 2.2 组件划分

| 组件 | 职责 | 文件位置（预期） |
|------|------|----------------|
| **ExportButton** | 导出按钮组件，包含按钮UI、下拉菜单、加载状态管理，点击后调用对应导出函数 | `components/export/export-button.tsx` |
| **ExportFormatMenu** | 格式选择下拉菜单，展示三种格式选项，点击触发onSelect回调 | `components/export/export-format-menu.tsx` |

| **exportToMarkdown** | Markdown格式导出策略：拼接模板字符串，生成Markdown Blob | `lib/export/strategies/markdown.ts` |
| **exportToDocx** | DOCX格式导出策略：使用docx.js构建Document，转换为Blob | `lib/export/strategies/docx.ts` |
| **exportToPdf** | PDF格式导出策略：使用html2canvas渲染目标DOM，jsPDF分页生成PDF | `lib/export/strategies/pdf.ts` |
| **buildExportTemplate** | 导出模板构建工具：生成统一的标题/日期/免责声明头部 | `lib/export/template.ts` |

## 3. 接口设计

### 3.1 对外接口

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `<ExportButton />` | React组件 | `{ contentType: ExportContentType, contentRef?: React.RefObject<HTMLElement>, markdownContent?: string, title: string }` | JSX.Element | 导出按钮组件，contentRef用于PDF/DOCX（DOM截图），markdownContent用于Markdown导出 |
| `exportContent(config: ExportConfig): Promise<void>` | 异步函数 | config: ExportConfig（见数据模型） | Promise<void> | 统一导出函数，可在代码中直接调用（不通过UI按钮） |
| `EXPORT_DISCLAIMER_TEXT` | 常量 | - | string | 标准医疗免责声明文本："本建议由AI生成，仅供参考，不能替代专业医生的诊断和治疗建议。如有不适请及时就医。" |

### 3.2 依赖接口

| 依赖 | 来源 | 用途 | 失败处理 |
|------|------|------|---------|
| `jsPDF` | npm包 `jspdf` | PDF文档生成 | html2canvas+jsPDF组合使用，如jsPDF加载失败则提示用户PDF暂不可用，建议使用其他格式 |
| `html2canvas` | npm包 `html2canvas` | DOM元素截图为canvas，解决中文字体问题 | 截图失败时降级尝试纯文本PDF生成，仍失败则提示导出失败 |
| `docx` | npm包 `docx` | DOCX文档生成（Document、Packer、Paragraph、TextRun、Table等） | docx.js加载失败时提示Word导出暂不可用 |
| `Blob` API | 浏览器原生 | 创建二进制文件对象 | 不支持Blob的浏览器显示升级提示 |
| `URL.createObjectURL` | 浏览器原生 | 创建Blob下载链接 | 降级为data URI（大文件可能有问题），失败则提示 |
| ShadCN/UI组件 | `components/ui/` | Button、DropdownMenu、Toast提示 | 基础UI组件已全局可用，无降级问题 |

## 4. 数据设计

### 4.1 数据库表/集合（如适用）

本模块为纯前端工具，无持久化存储需求，不涉及数据库表或localStorage存储（导出文件直接下载到用户本地文件系统）。

### 4.2 状态机

**导出按钮状态机**：

```
idle（可点击）→ 用户点击格式 → loading（导出中，按钮禁用）
     ├─ 导出成功 → success（显示成功toast）→ idle
     └─ 导出失败 → error（显示错误toast）→ idle
```

## 5. 错误处理

| 错误类型 | 处理方式 | 用户反馈 | 日志级别 |
|---------|---------|---------|---------|
| html2canvas渲染DOM失败 | 捕获异常，降级：1）尝试重试1次；2）仍失败则回退到纯文本PDF（使用jsPDF.text()，中文可能显示为方块）；3）回退也失败则抛出错误 | 显示toast："PDF导出失败，建议使用Markdown或DOCX格式" | error（含错误栈、DOM大小信息） |
| jsPDF/docx.js库加载失败（chunk加载错误） | 动态import失败，提示用户刷新页面 | Toast："导出组件加载失败，请刷新页面重试" | error（记录chunk路径） |
| Blob/URL API不支持（极旧浏览器） | 检测API存在性，不支持时提前返回 | 显示提示："您的浏览器版本过旧，不支持文件导出，请升级到最新版Chrome/Edge/Safari" | warn（记录userAgent） |
| 导出内容为空 | 前置检查，content为空时禁用按钮 | 按钮灰显，hover提示"暂无可导出内容" | -（正常状态） |
| 导出内容过大（DOM尺寸>5000px高或生成Blob>20MB） | 提示用户内容较大，继续导出可能需要时间 | Toast："内容较大，导出中请稍候..."；导出仍在后台继续 | info（记录内容大小） |
| 下载被浏览器拦截（弹窗拦截） | 检测<a>点击后一段时间内是否触发下载，未触发则提示用户 | 显示提示："下载可能被拦截，请检查浏览器下载设置或允许弹窗" | warn |
| DOCX中文字体问题（非错误，显示差异） | 不做特殊处理，docx.js使用系统默认字体，中文可正常显示 | 无用户提示，文档可正常打开编辑 | - |
| sanitize清理DOM时找不到对应元素 | 静默跳过，不影响导出 | 无感知 | warn（记录未找到的选择器） |

- **单元测试覆盖**：

  - `lib/export/sanitize.ts`：sanitizeExportContent正确移除思维链、工具调用卡片元素
  - `lib/export/strategies/markdown.ts`：Markdown拼接格式正确，标题层级正确
  - 文件名生成测试：验证不同contentType和日期生成的文件名符合"GerClaw_{title}_{date}.{ext}"格式

- **集成测试覆盖**：
  - ExportButton组件渲染：按钮显示正确，菜单点击弹出选项
  - 完整导出流程：点击PDF→加载状态→Blob生成→下载触发
  - 三种格式分别验证：可正常生成Blob，Blob大小合理（非空非0字节）
  - 内容过滤验证：导出的DOM/文本不包含thinking-block、tool-call-block等类名元素

- **测试场景**：
  - **主流程**：
    - 导出五大处方为PDF：文件下载成功，PDF可打开，包含标题/日期/免责声明/处方内容，中文正常
    - 导出CGA报告为DOCX：Word文档可正常打开，表格完整，格式整齐
    - 导出对话记录为Markdown：.md文件可在编辑器打开，对话格式正确
  - **失败路径1**：模拟html2canvas失败（传入无效DOM），验证降级逻辑和错误提示
  - **失败路径2**：空内容时导出按钮禁用，点击无效
  - **边界情况**：
    - 超长内容（>20页）导出，验证分页正确，内容不截断
    - 包含表格/列表/图片/代码块的内容导出，格式基本保持
    - 老年模式下导出按钮尺寸足够大（≥48px）
    - 快速连续点击导出按钮（验证防抖，不重复触发）
  - **内容验证**：导出文件中不包含"思考过程"、"工具调用"等内部信息

## 7. 安全考虑

- **内容过滤安全**：
  - sanitizeExportContent函数必须移除所有带有`thinking-block`、`tool-call-block`类名的元素，防止模型思考过程和工具调用细节意外导出
  - 移除所有`data-internal="true"`标记的内部调试元素
  - Markdown导出时过滤思维链文本（thinking block内容不拼接进Markdown）

- **XSS防护**：
  - DOCX导出使用docx.js的API构建文档（不直接拼HTML），避免XSS
  - PDF使用html2canvas截图，截图是静态图片，不执行脚本
  - Markdown导出为纯文本拼接，不渲染HTML

- **医疗合规**：
  - 所有导出文件必须包含固定的医疗免责声明，禁止用户配置移除
  - 免责声明文本使用灰色背景/斜体样式，在文档中清晰可见
  - 导出日期自动添加，确保文档有时间标识

- **数据隐私**：
  - 纯前端导出，所有内容处理在浏览器内存中完成，不上传到任何服务器
  - 导出完成后立即调用`URL.revokeObjectURL`释放Blob内存，避免内存泄漏
  - 不记录导出日志到localStorage或任何持久化存储

## 8. 可观测性

- **日志（console）**：
  - `info`：导出开始（格式、内容类型、内容大小）、导出成功（耗时ms）
  - `warn`：浏览器API缺失、下载可能被拦截、sanitize未找到元素、大文件导出
  - `error`：html2canvas/jsPDF/docx异常、库加载失败
  - 日志前缀：`[GerClaw][Export]`

- **指标**：
  - 各格式导出耗时（ms）
  - 导出成功率/失败率（按格式分类）
  - 平均导出文件大小（KB）

- **追踪**：
  - 每次导出生成唯一exportId，贯穿整个流程日志
  - 失败时记录错误类型、堆栈信息、浏览器userAgent（辅助定位兼容性问题）

- **告警**：
  - MVP纯前端无服务端告警，通过用户可见的toast提示实现错误反馈
  - 导出失败后用户可手动重试或选择其他格式

## 9. 技术选型说明

| 技术/库 | 用途 | 选型理由 | 替代方案 |
|---------|------|---------|---------|
| jsPDF | PDF文档生成 | 前端PDF生成最成熟的库，社区活跃，文档完善，支持添加图片/文本/分页，配合html2canvas可完美解决中文显示问题 | pdfmake（声明式API但中文支持需额外配置字体）、react-pdf（需要React元素渲染，DOM已存在时截图方案更简单） |
| html2canvas | DOM截图为canvas | 前端DOM截图标准方案，支持复杂CSS样式（Tailwind/ShadCN样式均可正确截取），截图为图片后嵌入PDF避免中文字体嵌入问题 | dom-to-image（维护不活跃）、Puppeteer（需要后端，MVP不可用） |
| docx (docx.js) | DOCX生成 | 纯前端生成Word文档的主流库，支持段落/表格/样式/标题层级，API友好，生成的.docx兼容Word和WPS | html-docx-js（将HTML转DOCX，样式保留差）、直接生成RTF（格式支持有限） |
| Blob + URL.createObjectURL | 前端文件下载 | 浏览器原生标准API，支持大文件流式下载，兼容性好（Chrome/Edge/Safari全部支持） | FileSaver.js（封装良好但原生API已足够，减少依赖）、data URI（大文件base64编码体积膨胀33%） |
| 纯前端实现 | 整体架构 | PRD明确MVP纯前端，无后端服务；导出功能适合端侧完成，减少服务器压力，保护用户隐私 | 后端渲染（二阶段可考虑：Chromium无头浏览器渲染PDF效果更好，但MVP无后端） |

## 10. 开放问题

- **PDF分页控制**：html2canvas截图后添加到jsPDF时，如何优雅处理页面分割（避免内容在页中被截断）？需实现简单的分页算法：按A4高度计算，在合适的位置（如段落间距处）分页，而非硬性切割
- **PDF中文字体**：虽然html2canvas截图方案规避了字体嵌入问题，但文件体积会比纯文本PDF大；是否需要在二阶段优化为可选的文本PDF（嵌入中文字体）待评估
- **DOCX模板样式**：DOCX导出的具体样式（标题字号、正文行距、表格边框样式）需与UI设计对齐，当前先实现功能可用，细节样式在UI阶段微调
- **处方/报告的打印样式**：导出的DOM元素在html2canvas截图时是否需要专用的打印样式类（如`print-optimized`）来优化导出效果？预留class钩子，UI优化时添加
- **导出进度指示**：大文件导出时是否需要显示进度条？目前仅显示"导出中..."，如后续测试发现大文件导出时间>5s可考虑添加进度指示（html2canvas无进度回调，只能用估算）
