# 语音交互 — 设计文档

> 模块：语音交互 | 关联产品规格：[product-specs/语音交互.md](../product-specs/语音交互.md)

---

## 1. 设计目标

为GerClaw MVP实现纯前端语音交互能力，核心目标：
- **适老化优先**：点击式录音（非长按）、大按钮、清晰视觉反馈，降低老年用户使用门槛
- **流式低延迟**：ASR实时转写预览、TTS首包<1s、PCM16流式播放，追求接近实时的对话体验
- **纯前端实现**：基于浏览器原生MediaRecorder API录音、Web Audio API播放PCM16流，无额外音频依赖
- **API抽象封装**：Mimo ASR/TTS调用封装在统一voice client层，为二阶段后端代理预留适配接口
- **优雅降级**：浏览器不支持、权限拒绝、API失败时均有明确提示，不影响文本输入核心流程
- **配置零硬编码**：MIMO_API_KEY、模型名、音色、风格prompt全部通过环境变量配置

## 2. 架构设计

### 2.1 模块位置

语音交互模块属于前端UI层 + API服务层，在整体架构中的位置：

│  │ (录音按钮)    │  │ (波形可视化)     │  │ (播放/暂停)    │  │
│  └──────┬───────┘  └────────┬────────┘  └───────┬────────┘  │
│         │                   │                    │           │
│  ┌──────▼───────────────────▼────────────────────▼────────┐  │
│  │                    Voice Hooks Layer                    │  │
│  │     useVoiceRecorder | useTTSPlayer | useVoiceConfig   │  │
│  └──────────────────────────┬─────────────────────────────┘  │
└─────────────────────────────┼───────────────────────────────┘
                              │
┌─────────────────────────────▼───────────────────────────────┐
│                      API Client Layer                        │
│  ┌──────────────────────────────────────────────────────┐   │
│  │                   VoiceClient                         │   │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐   │   │
│  │  │ ASR Client  │  │ TTS Client  │  │ Audio Utils │   │   │

└─────────────────────────────┬───────────────────────────────┘

**依赖关系**：
- 被依赖：ChatInput组件（麦克风按钮）、MessageBubble组件（TTS播放按钮）、适老化老年模式（自动播放控制）
- 依赖：环境变量配置、全局Toast提示组件、ChatContext（发送消息）

### 2.3 数据流

**录音→ASR→发送流程：**

```
用户点击VoiceMicButton
    ↓
useVoiceRecorder.toggleRecording()
    ↓
检查权限 → navigator.mediaDevices.getUserMedia({audio: true})
    ↓ 权限拒绝
抛出错误 → Toast提示
    ↓ 权限允许
创建MediaRecorder → 启动录音 (mimeType: 'audio/webm' 或 'audio/wav')
    ├─ 创建AudioContext + AnalyserNode → 波形数据采集 → WaveformVisualizer实时绘制
    ├─ 启动计时器 → 更新录音时长显示
    └─ ondataavailable → 收集audioChunks
    ↓
用户再次点击 → 停止MediaRecorder
    ↓
收集到完整Blob → audioUtils.blobToWavBase64()
    ↓
voiceClient.transcribe(audioBase64)
    ├─ POST请求 mimo-v2.5-asr, stream: true
    ├─ fetch + ReadableStream读取SSE
    ├─ 逐chunk解析delta.content → 更新interimTranscript → TranscriptPreview实时显示
    └─ 流结束 → 获取finalTranscript
    ↓
finalTranscript非空 → 调用ChatContext.sendMessage(text) → 自动发送消息
    ↓
清理录音状态 → 重置MediaRecorder/AudioContext
```

**TTS流式合成→播放流程：**

```
用户点击TTSPlayButton
    ↓
useTTSPlayer.togglePlay(messageId, text)
    ├─ 正在播放同一条 → 暂停/继续切换
    └─ 新播放 → 停止当前播放（如有）
    ↓
创建AudioContext (sampleRate: 24000，匹配TTS输出)
    ↓
voiceClient.synthesize(text)
    ├─ POST请求 mimo-v2.5-tts, audio.format: 'pcm16', voice: '冰糖', stream: true
    ├─ fetch + ReadableStream读取SSE
    ├─ 逐chunk解析delta.audio.data → base64.b64decode → Int16Array转Float32Array (除以32768)
    └─ PCM数据入队缓冲
    ↓
启动流式播放调度器：
    ├─ 缓冲≥2个chunk → 创建AudioBufferSourceNode → 播放第一批
    ├─ onended事件 → 检查队列是否有新数据 → 继续播放下一批
    ├─ 跟踪播放进度 (currentTime) → 更新进度UI
    └─ 流结束+队列清空 → 播放完成
    ↓
暂停：调用source.stop()，记录pausedAt位置
继续：从pausedAt位置创建新SourceNode继续播放
    ↓
播放完成/停止 → 清理AudioContext和队列
```

## 3. 接口设计

### 3.1 对外接口

**useVoiceRecorder Hook：**

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `isRecording` | state | - | `boolean` | 是否正在录音 |
| `recordingState` | state | - | `RecordingState` | 录音状态：idle/requesting_permission/recording/recognizing/error |
| `duration` | state | - | `number` | 录音时长（秒） |
| `interimTranscript` | state | - | `string` | 实时转写预览文本 |
| `waveformData` | state | - | `number[]` | 实时波形音量数据（0-1，数组长度根据可视化需要） |
| `toggleRecording` | function | - | `Promise<void>` | 切换开始/停止录音 |
| `error` | state | - | `Error \| null` | 录音/识别错误信息 |

**useTTSPlayer Hook：**

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `playingMessageId` | state | - | `string \| null` | 当前正在播放的消息ID |
| `playbackState` | state | - | `TTSPlaybackState` | 播放状态：idle/synthesizing/playing/paused/error |
| `progress` | state | - | `number` | 播放进度（0-1） |
| `togglePlay` | function | `messageId: string, text: string` | `Promise<void>` | 切换某条消息的播放/暂停 |
| `stop` | function | - | `void` | 停止当前播放 |
| `error` | state | - | `Error \| null` | TTS错误信息 |

**VoiceClient API：**

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `transcribe` | async function | `audioBase64: string, onDelta?: (text: string) => void` | `Promise<string>` | ASR语音转文字，支持onDelta实时回调增量文本，返回完整转写结果 |
| `synthesize` | async function | `text: string, onPCMChunk?: (pcm: Float32Array) => void` | `Promise<void>` | TTS文字转语音，onPCMChunk回调每个PCM数据块，流结束后Promise resolve |

### 3.2 依赖接口

| 依赖 | 来源 | 用途 | 失败处理 |
|------|------|------|---------|
| `navigator.mediaDevices.getUserMedia` | 浏览器原生API | 获取麦克风音频流 | 权限拒绝时catch错误，Toast提示用户手动开启权限 |
| `MediaRecorder` | 浏览器原生API | 录音采集音频chunk | 浏览器不支持时禁用语音按钮，提示使用Chrome/Edge |
| `AudioContext` / `AnalyserNode` | Web Audio API | 波形分析、PCM16播放 | 浏览器不支持时禁用语音功能，保留文本输入 |
| `fetch` + `ReadableStream` | 浏览器原生API | SSE流式请求/响应 | 网络错误时重试2次，仍失败则Toast提示 |
| `process.env.NEXT_PUBLIC_MIMO_API_KEY` | Next.js环境变量 | Mimo API认证 | 未配置时禁用语音按钮，控制台输出配置缺失警告 |
| `ChatContext.sendMessage` | 全局对话Context | 发送ASR转写后的文本消息 | 消息发送失败由对话模块处理 |
| `Toast`组件 | 全局UI组件 | 显示错误/提示信息 | Toast组件不可用时降级为console.error + alert |

## 4. 数据设计

### 4.1 本地存储（MVP阶段）

MVP语音模块不持久化任何语音相关数据到localStorage：
- 原始音频Blob仅在内存中处理，ASR完成后释放
- PCM音频chunk播放完成后释放，不缓存
- 最终转写文本通过对话模块保存到消息列表（由通用对话模块负责localStorage持久化）

### 4.2 状态机

**录音状态机：**

```
idle ──点击──→ requesting_permission ──权限允许──→ recording
                     │                              │
                     │ 权限拒绝                      │ 再次点击
                     ▼                              ▼
                  error ←────────────────── recognizing
                     │                              │
                     │ 用户重试                      │ ASR完成
                     └────────→ idle ←──────────────┘
```

状态转换说明：
- `idle`：初始状态，显示麦克风图标
- `requesting_permission`：正在请求麦克风权限，按钮显示加载状态
- `recording`：录音中，显示停止图标+波形+时长
- `recognizing`：正在ASR识别，显示加载动画+转写预览
- `error`：发生错误，显示Toast，点击可重试回idle

**TTS播放状态机：**

```
idle ──点击播放──→ synthesizing ──首包到达──→ playing ──流结束+队列空──→ idle
                     │          │              │    ▲
                     │ API失败   │              │    │ 继续
                     ▼          └──暂停──→ paused ─┘
                  error
                     │
                     │ 用户重试
                     └────────→ idle
```

状态转换说明：
- `idle`：未播放，显示喇叭播放图标
- `synthesizing`：正在请求TTS API，显示加载旋转
- `playing`：正在播放，显示暂停图标+进度条
- `paused`：已暂停，显示播放图标，保留进度
- `error`：合成/播放失败，Toast提示，可重试

## 5. 错误处理

| 错误类型 | 处理方式 | 用户反馈 | 日志级别 |
|---------|---------|---------|---------|
| 麦克风权限拒绝 | catch NotAllowedError，停止录音流程，不发起ASR请求 | Toast："无法访问麦克风，请点击地址栏锁图标允许麦克风权限" | warn |
| MediaRecorder不支持 | 检测typeof MediaRecorder === 'undefined'，禁用语音按钮 | 按钮置灰，hover提示："当前浏览器不支持录音，请使用最新版Chrome/Edge" | warn |
| Web Audio API不支持 | 检测typeof AudioContext === 'undefined'，禁用TTS播放 | 播放按钮置灰，提示："当前浏览器不支持语音播放" | warn |
| 录音时长过短（<1s） | 停止后检查duration，不发起ASR | Toast："录音时间太短，请重新说话" | info |
| ASR网络错误/超时(30s) | AbortController超时控制，自动重试2次（指数退避），重试仍失败则降级 | Toast："语音识别失败，请重试或改用文字输入"，输入框保持焦点可手动输入 | error |
| ASR返回空文本 | 检查finalTranscript.trim()为空，不发送消息 | Toast："未能识别到语音内容，请靠近麦克风重新说话" | warn |
| TTS网络错误/超时(30s) | AbortController超时，重试2次，失败停止播放 | Toast："语音合成失败，请稍后重试"，按钮恢复idle | error |
| TTS首包超时(5s未收到audio chunk) | 取消请求，提示用户网络不佳 | Toast："网络较慢，语音加载超时" | warn |
| PCM解码错误 | 跳过异常chunk，继续播放后续chunk，记录错误 | 轻微爆音（用户感知弱），不打断整体播放，控制台日志 | warn |
| MIMO_API_KEY未配置 | 启动时检查环境变量，未配置则禁用所有语音功能 | 语音按钮置灰，hover提示："语音服务未配置，请联系管理员"，控制台明确输出缺失的环境变量名 | error |
| AudioContext自动播放策略限制 | 用户首次点击时resume() AudioContext（浏览器要求用户交互后才能播放音频） | 首次点击可能有极短暂延迟（<100ms），无额外提示（符合浏览器规范） | info |

## 6. 测试策略

- **单元测试覆盖**：
  - `audioUtils.ts`：WAV编码、PCM base64解码、格式转换函数
  - `voiceClient.ts`：SSE解析逻辑、错误重试逻辑
  - 工具函数纯逻辑测试（无浏览器API依赖部分）

- **集成测试覆盖**：
  - 组件渲染测试：VoiceMicButton各状态渲染正确，TTSPlayButton状态切换正确
  - Hook状态流转测试：useVoiceRecorder状态机正确转换，useTTSPlayer播放/暂停/停止逻辑正确
  - Mock fetch测试：模拟Mimo API SSE响应，验证流式解析和回调触发

- **手工测试场景**：
  - 主流程1（语音输入）：点击麦克风→授权→说话→看到波形和实时转写→点击停止→等待识别→文本自动发送→AI回复正常
  - 主流程2（TTS播放）：AI回复完成→hover看到播放按钮→点击→听到声音（冰糖音色）→按钮变暂停→看到进度→播放完成按钮恢复
  - 主流程3（暂停继续）：播放中点击暂停→声音停止→再次点击继续→从暂停位置继续播放
  - 失败路径1（权限拒绝）：首次点击麦克风→拒绝权限→看到明确提示→再次点击仍可请求权限
  - 失败路径2（断网测试）：录音过程中断网→停止录音→ASR失败提示→可手动输入
  - 失败路径3（空录音）：点击立即停止（<1s）→提示录音太短
  - 边界情况1（老年模式）：开启老年模式→AI回复完成→自动开始TTS播放
  - 边界情况2（浏览器兼容）：Safari/Chrome/Edge最新版均可正常录音和播放
  - 边界情况3（长文本TTS）：长消息（>500字）流式播放流畅无卡顿，首尾衔接自然
  - 边界情况4（并发控制）：播放一条消息时点击另一条的播放→前一条停止→新一条开始播放

## 7. 安全考虑

- **API Key安全**：MIMO_API_KEY通过NEXT_PUBLIC_前缀环境变量注入（MVP纯前端限制），不硬编码到源码，不提交到Git仓库，.env.example提供模板但不含真实Key
- **音频数据安全**：原始音频仅在浏览器内存中处理，不持久化到localStorage/IndexedDB，ASR/TTS完成后立即释放Blob和ArrayBuffer引用
- **HTTPS要求**：MediaRecorder和getUserMedia要求HTTPS环境（localhost开发除外），部署平台IGA Pages默认HTTPS
- **用户隐私**：不在前端日志中记录音频数据或转写文本内容，错误日志仅记录错误类型和消息长度，不记录具体内容
- **自动播放策略**：遵循浏览器自动播放策略，不自动播放TTS（除老年模式用户主动开启的自动播放开关），所有音频播放由用户点击触发
- **输入验证**：ASR返回的文本在发送到对话前做基本校验（非空、长度限制<5000字），防止异常数据进入对话流程
- **CORS配置**：Mimo API需配置允许GerClaw部署域名的跨域请求，开发环境localhost需在API侧白名单中

## 8. 可观测性

- **日志**：
  - info级别：录音开始/停止、ASR开始/完成（含耗时）、TTS开始/首包到达/完成（含首包延迟）、播放暂停/继续
  - warn级别：权限拒绝、录音过短、空识别结果、首包超时、单个PCM chunk解码失败
  - error级别：API调用失败（含HTTP状态码）、API Key未配置、浏览器API不支持
  - 日志格式：`[Voice][ASR] Completed in 1200ms, text length: 42 chars`

- **指标（前端Performance API）**：
  - `voice_asr_latency`：录音停止到最终转写结果的时间（ms）
  - `voice_tts_first_packet`：点击播放到首包音频到达的时间（ms）
  - `voice_tts_play_start`：点击播放到实际听到声音的时间（ms）
  - `voice_recording_duration`：单次录音时长（s）

- **追踪**：
  - 每个语音会话生成唯一uuid，贯穿录音→ASR→发送全链路
  - TTS播放用messageId关联，便于排查特定消息的播放问题

- **告警（MVP阶段无后端，仅控制台）**：
  - API Key未配置：启动时console.error告警
  - API连续失败3次：console.error提示检查网络和API Key
  - 浏览器不支持核心API：console.warn列出不支持的API和浏览器版本

## 9. 技术选型说明

| 技术/库 | 用途 | 选型理由 | 替代方案 |
|---------|------|---------|---------|
| MediaRecorder API | 浏览器录音 | 浏览器原生支持，无需额外依赖，能输出标准音频格式，Chrome/Edge/Safari 14.1+均支持 | 引入recorder.js库（额外依赖，原生已满足需求，不采用） |
| Web Audio API (AudioContext/AnalyserNode/AudioBufferSourceNode) | 波形可视化 + PCM16流式播放 | 浏览器原生，24kHz采样率可控，能实现低延迟流式逐chunk播放，精确控制暂停/继续位置 | 引入howler.js（专为HTML5 Audio设计，不支持原始PCM流播放，不采用） |
| fetch + ReadableStream | SSE流式请求/响应 | 浏览器原生fetch API支持流式读取响应体，无需EventSource（EventSource不支持POST请求和自定义Header，无法传递api-key） | 使用eventsource库（EventSource仅支持GET，无法满足Mimo API的POST+api-key header要求，不采用） |
| Canvas 2D / CSS动画 | 波形可视化 | Canvas绘制波形效果流畅（≥30fps），或用简单CSS条形动画降低实现复杂度，二选一根据视觉效果决定 | 使用Web Audio可视化库（如wavesurfer.js，引入额外~200KB依赖，MVP不需要那么复杂，不采用） |
| 原生AbortController | API超时控制 | 浏览器原生API，支持fetch取消，无需额外超时库 | 引入axios/ky等HTTP客户端（增加打包体积，原生fetch已够用，不采用） |
| React Hooks (useState/useRef/useEffect/useCallback) | 状态管理 | 语音模块状态局部化，不需要全局状态，用React内置hooks足够，避免引入额外状态管理库 | Zustand/Redux（语音状态不需要跨组件全局共享，仅通过Hook暴露接口，过度设计，不采用） |

## 10. 开放问题

- **实时流式ASR预览的实现方式**：MVP MediaRecorder录制完整音频后再一次性发送ASR是更简单的实现，但实时转写预览需要边录边传。是否MVP先实现录音完成后识别（无实时预览），实时预览作为优化项？还是严格按PRD实现边录边流式识别？（PRD F-002定义了实时转写预览为P0，需实现，但需确认MediaRecorder切片+增量ASR的技术可行性）
- **TTS进度条精度**：PCM16流式播放时总时长无法提前预知（SSE流不返回Content-Length），进度条只能基于已接收chunk估算还是显示简单"播放中"动画不做精确进度？
- **PCM16播放缓冲策略**：缓冲多少chunk后开始播放最优？缓冲太多增加首包延迟，缓冲太少可能网络抖动时卡顿（暂定2个chunk约200-400ms音频缓冲开始播放，可调）
- **二阶段迁移接口预留**：当前VoiceClient直接调用Mimo API，二阶段迁移到FastAPI后端代理时，是否需要提前抽象出IVoiceClient接口，MVP用Mimo实现，二阶段换Backend实现？（建议提前抽象接口，但保持简单）
- **老年模式自动播放的范围**：老年模式下是所有AI回复都自动播放，还是仅处方/评估报告等长内容自动播放？短回复（如"好的"）自动播放可能打扰用户
