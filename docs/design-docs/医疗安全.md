# 医疗安全 — 设计文档

> 模块：医疗安全 | 关联产品规格：[product-specs/医疗安全.md](../product-specs/医疗安全.md)

---

## 1. 设计目标

基于Next.js纯前端架构构建医疗安全防护层，实现MVP阶段简化版医疗安全机制，满足以下设计目标：

1. **零漏检免责声明**：通过"系统Prompt约束 + 后置兜底追加"双重保障，确保100%的AI医疗输出包含法定免责声明，免责声明文本不可被模型覆盖或省略
2. **高风险症状前置拦截**：用户消息提交前先进行高风险关键词正则检测，命中时在回复最顶部输出红色紧急警告卡片，提示立即拨打120，响应延迟<10ms
3. **确定性诊断禁止**：通过系统Prompt严格约束AI输出措辞，并在后置过滤器检测违规表述（"您患有XX病"、"确诊"等）时自动修正为建议性表述
4. **循证引用可追溯**：联网搜索结果注入上下文时添加隔离标记防注入，AI引用处自动插入[1][2]上标角标，点击展示来源详情
5. **未联网状态感知**：检测当前轮次是否调用搜索工具，对用药/诊断/治疗类医学问题自动追加"未联网检索"提示
6. **Prompt Injection纵深防御**：外部内容（搜索结果、上传文档）使用分隔标记包裹，系统Prompt明确禁止执行外部指令，正则检测典型注入模式并过滤
7. **首次使用强制确认**：模态对话框展示免责声明，用户必须点击确认才能使用，确认状态持久化到localStorage
8. **导出报告合规**：所有导出格式（PDF/Markdown/DOCX）末尾自动追加免责声明
9. **二阶段平滑演进**：代码结构预留权限分级、HITL人工审批、PHI数据脱敏的扩展接口，MVP阶段不实现但不阻碍后续接入AgentScope PermissionEngine/Middleware

约束：
- 严格遵循gerclaw设计要求.md第4.12节隐私安全模块、第9节系统安全与合规
- 参考AgentScope权限引擎（07_AgentHarness回路.md）和数据脱敏隐私保护（09_数据脱敏隐私保护.md）的设计思路，MVP阶段用纯前端简化实现
- 所有安全逻辑在前端执行，二阶段迁移后端时前端校验保留作为纵深防御

## 2. 架构设计

### 2.1 模块位置

在整体架构中的位置：**横切关注点（Cross-cutting Concern）安全层**，嵌入在对话流程的输入预处理、Prompt组装、输出后处理三个环节，与通用对话模块深度集成。

- **上游依赖（被调用）**：通用对话模块（ChatProvider/StreamingChatClient）、联网搜索模块、文档解析模块、导出模块（五大处方/CGA评估）
- **本模块内部**：高风险检测器、免责声明管理器、Prompt注入过滤器、引用处理器、首次确认管理器、导出安全处理器、系统Prompt安全段构建器
- **下游依赖（使用）**：无直接下游依赖，作为横切层为其他模块提供安全保障；二阶段依赖AgentScope PermissionEngine/PHIDesensitizationMiddleware

## 3. 接口设计

### 3.1 对外接口

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `SafetyProvider` | React Context Provider | `children: ReactNode` | - | 包裹应用根组件，提供安全上下文 |
| `useSafety()` | React Hook | 无 | `SafetyContextValue` | 获取安全状态和方法 |
| `withSafetyCheck(Component)` | HOC | React组件 | 增强组件 | 为组件包裹安全检查（二阶段预留） |
| `checkHighRisk(text: string)` | 函数 | 用户输入文本 | `HighRiskCheckResult` | 检测文本是否包含高风险症状关键词 |
| `buildSafetyPromptSegment()` | 函数 | 无 | `string` | 生成医疗安全约束的系统Prompt片段 |
| `wrapExternalContent(content: string, source: 'search'\|'document', id?: number)` | 函数 | 外部内容、来源类型、ID | `string` | 用隔离标记包裹外部内容，过滤注入 |
| `sanitizeOutput(text: string)` | 函数 | AI输出原文 | `SanitizeResult` | 输出净化：修正违规表述、兜底追加免责声明 |
| `parseCitations(text: string, citations: Citation[])` | 函数 | AI输出、引用元数据 | `ReactNode` | 解析引用标记，替换为角标组件 |
| `appendExportDisclaimer(content: string, format: 'pdf'\|'markdown'\|'docx')` | 函数 | 报告内容、格式 | `string` | 为导出报告追加免责声明 |
| `hasConsentedDisclaimer()` | 函数 | 无 | `boolean` | 检查用户是否已确认免责声明 |
| `recordConsent(version: string)` | 函数 | 免责声明版本 | `void` | 记录用户确认状态到localStorage |
| `shouldShowOfflineHint(text: string, searchUsed: boolean)` | 函数 | AI输出文本、是否用了搜索 | `boolean` | 判断是否需要显示未联网提示 |
| `SafetyProvider` | React Context | `{ children }` | Context Value | 全局安全状态提供者 |

**SafetyContextValue类型定义：**
```typescript
interface SafetyContextValue {
  hasConsented: boolean;
  showDisclaimerModal: boolean;
  consentDisclaimer: () => void;
  pendingWarning: string | null;
  setPendingWarning: (warning: string | null) => void;
}
```

### 3.2 依赖接口

| 依赖 | 来源 | 用途 | 失败处理 |
|------|------|------|---------|
| `useChatContext()` | 通用对话模块 (chat-context.tsx) | 获取当前消息列表、是否流式中、是否调用了搜索工具、搜索结果元数据 | 若context不可用，降级为非安全模式但仍强制追加免责声明 |
| `useRoleContext()` | 角色模式模块 | 获取当前是医生端/患者端/老年模式状态，调整警告和免责声明样式 | 默认使用患者端老年模式样式（最安全） |
| `searchResults` | 联网搜索模块 | 获取搜索结果列表，构建引用元数据 | 无搜索结果时不展示引用角标 |
| `exportTo*()` | 导出模块 | 五大处方/CGA评估/对话记录导出，在导出前调用appendExportDisclaimer | 若导出模块未调用安全处理器，ExportSafetyProcessor提供独立工具函数兜底 |
| `localStorage` | 浏览器原生 | 持久化免责声明确认状态 | localStorage不可用（隐私模式）时每次打开都显示模态框，不阻塞使用但每次提示 |
| `Vercel AI SDK onToolCall` | StreamingChatClient | 检测本轮对话是否调用了联网搜索工具 | onToolCall回调未触发时，降级为基于关键词判断是否是医学问题 |

## 4. 数据设计

### 4.1 数据库表/集合（MVP纯前端无数据库）

**localStorage存储：**

| Key | 数据结构 | 说明 |
|-----|---------|------|
| `gerclaw_disclaimer_consent` | `{ version: string, acceptedAt: number }` | 用户免责声明确认状态，version变更时需重新确认 |
| `gerclaw_safety_log` | `Array<SafetyLogEntry>` | 本地安全事件日志（仅内存+localStorage，不上传），最多保留100条 |

**SafetyLogEntry类型：**
```typescript
interface SafetyLogEntry {
  type: 'high_risk_detected' | 'injection_blocked' | 'diagnosis_phrase_corrected' | 'disclaimer_auto_appended';
  timestamp: number;
  details: Record<string, unknown>;
}
```

**内存中运行时数据：**

| 数据 | 存储位置 | 生命周期 | 说明 |
|------|---------|---------|------|
| `citationMetadata` | ChatProvider状态 | 单会话 | 当前轮次搜索结果的元数据（id/title/url/source），用于引用解析 |
| `searchUsedInThisTurn` | StreamingChatClient | 单轮对话 | 标记本轮是否调用了搜索工具，用于未联网提示判断 |
| `highRiskKeywords` | KeywordLibrary模块 | 全局单例 | 内置高风险症状关键词库，启动时加载 |

### 4.2 状态机（如适用）

**首次使用免责声明确认流程：**
```
页面加载
    ↓
检查localStorage: hasConsentedDisclaimer()?
    ├─ 否 → SHOW_MODAL → 用户点击"我已阅读并理解" → CONSENTED → 记录到localStorage → 正常使用
    └─ 是（版本匹配） → 正常使用
    ↓
版本不匹配（免责声明更新了） → SHOW_MODAL → 重新确认
```

**高风险警告流程：**
```
用户输入 → HighRiskDetector检查
    ├─ 命中 → setPendingWarning(warningTemplate)
    │       → LLM流式回复开始
    │       → 渲染时EmergencyWarningCard置于消息顶部
    │       → 后续正常回复内容
    │       → setPendingWarning(null)（下一轮重置）
    └─ 未命中 → 正常回复流程
```

**免责声明追加兜底流程：**
```
LLM流结束 → OutputSanitizer.sanitize(fullText)
    ↓
检查是否已包含免责声明关键词？
    ├─ 是（模型自己加了） → 检查格式是否合规 → 不合规则修正
    └─ 否 → 自动追加DISCLAIMER_TEXT到末尾
    ↓
追加MessageDisclaimer组件到消息底部（视觉层双重保障）
```

## 5. 错误处理

| 错误类型 | 处理方式 | 用户反馈 | 日志级别 |
|---------|---------|---------|---------|
| 高风险检测器正则匹配异常（如正则表达式错误） | 捕获异常，降级为"无风险"，但仍在回复末尾追加免责声明 | 用户看不到检测过程，但免责声明仍然存在 | error |
| localStorage写入失败（隐私模式/配额满） | 静默失败，hasConsented状态只在内存中保持true，刷新后重新显示模态框 | 用户仍可正常使用，只是每次刷新都需要点一次确认 | warn |
| Prompt注入过滤器误判（正常内容被误过滤） | 只做标记不做删除，用注释包裹不影响LLM理解 | 用户无感知，正常获得回复 | warn |
| 引用解析失败（角标不匹配） | 保留原始[1][2]文本不替换，不崩溃 | 用户看到原始数字标记，不影响阅读 | warn |
| 导出时免责声明追加失败 | 先检查内容是否已有，若追加失败则返回原内容，导出功能不阻塞 | 用户可导出成功，但需console提示免责声明可能缺失 | error |
| 二阶段适配器调用失败（PermissionAdapter未实现） | MVP阶段适配器为空函数，直接return allow | 无影响 | debug |
| 关键词库加载失败 | 使用硬编码的最小关键词集（胸痛/呼吸困难/大出血/自杀倾向/120）作为兜底 | 高风险检测仍可工作，只是覆盖范围略小 | error |

## 7. 安全考虑

- **认证/授权要求**：
  - MVP阶段访客模式无认证，医疗安全逻辑前端本地执行
  - 二阶段接入账号系统后，医生/患者权限分级由PermissionEngine管控
  - 所有安全检查在前端和后端（二阶段）双重执行，前端校验不可绕过

- **输入验证要点**：
  - 用户输入文本不做长度截断之外的过滤（避免误删用户症状描述）
  - 高风险检测仅做关键词匹配，不修改用户原始输入
  - 外部内容（搜索结果/文档）必须经过PromptInjectionFilter处理后才能注入上下文
  - 禁止直接将外部URL返回内容拼接到Prompt，必须经过extract和过滤

- **数据保护措施**：
  - MVP访客模式下所有对话数据仅存储在用户浏览器localStorage，不上传服务器
  - 高风险检测触发时，日志不记录用户完整输入文本，只记录匹配到的关键词
  - 二阶段接入PHIDesensitizationMiddleware，对身份证/手机号/姓名等PHI自动脱敏
  - 二阶段Workspace沙箱隔离，医疗数据目录设置严格的working_directories白名单

- **Prompt injection防护（纵深防御）**：
  - 第1层：外部内容使用HTML注释标记包裹，系统Prompt明确说明标记内是参考资料非指令
  - 第2层：正则检测典型注入模式（"忽略之前"、"你现在是"、"忘记以上"、"新指令"等）并过滤
  - 第3层：系统Prompt安全段明确指令优先级："系统安全指令 > 开发者指令 > 用户输入 > 外部参考资料"
  - 第4层：输出后置过滤，检测AI是否被诱导输出违规内容
  - 第5层（二阶段）：PermissionEngine DENY规则禁止危险工具被外部内容调用

- **医疗安全底线保障**：
  - 免责声明通过"Prompt要求 + 后置兜底追加 + UI组件渲染"三层保障，任何一层失效其他层仍可兜底
  - 高风险检测遵循"宁可误报不可漏报"原则，False Negative容忍度为0
  - 即使所有AI安全约束都失效，UI层的MessageDisclaimer组件固定渲染免责声明（与AI输出内容独立）

## 8. 可观测性

- **日志**：
  - `info`：免责声明正常追加、引用解析成功、用户确认免责声明
  - `warn`：确定性诊断表述被修正、引用标记不匹配、localStorage写入失败、注入被拦截
  - `error`：高风险检测器异常、免责声明追加失败、关键词库加载失败、导出安全处理器异常
  - 日志格式：`[Safety][{type}] {message} {details}`

- **指标（二阶段后端实现）**：
  - 高风险症状触发次数/日
  - Prompt注入检测拦截次数/日
  - 确定性诊断表述修正次数/日
  - 免责声明兜底追加率（衡量Prompt约束有效性）
  - 首次免责声明弹窗到确认的转化率
  - 未联网提示触发次数/日

- **追踪（二阶段）**：
  - 每个对话轮次的安全检查span：high_risk_check → injection_filter → output_sanitize
  - 二阶段接入TracingMiddleware，脱敏Middleware放在Tracing之前（先脱敏再追踪）

- **告警（二阶段）**：
  - 高风险症状短时间内触发次数异常（如同一用户多次触发，可能真的需要帮助）
  - Prompt注入攻击频次突增
  - 免责声明兜底追加率>5%（说明Prompt约束失效，需要优化系统Prompt）
  - 安全模块错误率>1%

## 10. 开放问题

- 高风险症状关键词库的具体覆盖范围需要与医学专业人士确认，MVP先使用常见急重症列表，后续持续扩充
- 未联网提示的触发条件需要细化：哪些医学问题算"专业类问题"必须提示，哪些常识类问题不需要（如"正常体温是多少"），MVP先采用关键词粗判
- 确定性诊断表述的过滤规则需要持续优化，避免误判正常表述（如"如果您患有XX病需要注意"是正确表述不应被修正）
- 二阶段权限分级中具体哪些操作属于低/中/高风险，需要与医生用户共同定义，MVP暂不实现但预留接口
- 二阶段PHI脱敏是否需要引入NER模型（如PaddleNLP医疗NER）提升姓名/地址识别准确率，MVP仅正则不做NER
- 免责声明的具体文本内容是否需要法务审核，当前使用PRD要求的标准文本："AI生成，仅供参考，不构成医疗诊断，如有不适请及时就医"
