# 用药审查 — 设计文档

> 模块：用药审查 | 关联产品规格：[product-specs/用药审查.md](../product-specs/用药审查.md)

---

## 1. 设计目标

本模块实现老年患者用药安全审查引擎，提供DDI药物相互作用检查、Beers Criteria老年人潜在不适当用药筛查、剂量合理性与重复用药检查三重安全保障。设计目标如下：

1. **纯前端规则引擎为核心**：MVP阶段所有审查逻辑在前端TypeScript实现，DDI规则、Beers标准、剂量参考数据以静态JSON/TS文件内置，无需后端API、无需网络请求即可完成核心审查，响应速度<200ms
2. **四级风险分级可视化**：严格按照禁忌(红)、严重(橙)、中等(黄)、轻微(蓝)四级分级，颜色+图标+文字三重编码，不单纯依赖颜色区分
3. **确定性规则优先，LLM仅辅助解释**：所有风险判断由确定性规则引擎给出（杜绝LLM幻觉导致医疗安全问题），LLM仅用于自然语言解释和报告润色（MVP可简化，纯规则输出即可）
4. **双入口无缝集成**：(1) 独立审查入口：医生端可手动输入药物列表专项审查；(2) 处方联动：五大处方药物生成时自动后台触发审查，结果嵌入处方预览
5. **老年患者特殊适配**：内置≥75岁风险自动升级逻辑、老年人起始剂量减半原则、多重用药预警（≥5种标记/≥10种警示）
6. **规则数据与逻辑分离**：DDI规则库、Beers标准条目、剂量参考表、商品名→通用名映射均为独立数据文件，便于医学团队后续更新维护，版本号可追溯
7. **医疗安全底线**：所有审查结果强制附带免责声明；禁忌级风险置顶醒目警示；明确标注"AI辅助生成，不替代专业医师/药师判断"

关键约束：
- MVP不依赖后端，不调用外部药物数据库API，使用内置精选规则库（覆盖老年科常用100+药物、50+高危DDI组合）
- 审查逻辑100%确定性代码实现，不依赖LLM做事实判断，LLM幻觉不影响核心风险检测
- 审查数据仅会话内有效，不持久化到localStorage（保护患者隐私）
- 商品名→通用名映射覆盖老年科常见药物（如"倍他乐克"→美托洛尔、"拜唐苹"→阿卡波糖等）
- 未识别药物不阻断审查流程，明确标注"未收录，结果仅供参考"

## 2. 架构设计

### 2.1 模块位置

在整体架构中的位置：
- **层级**：前端业务功能模块（MVP纯前端TypeScript），属于医疗安全核心引擎层
- **所属端**：医生端（主）+ 患者端（查看审查结果+语音朗读）
- **依赖模块**：
  - 右侧动态面板（复用面板容器、展开收起、宽度拖拽）
  - 通用对话模块（复用工具调用卡片展示审查过程、消息气泡展示结果摘要）
  - 五大处方模块（药物处方生成完成后调用审查引擎，结果嵌入处方）
  - 语音交互模块（TTS语音朗读审查结论）
  - 状态管理（Zustand存储审查状态、药物列表、结果数据）
  - 导出模块（Markdown导出复用现有逻辑）
- **被依赖**：
  - 五大处方模块（药物处方专项校验调用本模块）
  - 医生端工具面板（独立审查入口）
  - 欢迎页快捷入口
- **二阶段迁移**：审查核心逻辑（规则匹配、风险计算）可无缝移植到Python FastAPI后端，作为AgentScope PharmacistAgent的内置工具；前端仅保留UI和状态管理，通过REST API调用后端审查服务

**审查状态机流转**：

```
idle → inputting → reviewing → done
  ↑       ↑           │          │
  │       │           │          │
  └───────┴───────────┴──────────┘
        (清空/重新审查/修改药物列表)
```

各状态说明：
- `idle`：未启动审查，面板未展开或显示入口
- `inputting`：用户输入/编辑药物列表和年龄，可增删药物
- `reviewing`：审查执行中，显示步骤进度，禁止修改药物列表
- `done`：审查完成，显示结果，可重新审查或修改后再次审查

## 3. 接口设计

### 3.1 对外接口

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `startDrugReview` | 函数 | `options?: { initialDrugs?: string[]; patientAge?: number; autoTrigger?: boolean }` | `void` | 启动用药审查，展开右侧面板，进入inputting状态；autoTrigger=true时药物添加完自动审查（处方联动用） |
| `addDrug` | 函数 | `drugName: string` | `DrugInput` | 添加单个药物，自动执行名称归一化，返回归一化结果 |
| `removeDrug` | 函数 | `index: number` | `void` | 删除指定位置药物 |
| `setPatientAge` | 函数 | `age: number` | `void` | 设置患者年龄 |
| `clearDrugs` | 函数 | 无 | `void` | 清空药物列表 |
| `loadExampleDrugs` | 函数 | 无 | `void` | 加载示例老年高危药物组合（华法林+阿司匹林+氨氯地平等，用于演示） |
| `runReview` | 函数 | 无（使用当前药物列表和年龄） | `Promise<DrugReviewResult>` | 执行审查流水线，返回结构化结果，状态进入reviewing→done |
| `reviewPrescriptionDrugs` | 函数 | `drugs: MedicationItem[]; patientAge?: number` | `Promise<DrugReviewResult>` | 供五大处方模块调用：传入处方药物列表，后台执行审查，不展开独立面板 |
| `getReviewResult` | 函数 | 无 | `DrugReviewResult \| null` | 获取当前审查结果（未审查返回null） |
| `exportReviewMarkdown` | 函数 | 无 | `string` | 导出当前审查结果为Markdown文本，供下载/复制 |
| `copyReviewToClipboard` | 函数 | 无 | `Promise<boolean>` | 一键复制审查结果到剪贴板 |
| `resetReview` | 函数 | 无 | `void` | 重置审查状态，回到inputting，保留药物列表 |
| `closeReviewPanel` | 函数 | 无 | `void` | 收起右侧面板 |

### 3.2 核心引擎内部接口

ReviewEngine核心调用接口：

```typescript
// 审查主函数（纯函数，可移植到Python）
interface ReviewEngine {
  review(options: {
    drugList: string[];
    patientAge?: number;
  }): DrugReviewResult;
}

// 各检查器统一接口
interface Checker {
  check(drugs: DrugInput[], patientAge?: number): RiskItem[];
}

// 名称归一化
interface DrugNormalizer {
  normalize(drugName: string): DrugInput;
  normalizeBatch(drugNames: string[]): DrugInput[];
}
```

| 依赖 | 来源 | 用途 | 失败处理 |

### 4.3 审查流水线执行步骤

```
ReviewEngine.runReview() 执行顺序：

1. 归一化阶段（DrugNormalizer）：
   - 遍历drugList中每个药名
   - 先查BRAND_TO_GENERIC映射
   - 再查GENERIC_TO_CATEGORY映射
   - 未识别标记recognized: false
   - 输出：DrugInput[]

2. DDI检查阶段（DDIChecker）：
   - combinations(drugs, 2) 生成所有两两配对
   - 对每对(a, b)，按优先级匹配DDI_RULES：
     * (a, b)精确匹配
     * (b, a)顺序调换精确匹配
     * (a_category, b)类别-药名匹配
     * (a, b_category)药名-类别匹配
     * (a_category, b_category)类别-类别匹配
   - patientAge >= 75 且 severity == moderate → 升级为major，标记upgraded: true
   - 输出：RiskItem[]（type: 'ddi'）

3. Beers筛查阶段（BeersChecker）：
   - 遍历每个药物
   - 匹配drug_or_class（通用名或类别名）
   - 检查age_group约束（>=65 / >=75）
   - 输出：RiskItem[]（type: 'beers'）

4. 剂量检查阶段（DoseChecker）：
   - MVP简化：不解析具体剂量（仅药物名无剂量信息）
   - 输出通用老年剂量原则提示："老年患者用药建议从小剂量起始，注意监测肝肾功能"
   - P1阶段：从处方中解析剂量，与dose-reference.ts比对
   - 输出：RiskItem[]（type: 'dose'，多为minor级提示）

5. 重复用药检测（DuplicateChecker）：
   - 检查normalized_name重复（同一通用名不同商品名）
   - 检查category重复（同一类别≥2种，如两种CCB）
   - 输出：RiskItem[]（type: 'duplicate'）

6. 多重用药检测（PolypharmacyDetector）：
   - drugList.length >=5 → polypharmacy_warning level 'watch'
   - drugList.length >=10 → polypharmacy_warning level 'warning'
   - 输出：RiskItem[]（type: 'polypharmacy'，如有）+ polypharmacy_warning对象

7. 汇总排序阶段（RiskAggregator）：
   - 合并所有RiskItem[]
   - 按severity排序：contraindicated(0) > major(1) > moderate(2) > minor(3)
   - 统计各等级数量，生成risk_summary
   - 附加rule_versions、disclaimer、时间戳、id
   - 输出：DrugReviewResult（完整结构）
```

## 7. 安全考虑

### 认证/授权
- MVP访客模式：无认证，审查数据仅存浏览器内存，关闭即清除，不写入localStorage（隐私保护）
- 二阶段账号模式：审查操作记录审计日志（谁、什么时候、审查了哪些药物、发现什么风险）
- 患者端只能查看审查结果，不能发起审查（P1根据需求调整）

### 输入验证
- 药物名称输入：最大长度限制（50字符/药名），特殊字符过滤（防止XSS），最大药物数量限制（20种，防止性能问题）
- 年龄输入：数字类型校验，范围18-120，非法输入自动修正
- 从处方模块传入的药物列表：做类型校验和数据清洗，防止恶意构造数据导致引擎异常

### 数据保护
- **不持久化审查数据**：默认不写入localStorage/sessionStorage，内存状态关闭即清除（医疗隐私）
- 规则数据静态导入：不包含任何患者信息，不收集用户输入药物列表做统计（MVP阶段）
- 导出Markdown：仅包含审查结果和药物列表，不包含系统内部信息、API Key等
- HTTPS：部署平台强制HTTPS（IGA Pages默认支持）

### 医疗安全专项（最关键）
- **100%确定性规则引擎**：所有风险判断由TypeScript代码基于静态规则给出，**不依赖LLM做事实判断**，杜绝AI幻觉导致漏报/误报高危用药风险
- **禁忌级强制醒目**：contraindicated级别风险使用红色大字+⛔图标+置顶+"禁止联用"强警示语，不可折叠隐藏
- **免责声明强制添加**：MarkdownFormatter最后一步自动追加固定免责声明，不依赖任何外部输入
- **明确"仅供参考"定位**：所有结果开头/结尾多次强调"AI辅助生成，不替代专业医师/药师判断"，禁止使用"绝对安全"、"没有问题"等确定性表述
- **未收录药物明确提示**：未在内置库识别的药物标记灰色标签，结果概要说明"X种药物未收录，结果仅供参考"
- **规则版本可追溯**：结果中显示DDI/Beers/剂量表版本号，便于后续审计和规则更新
- **高风险不静默**：禁忌/严重级风险在处方联动时使用toast弹窗提示，不做静默处理
- **老年人特殊保护**：≥75岁自动风险升级、老年剂量原则提示、多重用药预警，体现老年专科特色

### XSS防护
- 所有用户输入的药物名在渲染时进行React转义（React默认XSS防护）
- Markdown渲染使用react-markdown，不使用dangerouslySetInnerHTML
- 导出Markdown时不注入HTML标签

- 独立审查 vs 处方联动占比
- 导出/复制使用率
- 禁忌级风险检出频次（用于评估规则库价值）

### 追踪
- 每次审查分配唯一review_id
- 各步骤添加时间戳：normalization、ddi_check、beers_check、dose_check、duplicate_check、polypharmacy、aggregation、formatting
- 记录每步开始/结束时间，便于性能优化

### 告警（二阶段生产环境）
- 审查引擎错误率>1%→告警（代码可能有bug）
- 平均审查耗时>500ms→告警（性能问题）
- 禁忌级检出率突增→关注（是否有批量异常请求或真实高危处方集中）

## 9. 技术选型说明

| 技术/库 | 用途 | 选型理由 | 替代方案 |
|---------|------|---------|---------|
| **TypeScript静态数据文件** | DDI/Beers/剂量/名称映射规则存储 | 类型安全、便于重构、无需异步加载、import即可使用、与审查引擎同语言无转换成本 | JSON文件+类型声明（缺少类型检查，重构不安全）；SQLite/WASM（太重，MVP不需要查询能力） |
| **纯函数审查引擎** | 核心检查逻辑 | 无副作用、易测试、输入→输出确定性、可无缝移植到Python后端、不依赖React/浏览器API可独立测试 | 面向对象类设计（增加复杂度，纯函数更适合数据转换流水线） |
| **Zustand** | 前端状态管理 | 项目已统一选用、轻量、简单、无需Provider包裹、适合本模块简单状态需求；**不启用persist**保护隐私 | React Context（prop drilling、重渲染问题）；Redux（太重，样板代码多） |
| **react-markdown** | 审查结果Markdown渲染 | 项目通用对话模块已使用、支持GFM、安全（不注入HTML）、可自定义组件渲染风险徽章 | 手写Markdown解析（重复造轮子）；marked（React集成差，默认不安全） |
| **浏览器Clipboard API** | 一键复制结果 | 现代浏览器原生支持、用户体验好 | document.execCommand('copy')（旧API，已废弃但兼容性好，可作为降级） |
| **组合式检查器设计** | DDI/Beers/剂量/重复各为独立Checker | 单一职责、易扩展（P1加肝肾检查、STOPP/START只需新增Checker）、易测试、规则数据不耦合 | 单个大函数做所有检查（不可维护、难测试、难扩展） |
| **itertools combinations** | 两两药物配对 | 成熟算法、无需自己实现组合逻辑；自行实现简单for循环也可 | 自行实现嵌套for循环（代码冗余，容易错） |

**说明**：
- MVP阶段审查引擎**不引入任何新的npm依赖**，仅使用项目已有依赖（React、Zustand、react-markdown等）
- P1阶段如需PDF/DOCX导出，复用项目已有的jsPDF/docx.js，不新增库
- 规则数据与逻辑严格分离：医学团队可直接编辑*.ts数据文件更新规则，无需理解审查逻辑代码
- 核心引擎代码不依赖React/DOM，纯TypeScript函数，便于编写单元测试和移植到Python

## 10. 开放问题

- [ ] Beers 2023标准MVP需要覆盖多少条目？当前设计覆盖20-30条核心PIM（苯二氮卓类、一代抗组胺、某些抗胆碱药等），完整Beers有115+条，是否需要更全？（当前建议：MVP覆盖老年科最常见20-30条，后续迭代扩充）
- [ ] MVP剂量检查做到什么程度？当前设计为通用老年剂量原则提示（不解析具体剂量），P1从处方解析具体剂量数值与参考表比对。是否需要MVP就实现剂量解析？（建议：MVP不做具体剂量数值比对，因为独立审查仅输入药名无剂量；处方联动P1再做）
- [ ] 商品名→通用名映射MVP需要覆盖多少？当前设计50+老年科常见，是否够用？
- [ ] 中成药/保健品交互MVP是否完全不做？国内老年患者常服用银杏/丹参/人参等，是否加入少量常见中成药交互（如银杏+华法林出血风险）？（建议：MVP加入5-10条最常见中西药交互，标注规则来源）
- [ ] 审查结果的LLM自然语言润色是否需要？当前设计纯规则输出结构化文本，可让LLM补充患者易懂的通俗解释，但引入幻觉风险。是否需要？（建议：MVP纯规则输出，不加LLM润色，确保医疗安全；P1可加LLM解释层但明确标注"AI解释仅供参考"）
- [ ] 审查结果是否需要支持语音朗读完整报告？还是仅朗读风险概要？（建议：朗读高风险部分+概要，详细内容用户自行阅读）
- [ ] 二阶段Python移植时，规则数据是保持JSON/TS格式还是转为Python字典？建议保持JSON为中间格式，TS/Python两端加载同一JSON数据源
