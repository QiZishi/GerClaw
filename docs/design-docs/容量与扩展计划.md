# 容量与扩展计划

> 状态：设计基线，未完成 1,000 并发压测  
> 更新：2026-07-18  
> 适用范围：MVP BFF、API、SSE、RAG、模型调用与临床工作流

## 已验证边界

当前有两项可复现的 Docker Compose ≤10 并发结果：确定性高风险短路 SSE（10/10 完成、失败率 0、p50 99ms、p95 100ms）和确定性、来源可追溯的用药审查 workflow（10/10 HTTP 200、失败率 0、p50 50ms、p95 53ms）。后者在并发计时前准备独立访客、session 和 intake，并验证规则结果、完成态 Trace 与跨访客 404。原始 PHI-free 报告见 [SSE](../evidence/perf-sse-safety-short-circuit-compose-2026-07-18.json) 与 [用药审查](../evidence/perf-medication-review-workflow-compose-2026-07-18.json)。两者都不包含外部模型、RAG、MinerU、语音或五大处方长流程，不能外推为这些路径的吞吐量，更不能表述为千级容量已经验证。

以下设计目标是让系统在未来能以受控方式扩展至 1,000 个活跃会话或 SSE 连接；是否允许扩大流量必须以分阶段压测和临床安全门禁为准。

## 运行拓扑与隔离

```text
Browser → Next BFF (stateless) → API ingress → API worker pools
                                        ├─ fast lane: 身份、健康检查、红旗短路
                                        ├─ stream lane: Chat/SSE、取消、Trace
                                        └─ job lane: RAG ingestion、MinerU、长临床草案
API worker pools → PostgreSQL / Redis / Qdrant / model & provider adapters
```

- BFF 和 API 都不得把主体身份、会话状态或 SSE 进度仅保存在进程内；认证、会话归属、Trace、checkpoint 和幂等事实分别由服务端 JWT、PostgreSQL 与 Redis 持久化。
- 每条 SSE 流由 Redis owner lease/fencing token 标识。重连可由任何健康实例接续；旧 owner 只能停止写入，不能重复发送终态或执行工具。
- RAG ingestion、MinerU 和五大处方长流程走独立 job lane，不能耗尽普通 Chat 的 worker、连接池或 SSE 队列。
- Redis、PostgreSQL 与 Qdrant 在生产环境分别采用可用区容错/备份方案；单机 Compose 只用于本地和验收，不是千级部署拓扑。

## 准入、背压与资源预算

所有数值由配置和运行指标控制，不能用前端隐藏按钮替代服务端限制。

| 层级 | 控制 | 超限行为 |
|---|---|---|
| 入口 | tenant、actor、IP 的令牌桶与并发 SSE 槽位 | 返回可重试的受控拒绝，不创建模型任务 |
| API | 每请求 deadline、步骤数、工具数、输入大小、32768 输出 token 上限 | 取消下游、持久化终态与 Trace 分类 |
| SSE | 有界事件队列、慢客户端检测、心跳、断连取消 | 停止生成；重连仅恢复同一主体的允许状态 |
| Provider | 每模型并发 semaphore、排队上限、熔断和候选共享上下文 | 选择合规备选或给出可理解的稍后重试状态 |
| 长任务 | Redis-backed job queue、租户公平队列、幂等键与 checkpoint | 排队位置可见；超时后可安全重试或恢复 |
| 数据层 | 连接池上限、查询 deadline、Qdrant 超时与缓存隔离 | 快速失败或低风险降级；不得跨主体复用缓存 |

容量计算以实测并发连接数为输入，而不是预设 QPS。任一 lane 的安全准入上限取以下最小值：

`min(worker 可用槽位, provider 可用槽位, DB/Redis/Qdrant 连接预算, tenant 配额, SSE 队列余量)`。

扩容时先增加无状态 BFF/API 实例和对应连接池预算，再验证 provider 配额、数据库连接、向量检索延迟与成本预算。禁止只扩 API 副本而让每个副本争抢同一有限模型或数据库连接池。

## 分阶段验证与放量门槛

1. **≤10（当前）**：在隔离 Compose 中分别覆盖红旗短路、普通 Chat、取消/断连、RAG、上传解析和长临床草案；每个场景最多 10 个并发活跃会话。记录完成率、p50/p95、SSE 终态、Trace 唯一性、重复写入、跨主体隔离和取消后的资源回收。
2. **预生产阶梯**：在与生产同类依赖及合成、去标识数据下逐级扩至 25、50、100、250、500、1,000 活跃连接；每阶稳定运行后才进入下一阶。模型与外部 Provider 必须使用明确成本上限和批准的测试配额。
3. **故障注入**：每阶验证一个 API 实例丢失、Redis 短暂不可用、数据库慢查询、Qdrant 超时、模型 429/5xx、慢 SSE 客户端和 job 重复投递。验收点是无跨主体泄露、无重复临床副作用、无孤儿任务与可观察的降级，而非单纯 HTTP 200。
4. **上线门槛**：每一 lane 的错误率、尾延迟、取消成功率、队列等待、连接池饱和、provider 429、token/成本和安全拒绝率须有基线与告警。任一安全或数据隔离门禁失败，停止放量并回滚到上一档配额。

## 观测、成本与恢复

- Trace 记录 workflow/model/tool/skill/policy/index 版本、预算消耗、排队和终态；不记录不必要的患者原文、图片 base64 或隐藏推理。受权限保护的原始输入仅用于既定 Trace/bad-case 流程。
- Dashboard 按 lane、tenant 和 Provider 观察活跃连接、排队、队列丢弃、重试、熔断、取消、p50/p95/p99、连接池、Qdrant 命中/超时和 token/成本。告警引用 Trace 分类，不回显 PHI。
- 模型、MinerU、ASR/TTS 出现容量或成本压力时优先保持红旗短路、身份与历史读取；长任务进入受控排队或明确失败，绝不跳过证据、授权、schema 或医疗安全检查来“保吞吐”。
- SSE 重连、job 重试和 provider failover 均使用既有 idempotency/checkpoint 事实；任何临床副作用都必须在恢复前重新通过 Runtime Permission/HITL。
