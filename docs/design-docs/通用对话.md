## 1. 设计目标

基于Next.js 15 + Vercel AI SDK构建纯前端静态导出的通用对话核心模块，满足以下设计目标：

1. **MVP纯前端架构**：所有API调用由浏览器直连，通过IGA Pages环境变量注入API Key，无后端服务依赖，静态导出可直接部署
2. **多协议流式统一**：通过Vercel AI SDK抽象层统一封装openai/dashscope/anthropic三种协议的SSE流式调用，支持thinking/tool_calls/content增量解析
3. **高可用模型切换**：实现主备模型5属性配置化（url/api_key/model_name/protocol/preference），超时/5xx/429自动降级切换，用户无感知
4. **Trae Work级可视化**：思维链默认折叠、工具调用独立卡片（运行/完成/失败三态+耗时+重试）、Markdown实时渲染、引用角标交互，交互体验对齐Trae Work
5. **访客会话持久化**：基于localStorage实现会话和消息的完整CRUD持久化，支持刷新恢复，存储上限保护
6. **多模态输入支持**：文本/图片/文件统一抽象为ContentBlock，图片转base64/objectURL直接多模态输入，文件通过MinerU云API解析为Markdown后注入上下文
7. **适老化兼容**：组件样式适配老年模式（大字体/大按钮/高对比度），不做硬编码尺寸，使用Tailwind CSS主题变量

约束：
- 严格遵循gerclaw设计要求.md第3.1-3.4节UI规范、第4.1节通用对话、第4.2节模型调用、第4.5节配置管理
- 二阶段迁移FastAPI后端时，API调用层需预留适配接口，前端业务组件无需改动

## 2. 架构设计

### 2.1 模块位置

在整体架构中的位置：**前端表现层核心模块**，位于Next.js App Router的主页面区域，是所有其他功能模块（五大处方、CGA评估、用药审查、联网搜索、技能管理）的基础交互载体。

- **上层依赖**：页面布局组件（三栏布局）、主题Provider、角色模式Provider（医生/患者/老年模式）
- **本模块内部**：会话管理、消息列表渲染、输入框、流式调用层、模型配置层、localStorage存储层
- **下层依赖（被使用）**：五大处方模块、CGA评估模块、用药审查模块、联网搜索模块、技能管理模块、文档解析模块、语音交互模块（TTS朗读/ASR输入）

## 3. 接口设计

### 3.1 对外接口

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `ChatProvider` | React Context Provider | `children: ReactNode` | - | 包裹应用，提供对话状态上下文 |
| `useChatContext()` | React Hook | 无 | `ChatContextValue` | 获取对话状态和操作方法 |
| `useChat()` | Vercel AI SDK封装Hook | `options?: UseChatOptions` | `{ messages, sendMessage, stop, isLoading, ...}` | 流式对话核心hook，本模块内部使用 |
| `sendMessage(content: string, attachments?: Attachment[])` | Context Action | 文本内容、附件列表 | `Promise<void>` | 发送消息，触发流式调用，自动持久化 |
| `stopGeneration()` | Context Action | 无 | `void` | 中断当前SSE流 |
| `createNewConversation()` | Context Action | 无 | `string` (conversationId) | 创建新会话，自动切换到新会话 |
| `switchConversation(id: string)` | Context Action | 会话ID | `void` | 切换到指定历史会话，加载消息 |
| `renameConversation(id: string, title: string)` | Context Action | 会话ID、新标题 | `void` | 重命名会话，持久化 |
| `deleteConversation(id: string)` | Context Action | 会话ID | `void` | 删除会话（二次确认后），从localStorage移除 |
| `regenerateMessage(messageId: string)` | Context Action | AI消息ID | `Promise<void>` | 删除该AI消息，基于上文重新生成 |
| `copyMessageContent(messageId: string)` | Util Function | 消息ID | `Promise<boolean>` | 复制消息纯文本到剪贴板 |
| `attachFile(file: File)` | Context Action | File对象 | `Promise<void>` | 上传文件：图片直接读为base64，文档调MinerU解析，添加到附件标签 |
| `removeAttachment(fileId: string)` | Context Action | 附件ID | `void` | 从附件标签移除文件 |
| `toggleCitation(citationIndex: number)` | UI事件 | 引用序号 | `void` | 切换引用详情显示/隐藏 |

| 依赖 | 来源 | 用途 | 失败处理 |

## 4. 数据设计

### 4.1 localStorage存储结构（"数据库表"）

MVP无后端数据库，所有数据存储在浏览器localStorage，单key存储版本化数据：

| Key | 结构 | 说明 |
|-----|------|------|
| `gerclaw_chat_v1` | `LocalStorageData`（见product-spec数据模型） | 所有会话和消息数据，单一key便于原子读写 |

**LocalStorageData字段详解**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `version` | `string` | 数据版本号（当前`"1"`），用于后续迁移 |
| `conversations` | `Record<string, Conversation>` | 会话ID→会话元数据映射 |
| `messages` | `Record<string, Message>` | 消息ID→消息对象映射（所有会话共享） |
| `conversationOrder` | `string[]` | 会话ID数组，按`updatedAt`倒序排列，用于侧边栏渲染 |

**存储策略**：
- 每次消息流结束、会话创建/重命名/删除时自动写入
- 使用防抖（300ms）合并频繁写入
- 写入前估算大小，超过4.5MB（保守阈值，localStorage通常5-10MB）时触发自动清理：
  1. 优先删除未固定的最早会话（按updatedAt升序）
  2. 直到总大小<4MB
  3. 清理后显示toast提示用户导出重要内容
- 写入失败（QuotaExceededError）时：
  1. 立即尝试清理最旧的非固定会话
  2. 仍失败则降级为内存存储（刷新丢失），显示红色警告提示

### 4.2 状态机

**消息流状态机**（单条AI消息生命周期）：

```
idle → 用户发送消息 → streaming
     ↓                    ├─ 收到thinking delta → thinking（显示思考块）
     ↑                    ├─ 收到tool_call delta → tool_running（工具卡片旋转）
     │                    ├─ 收到tool_result → tool_completed/tool_failed
     │                    ├─ 收到text delta → text_streaming（打字机效果）
     │                    └─ 流正常结束 → completed（收起思考块，显示操作按钮，保存到localStorage）
     │                    └─ 用户点击停止 → stopped（保留已生成内容，标记中断）
     │                    └─ API错误/网络错误 → error（显示错误提示，重试按钮）
     └─ regenerate/重试 → streaming
```

**工具调用卡片状态机**：

```
idle → 收到tool_call delta → running（旋转动画，显示工具名）
     ├─ 收到tool_result（成功）→ completed（绿色✓，显示耗时，可展开查看参数/结果）
     └─ 收到tool_result（失败/超时）→ failed（红色✗，显示错误信息，显示重试按钮）
     └─ 用户点击重试 → running
```

**文件解析状态机**：

```
idle → 用户选择文件 → uploading（标签显示上传中）
     ├─ 图片文件 → uploaded（显示缩略图，作为多模态输入）
     └─ 文档文件 → parsing（标签显示解析中）
         ├─ MinerU返回成功 → parsed（标签显示完成，Markdown注入上下文）
         └─ MinerU返回失败 → parse_failed（标签红色错误状态，显示重试按钮）
     └─ 用户点击×移除 → removed
```

## 5. 错误处理

| 错误类型 | 处理方式 | 用户反馈 | 日志级别 |
|---------|---------|---------|---------|
| 主模型API超时（60s） | 自动取消请求，切换到backup1重试，记录切换日志 | 无感知切换；如所有模型均超时，显示错误卡片 | warn（含模型名、超时时间） |
| 主模型返回5xx错误 | 同上，自动切换备模型 | 无感知切换；全部失败显示"服务暂时不可用，请稍后重试" | warn（含状态码、模型名） |
| 主模型返回429限频 | 自动切换备模型，不重试同一模型 | 无感知切换 | warn |
| 主模型返回401/403鉴权失败 | 跳过该模型（配置错误），切换备模型 | 如主模型鉴权失败直接切备模型；全部失败显示"API配置错误，请检查环境变量" | error（含模型名、状态码） |
| SSE流中断（网络断开） | 标记消息为不完整状态，停止loading | 消息末尾显示"网络中断，点击重试"按钮，保留已接收内容 | error |
| JSON解析错误（SSE chunk格式异常） | 跳过该chunk，继续解析后续chunk，不中断流 | 无感知（仅控制台记录）；如连续解析错误>10次则终止流显示错误 | warn（记录原始chunk片段） |
| 图片格式/大小不合法 | 前端验证阻止上传 | 明确提示："仅支持JPG/PNG/WebP格式，单张不超过10MB" | -（用户输入错误） |
| 文件MinerU解析失败 | 标记文件标签为失败状态，不将文件加入上下文，不阻塞消息发送 | 文件标签红色错误状态，hover显示"解析失败"，提供重试按钮 | error（含文件名、错误信息） |
| localStorage写入失败（QuotaExceeded） | 触发自动清理最早非固定会话；仍失败则降级为内存存储 | 清理后toast提示"存储空间不足，已清理最早对话"；降级后显示红色警告"本地存储不可用，刷新后数据将丢失" | error |
| localStorage读取失败（损坏/版本不兼容） | 尝试版本迁移；迁移失败则重置为空数据 | 显示"历史数据已重置"提示 | error |
| Clipboard API复制失败 | 降级为execCommand('copy')；仍失败则提示用户手动复制 | 显示"复制失败，请手动选择复制" | warn |
| 麦克风权限拒绝（语音输入） | 显示权限引导提示 | 弹出提示"请在浏览器设置中允许麦克风权限" | -（用户行为） |

## 7. 安全考虑

- **API Key保护**：
  - 所有API Key通过`NEXT_PUBLIC_*`环境变量注入（IGA Pages静态导出要求），不硬编码在源码中
  - `.env.example`仅留占位符，`.env.local`加入`.gitignore`
  - 注意：MVP纯前端直连API Key会在浏览器网络请求中可见，PRD已明确记录该风险，二阶段通过FastAPI后端代理解决

- **输入验证与XSS防护**：
  - Markdown渲染时禁用原生HTML（`rehype-raw`不启用），所有HTML标签转义
  - 代码块使用专门的高亮组件渲染，不执行其中的脚本
  - 用户消息输入做基本长度限制（单条≤10000字符），防止超长输入导致渲染卡顿

- **Prompt Injection基础防护**：
  - 系统prompt中明确指令优先级：系统指令>开发者prompt>用户输入
  - 图片/文件解析后的内容用明确分隔符包裹，标注为"用户上传内容"，防止注入混淆
  - MVP阶段不做复杂prompt注入检测（二阶段通过AgentScope权限引擎处理）

- **医疗安全合规**：
  - 所有AI输出区域底部固定显示医疗免责声明："本建议仅供参考，不能替代专业医生诊断"
  - 系统prompt强制约束：禁止给出确定性诊断结论（如"您患有XX病"），高风险症状（胸痛/呼吸困难/意识障碍等）必须立即提示就医
  - 医学知识回答必须标注来源依据，联网搜索结果通过角标引用

- **数据隐私**：
  - 访客模式不收集用户真实身份信息，所有数据仅存本地浏览器localStorage
  - 不向第三方（除模型API外）发送数据
  - 用户可手动删除所有会话数据（删除操作清空对应localStorage数据）
  - 模型API调用遵循对应服务商隐私政策

## 8. 可观测性

- **日志（console）**：
  - `info`：应用初始化、配置加载成功、会话创建/切换/删除
  - `warn`：模型切换（主→备）、SSE chunk解析错误（非致命）、localStorage清理、API非致命错误（429/5xx触发切换）
  - `error`：API鉴权失败（401/403）、所有模型均失败、localStorage读写致命错误、MinerU解析失败
  - 日志格式统一前缀：`[GerClaw][Chat]`，便于过滤

- **指标（前端内存统计，可选）**：
  - 当前会话消息数量
  - localStorage已使用/总容量估算
  - 首token响应时间（从发送到第一个text delta）
  - 模型切换次数/频率

- **追踪**：
  - 每条消息生成唯一`messageId`，贯穿发送→流式→完成→持久化全流程，日志中携带该ID便于追踪问题
  - 模型切换时记录：从哪个模型切换到哪个模型、触发原因（超时/状态码）、切换耗时

- **告警**：
  - MVP纯前端无服务端告警，通过用户可见的错误提示实现"用户侧告警"
  - 关键错误（配置错误、所有模型不可用）显示醒目红色错误卡片，引导用户检查配置或稍后重试

## 9. 技术选型说明

| 技术/库 | 用途 | 选型理由 | 替代方案 |
|---------|------|---------|---------|
| Vercel AI SDK (`ai`) | SSE流式处理、多模型统一调用接口、React hooks封装 | 官方支持OpenAI/Anthropic兼容协议，`useChat` hook内置流式delta解析、loading状态管理、错误处理，大幅减少SSE手写代码量；Next.js同团队出品兼容性最佳 | 手写fetch+ReadableStream SSE解析（代码量大，三种协议需分别适配）、LangChain JS（过重，MVP不需要Agent/RAG等复杂能力） |
| Next.js 15 App Router + 静态导出 | 前端框架，SSG输出适配IGA Pages | PRD指定，React 18+支持，静态导出(`output: 'export'`)满足IGA Pages纯静态托管要求 | 纯React+Vite（缺少App Router的路由组织和内置优化，后续SSR/SSG迁移成本高） |
| Tailwind CSS 4 | 样式方案 | PRD指定，原子化CSS，主题变量便于实现深浅双主题和老年模式大字体适配 | CSS Modules（类名管理成本高）、styled-components（运行时开销，SSR/SSG兼容问题） |
| ShadCN/UI | 基础UI组件 | PRD指定，高质量无样式组件，可深度定制Tailwind样式，符合医疗产品专业感 | Ant Design（样式定制困难，医疗场景风格不符）、MUI（包体积大） |
| react-markdown + remark-gfm + rehype-highlight | Markdown渲染 | react-markdown是React生态最成熟的Markdown渲染库，remark-gfm支持表格/任务列表等GFM扩展，rehype-highlight代码高亮 | marked（需要手动包装React）、markdown-it（插件生态但React集成不如react-markdown） |
| Zustand | 轻量状态管理（备选Context） | 如ChatContext性能不足时切换；API简洁，无需Provider嵌套，适合中等复杂度状态 | React Context（简单但频繁更新可能导致不必要重渲染）、Jotai（原子化但学习成本）、Redux（过重） |
| localStorage原生API | 访客会话持久化 | PRD指定，浏览器原生无需引入额外库，简单键值对存储足够MVP需求，5-10MB容量满足会话文本存储 | IndexedDB（复杂度高，MVP无结构化查询和大容量需求，二阶段迁移时再引入）、Dexie.js（IndexedDB封装，同上） |

## 10. 开放问题

- **MinerU云API具体规范**：endpoint URL、认证方式、请求参数、响应格式尚未最终确定，文件解析功能的具体实现需API文档提供后补充；当前先预留FileBlock数据结构和上传交互，API对接部分标记TODO
- **工具调用具体工具集范围**：通用对话模块定义了ToolCallBlock的可视化框架，但MVP阶段实际支持哪些工具（联网搜索是独立模块，用药审查/CGA/处方是独立模块），本模块仅负责工具调用的UI可视化，具体工具注册和执行逻辑由各功能模块实现，接口边界需后续对齐
- **localStorage数据迁移方案**：当前版本为v1，如后续数据结构变化（二阶段前），需要明确的版本迁移策略；已预留`version`字段，但具体迁移函数待数据结构变更时编写
- **代码高亮主题**：深浅双主题下代码高亮的配色方案需与整体设计系统对齐，后续UI设计阶段确定具体主题色
- **TTS朗读按钮具体集成点**：语音朗读功能依赖语音交互模块的TTS播放器，本模块仅在MessageActions中预留按钮位置和点击回调，具体TTS播放逻辑由语音模块实现，集成时机待语音模块完成后联调
