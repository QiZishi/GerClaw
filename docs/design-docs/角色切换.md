# 角色切换 — 设计文档

> 模块：角色切换 | 关联产品规格：[product-specs/角色切换.md](../product-specs/角色切换.md)

---

## 1. 设计目标

为GerClaw MVP实现医生端/患者端双角色切换系统，核心设计目标：
- **刷新式切换简单可靠**：MVP阶段角色切换采用页面刷新方案（`window.location.reload()`），避免复杂的组件重载/状态重置问题，保证状态一致性，二阶段账号模式再演进为无刷新切换
- **角色状态全局可访问**：通过React Context提供全局角色状态，所有组件可通过`useRole()` Hook获取当前角色并进行条件渲染
- **localStorage持久化兜底**：角色偏好持久化到localStorage，带类型校验和异常容错，写入失败时静默降级到内存状态
- **医生端标准模式强制锁定**：通过CSS类优先级（`.doctor-mode`特异性高于`.elder-mode`）确保医生端始终标准模式，老年模式开关在医生端完全隐藏不渲染
- **患者端适老化联动**：患者端默认触发适老化模块的老年模式，通过Context间联动实现，角色切换模块不重复实现适老化样式
- **医生端特有功能组件化**：患者列表、处方编辑审核、CGA工作区入口作为独立组件开发，根据角色条件渲染
- **访客模式零配置可用**：首次访问无登录直接进入患者端老年模式，所有功能无需注册即可使用
- **关键操作防误触**：切换角色前弹出确认对话框，有正在生成的内容时额外提示中断风险

### 2.3 数据流

**应用初始化流程**：

```
app/layout.tsx 加载
    ↓
RoleProvider 初始化
    ├─ 调用 roleStorage.loadCurrentRole()
    │   ├─ try读取localStorage 'gerclaw_current_role'
    │   ├─ 读取失败/无数据/非法值 → 返回默认值 'patient'
    │   └─ 读取成功且值为 'doctor' → 返回 'doctor'
    ├─ 设置 state.currentRole = loadedRole
    ├─ 根据 currentRole 设置 document.documentElement.classList:
    │   ├─ 'patient' → 添加'patient-mode'类，移除'doctor-mode'
    │   └─ 'doctor' → 添加'doctor-mode'类，移除'patient-mode'
    ├─ 通知 AccessibilityProvider（通过Context联动或事件）：
    │   ├─ role='patient' → 触发老年模式开启（如果elderModeEnabled为true）
    │   └─ role='doctor' → 强制关闭老年模式，隐藏ElderModeToggle
    └─ 渲染子组件
        ├─ 侧边栏渲染 RoleIndicator（显示当前角色）
        ├─ 侧边栏渲染 RoleSwitcher（显示切换目标按钮）
        ├─ isDoctor=true 时渲染患者列表、CGA入口等
        ├─ isDoctor=false 时渲染适老化组件
        └─ 所有组件通过 useRole() 响应角色变化
```

**角色切换完整流程**：

```
用户点击 RoleSwitcher 按钮
    ↓
调用 switchRole(targetRole) 函数（来自useRole Hook）
    ↓
检查是否有正在进行的流式生成（通过ChatContext）
    ├─ isGenerating=true → hasActiveGeneration=true
    └─ isGenerating=false → hasActiveGeneration=false
    ↓
调用 roleSwitchConfirm.confirm({ targetRole, hasActiveGeneration })
    ↓
弹出 ConfirmDialog：
    标题："切换到{targetRole === 'doctor' ? '医生端' : '患者端'}"
    正文：基础提示 + (hasActiveGeneration ? "\n\n当前有正在生成的内容，切换后将中断。" : "")
    按钮：[取消] [确认切换]
    ↓
用户点击"取消" → Dialog resolve(false) → switchRole return false → 流程终止
用户点击"确认切换" → Dialog resolve(true) → 继续
    ↓
如有正在进行的流式请求，调用 chatContext.cancelGeneration()
    ↓
调用 roleStorage.saveCurrentRole(targetRole)
    ├─ try写入localStorage 'gerclaw_current_role' = targetRole
    └─ 写入失败 → console.warn，继续流程（内存状态已更新）
    ↓
设置 state.currentRole = targetRole（触发重渲染，但马上刷新意义不大，保证刷新前状态一致）
    ↓
window.location.reload() 触发页面刷新
    ↓
页面重新加载 → 初始化流程读取新角色 → 加载对应UI
```

**医生端患者列表点击流程**：

```
医生端用户点击 PatientListItem
    ↓
调用 onSelectPatient(patientId)
    ↓
PatientList 更新选中状态（高亮当前患者）
    ↓
调用 chatContext.switchToPatientContext(patientId)
    ├─ 新建或加载该患者的对话会话
    ├─ 会话元数据关联patientId
    └─ 欢迎消息/上下文切换为该患者
    ↓
主聊天区显示该患者的对话界面
    ↓
可进行对话、开处方、CGA评估等操作
```

**处方审核操作流程**：

```
处方生成完成，右侧预览面板显示
    ↓
PrescriptionActions 组件（仅医生端）渲染：[编辑] [驳回] [审核通过]
    ↓
用户点击"编辑"
    ↓
处方预览区切换为编辑模式，可修改各处方便是内容
    ↓
用户点击"保存" → 更新处方内容，状态变为'draft'
用户点击"取消" → 返回预览模式
    ↓
用户点击"审核通过"
    ↓
弹出确认："确认审核通过此处方？审核后将标记为已确认。"
    ↓
确认后更新处方状态为'approved'，记录reviewedAt/reviewedBy（模拟值）
    ↓
显示绿色"已审核"徽章，操作按钮变为"查看已审核处方"
    ↓
用户点击"驳回"
    ↓
弹出对话框输入驳回原因（可选）
    ↓
确认后更新处方状态为'rejected'，记录rejectReason
    ↓
显示红色"已驳回"徽章，可重新编辑
```

## 3. 接口设计

### 3.1 对外接口

**useRole Hook**：

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `currentRole` | state | - | `UserRole` | 当前角色：`'patient'` \| `'doctor'` |
| `isDoctor` | derived | - | `boolean` | 是否医生端（便于条件渲染） |
| `isPatient` | derived | - | `boolean` | 是否患者端 |
| `switchRole` | function | `targetRole: UserRole` | `Promise<boolean>` | 执行角色切换：弹出确认框→确认→保存→刷新；返回用户是否确认切换 |
| `isSwitching` | state | - | `boolean` | 是否正在切换过程中（用于按钮loading状态） |

**RoleProvider Props**：
```typescript
interface RoleProviderProps {
  children: React.ReactNode;
  defaultRole?: UserRole; // 默认'patient'，可用于测试覆盖
}
```

**roleStorage 工具函数**：

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `loadCurrentRole` | function | - | `UserRole` | 从localStorage读取当前角色，带类型校验和默认值兜底 |
| `saveCurrentRole` | function | `role: UserRole` | `boolean` | 保存角色到localStorage，返回是否保存成功；失败时console.warn不抛错 |
| `clearRoleStorage` | function | - | `void` | 清除localStorage中的角色数据（用于重置/测试） |

**roleSwitchConfirm 工具**：

| 接口 | 类型 | 参数 | 返回值 | 说明 |
|------|------|------|--------|------|
| `confirm` | function | `options: RoleSwitchConfirmOptions` | `Promise<boolean>` | 弹出角色切换确认对话框，返回用户是否确认 |

**PatientList 组件 Props**：

### 3.2 依赖接口

| 依赖 | 来源 | 用途 | 失败处理 |
|------|------|------|---------|
| `localStorage` | 浏览器原生API | 持久化角色偏好 | try-catch包裹，失败时使用内存state，刷新后恢复默认，console.warn记录 |
| `window.location.reload` | 浏览器原生API | 角色切换后刷新页面 | 捕获异常（如沙箱限制），显示提示引导用户手动刷新 |
| `document.documentElement.classList` | DOM API | 添加/移除`patient-mode`/`doctor-mode`类名切换CSS | SSR环境下useEffect中执行（避免hydration不匹配） |
| `AccessibilityProvider` / `useAccessibility` | 适老化模块 | 患者端联动老年模式开关，医生端强制关闭老年模式 | 通过props或Context事件联动；如果适老化模块未就绪，医生端通过CSS`.doctor-mode`强制标准样式兜底 |
| `ConfirmDialog` | UI组件层/适老化模块 | 弹出二次确认对话框 | ConfirmDialog不可用时降级为window.confirm（不推荐，但作为兜底） |
| `ChatContext` / `useChat` | 对话模块 | 切换角色时取消正在进行的流式生成；患者列表点击时切换患者上下文 | 检查ChatContext是否存在，不存在则跳过取消生成步骤（不阻塞切换流程） |
| `Toast` | 全局UI组件 | 操作成功/失败提示 | Toast不可用时降级为console.log，关键错误用alert兜底 |
| `useTTSPlayer` | 语音交互模块 | 患者端老年模式下AI回复完成自动TTS播放 | 语音模块未就绪时自动播放静默失败，不阻塞消息流程 |
| `Next.js useRouter` | Next.js路由 | CGA工作区等子页面导航（如采用路由方案） | 纯前端面板方案可不用路由，用状态控制面板显示隐藏 |

**Provider嵌套顺序**（app/layout.tsx）：
```tsx
// AccessibilityProvider在外侧或内侧均可，建议RoleProvider在内层以便访问AccessibilityContext
<AccessibilityProvider>
  <RoleProvider>
    {children}
  </RoleProvider>
</AccessibilityProvider>
```

**Provider间联动逻辑**：
- RoleProvider初始化后，在useEffect中根据currentRole调用AccessibilityProvider的setElderModeEnabled：
  - role变为'patient'：恢复用户之前的elderModeEnabled偏好（默认true）
  - role变为'doctor'：强制setElderModeEnabled(false)
- RoleProvider不直接修改CSS变量，只负责切换角色标识类（patient-mode/doctor-mode），适老化CSS变量由AccessibilityProvider的elder-mode类管理
- `.doctor-mode`类在CSS中通过更高特异性强制标准样式变量，作为双重保险

## 4. 数据设计

### 4.1 本地存储

**localStorage键值设计**：

| 键名 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `gerclaw_current_role` | `'patient' \| 'doctor'` | `'patient'` | 当前用户角色 |

**存储读写规则**：
- 只能存字符串，写入时将UserRole枚举值直接存为字符串（'patient'/'doctor'）
- 读取时做严格类型校验：值必须是`'patient'`或`'doctor'`，否则视为非法值
- 非法值处理：回退到默认值`'patient'`，并覆盖写入正确默认值修复
- 所有读写操作try-catch包裹（隐私模式/配额满时localStorage可能抛异常）
- 与适老化模块的`gerclaw_elder_mode`键独立存储，但逻辑上联动：医生端时elderMode值被忽略

### 4.2 状态机

**角色状态流转**：

```
首次访问/无localStorage数据
    ↓
PATIENT_MODE（默认患者端）
    │
    │ 点击"切换到医生端" → 确认对话框
    │                         ↓ 确认（保存localStorage）
DOCTOR_MODE ←──── 页面刷新 ←──┘
    │
    │ 点击"切换到患者端" → 确认对话框
    │                         ↓ 确认（保存localStorage）
PATIENT_MODE ←──── 页面刷新 ←──┘
```

**角色切换确认对话框状态**：

```
closed ──switchRole()调用──→ opening ──动画──→ open
                                                  │
                                        ┌─────────┴─────────┐
                                        ↓                   ↓
                                    点击取消            点击确认
                                        │                   │
                                        └─────────┬─────────┘
                                                  ↓
                                             closing ──动画──→ closed
                                                  ↓
                                        Promise resolve(false/true)
                                                  ↓
                                    (true → 保存localStorage → reload)
```

**处方审核状态流转**：

```
处方生成完成 → pending_review（待审核）
    │
    ├─ 点击"编辑" → draft（编辑中）→ 保存 → pending_review
    │
    ├─ 点击"审核通过" → 确认 → approved（已审核）→ 终态
    │
    └─ 点击"驳回" → 输入原因 → rejected（已驳回）→ 可重新编辑→draft
```

**CSS类状态与DOM关系**：
- `document.documentElement`（`<html>`根元素）始终有且仅有一个角色类：`patient-mode` 或 `doctor-mode`
- 初始渲染（SSR hydration）时默认使用`patient-mode`类，避免hydration mismatch
- useEffect中根据实际角色状态校正类名

## 5. 错误处理

| 错误类型 | 处理方式 | 用户反馈 | 日志级别 |
|---------|---------|---------|---------|
| localStorage读写失败（隐私模式/配额满/存储被禁用） | try-catch捕获所有localStorage操作；失败时使用React state内存存储；角色切换功能本身仍可正常工作，但刷新后恢复默认患者端 | 无用户提示（静默降级）；用户刷新后发现角色重置可重新切换；console.warn记录失败原因 | warn |
| localStorage中角色值非法（被手动篡改为undefined/null/''/'invalid'等） | 读取时做严格字符串校验：值必须是`'patient'`或`'doctor'`；非法值立即回退到默认`'patient'`并覆盖写入正确值修复 | 无用户提示，自动恢复默认角色；console.warn记录检测到的非法值 | warn |
| `window.location.reload()`被浏览器/扩展阻止（极少数情况） | 用try-catch包裹reload调用；捕获异常后显示明确提示引导用户手动刷新 | Alert/大提示条："页面自动刷新被阻止，请按F5或Ctrl+R手动刷新页面以完成角色切换。"按钮："知道了" | error |
| SSR/hydration不匹配（Next.js服务端无window对象） | 初始渲染（服务端+首次客户端hydration）统一使用默认角色`patient`和`patient-mode`类；useEffect中再根据实际localStorage状态校正 | 用户无感知（hydration正确无warning）；如localStorage是doctor，useEffect校正后可能闪一下患者端再跳医生端（但因为紧接着reload或class切换很快，影响极小） | info |
| 用户在医生端通过DevTools手动添加`elder-mode`类或修改localStorage的elder_mode值 | 双重保险：1) RoleProvider检测到doctor角色时通知AccessibilityProvider强制关闭老年模式；2) CSS中`.doctor-mode`选择器特异性高于`.elder-mode`，强制覆盖老年模式CSS变量为标准值 | 无用户提示；医生端UI始终保持标准模式；console.info记录检测到异常class操作 | info |
| ConfirmDialog组件未挂载或调用失败 | 降级方案：先用console.error记录；再降级使用原生`window.confirm()`作为兜底（虽样式不好看但功能可用） | 原生confirm弹窗（浏览器默认样式）；提示文字与自定义Dialog一致 | error |
| 患者列表模拟数据异常（数据格式错误/空数组） | PatientList组件做props类型校验和空数组检查；空数组时显示"暂无患者"空状态；数据项字段缺失时兜底显示（如无名氏、年龄未知） | 空状态友好提示"暂无患者数据"；单个患者项显示异常时不影响整个列表渲染 | warn |
| 切换角色时ChatContext不存在（如对话模块未加载） | 检查ChatContext是否为null/undefined；不存在则跳过"取消生成中内容"步骤，直接继续切换流程 | 无用户提示；切换流程不阻塞 | info |
| 连续快速多次点击RoleSwitcher按钮 | switchRole函数加锁：isSwitching状态为true时直接返回，不重复弹出对话框或触发刷新 | 按钮点击时isSwitching=true，按钮显示loading状态，防止重复点击 | warn |
| 处方审核操作时状态更新失败 | 审核状态使用本地state管理，失败概率极低；操作失败时回滚状态并Toast提示 | Toast提示："操作失败，请重试"；状态回滚到操作前 | error |

## 7. 安全考虑

- **访客隐私保护**：MVP访客模式不收集任何PII（个人身份信息）；localStorage仅存储角色偏好（'patient'/'doctor'）和会话文本；不存储姓名/身份证/手机号等真实身份信息；模拟患者数据是虚构数据不对应真实人物
- **localStorage安全**：角色值仅为两个枚举字符串（'patient'/'doctor'），不包含任何敏感信息；XSS攻击无法通过本模块获取敏感数据；写入前不做HTML转义（因为是枚举值固定），读取时做严格白名单校验
- **医生端样式强制锁定**：通过CSS特异性（specificity）确保`.doctor-mode`的样式优先级高于`.elder-mode`和用户自定义样式；防止用户或浏览器扩展篡改class导致医生端意外显示老年模式大字体，影响专业使用
- **刷新切换的状态一致性**：刷新式切换虽然用户体验不如无刷新，但从安全和一致性角度更可靠：避免内存状态残留、避免组件未正确重置、避免权限边界模糊；刷新后从零初始化保证所有组件状态干净
- **确认对话框防误触**：切换角色是破坏性操作（中断当前流程、刷新页面），必须二次确认；有正在生成的内容时额外提示中断风险；确认按钮文字明确为"确认切换"而非模糊的"确定"；取消按钮与确认按钮同等醒目不做灰色弱化
- **条件渲染的权限边界**（虽无真实权限但逻辑边界清晰）：医生端特有组件（PatientList、PrescriptionActions、CGAWorkspace）使用`isDoctor`条件渲染，患者端根本不挂载这些组件，而非仅仅CSS隐藏；减少组件意外暴露的风险
- **模拟数据安全**：mockPatients中所有数据为虚构，不包含真实患者信息；不使用真实身份证号、手机号、具体医院名称等可识别信息
- **无外部请求**：角色切换操作是纯前端状态变化+页面刷新，不触发任何外部API请求；不会将用户角色偏好发送到任何服务器（除了页面刷新本身的静态资源请求）
- **DOM class白名单**：只操作`patient-mode`和`doctor-mode`两个预定义class，不接受外部输入拼接class名，避免XSS通过class注入

## 8. 可观测性

- **日志**（统一`[RoleSwitch]`前缀）：
  - info级别：
    - `[RoleSwitch] Initialized: role=patient/doctor, source=localStorage/default`
    - `[RoleSwitch] Switch requested: from=X, to=Y, hasActiveGeneration=true/false`
    - `[RoleSwitch] Switch confirmed, saving to localStorage and reloading`
    - `[RoleSwitch] Switch cancelled by user`
    - `[RoleSwitch] Patient selected: patientId=xxx, patientName=xxx`
    - `[RoleSwitch] Prescription action: prescriptionId=xxx, action=approve/reject/edit`
  - warn级别：
    - `[RoleSwitch] localStorage read/write failed: {error.message}, using in-memory fallback`
    - `[RoleSwitch] Invalid role value in localStorage: {badValue}, falling back to 'patient'`
    - `[RoleSwitch] reload() was blocked, showing manual refresh prompt`
    - `[RoleSwitch] ChatContext not available, skipping generation cancel`
    - `[RoleSwitch] ConfirmDialog not available, falling back to window.confirm`
  - error级别：
    - `[RoleSwitch] Critical error during switch: {error}`
    - `[RoleSwitch] DOM class operation failed (unexpected)`

- **前端指标**（console采集，二阶段接入后端监控）：
  - `role_initial_distribution`：初始化时角色分布（patient/doctor比例）
  - `role_switch_count`：角色切换总次数，按方向分（patient→doctor / doctor→patient）
  - `role_switch_confirm_rate`：切换确认率（确认数/请求数），判断确认提示是否过于打扰
  - `role_switch_cancel_reasons`（可选）：取消原因（无直接统计，主要通过是否有生成中分析）
  - `patient_list_click_count`：医生端患者列表点击次数
  - `prescription_action_counts`：处方审核操作分布（编辑/通过/驳回次数）
  - `role_switch_latency`：从点击切换按钮到页面开始unload的耗时

- **追踪**：
  - 每次会话初始化记录初始角色
  - 每次角色切换记录from/to/是否确认/是否有活跃生成
  - 医生端患者选择记录patientId

- **告警**（MVP阶段仅控制台，二阶段后端告警）：
  - localStorage完全不可用且内存模式运行时：console.warn提示"角色偏好将不会在刷新后保留"
  - 连续3次角色切换reload被阻止：console.error提示检查浏览器扩展
  - 检测到非法角色值频率过高（可能是存储损坏或攻击）：console.warn

## 9. 技术选型说明

| 技术/库 | 用途 | 选型理由 | 替代方案 |
|---------|------|---------|---------|
| React Context + useRole Hook | 全局角色状态管理 | 角色状态是全局低频变更状态（仅切换时变化），Context足够轻量；无需引入额外状态管理库；useRole Hook提供类型安全的访问方式，组件内一行const { isDoctor } = useRole()即可使用 | Zustand（对于单一角色状态过度设计，增加包体积）；Redux（更重量级，不适合）；Props逐层传递（嵌套过深，维护灾难） |
| window.location.reload() | 角色切换页面刷新 | MVP阶段刷新方案最简单最可靠：1) 所有组件状态从零初始化，无残留；2) 无需编写复杂的状态重置逻辑；3) 避免内存泄漏和未重置的副作用；4) 代码量极小（一行reload）；5) 二阶段可平滑演进到无刷新切换（去掉reload改为状态驱动） | 无刷新动态切换（需重置所有Context状态、清空对话state、重新挂载所有功能组件、处理useEffect清理函数，复杂度高且易出bug，MVP阶段ROI低，二阶段账号模式再实现）；路由切换（同域下用不同路由/doctor /patient，但刷新仍然存在且增加路由复杂度） |
| localStorage | 角色偏好持久化 | 浏览器原生、简单可靠、5MB容量绰绰有余存几个字符串；同步API读写简单无需异步处理；所有现代浏览器100%支持 | Cookie（每次HTTP请求都携带，浪费带宽且大小受限4KB）；IndexedDB（对于单个键值对过度设计，API复杂异步）；URL参数（刷新后丢失，分享链接会泄露角色偏好，不持久） |
| CSS类名切换（patient-mode/doctor-mode） | 角色样式隔离 | 通过给根元素加class切换样式，浏览器原生支持性能极佳；可与适老化模块的elder-mode类配合使用；CSS特异性控制优先级简单可靠；DevTools可直接调试 | CSS-in-JS动态样式（运行时开销大，不如class切换高效）；多套CSS文件按需加载（切换时需动态加载CSS文件，有FOUC风险且复杂度高）；React条件渲染整套组件树（代码重复量大，样式维护困难） |
| 命令式ConfirmDialog复用 | 切换确认对话框 | 复用适老化/UI层已有的ConfirmDialog组件，Promise式API调用简洁；避免重复造轮子；老年模式下自动使用大尺寸对话框 | 单独写一个RoleSwitchConfirm组件（代码重复）；原生window.confirm（样式丑、无法自定义大小和文案、老年模式下按钮小，作为兜底不做首选） |
| 前端硬编码mockPatients | 医生端患者列表数据 | MVP无需后端，前端硬编码8条模拟数据足够演示；不依赖外部API即可完整体验医生端流程；二阶段无缝替换为API调用 | 从JSON文件import（与硬编码本质相同，TS类型校验不如直接写TS对象方便）；空数据（无法演示多患者管理流程） |
| TypeScript枚举/字面量类型 | 角色类型安全 | `type UserRole = 'patient' \| 'doctor'` 确保类型安全；非法字符串在编译期就被拦截；运行时仍做校验兜底 | 字符串常量（无类型检查，易拼写错误）；数字枚举（调试时不直观，localStorage存数字不如存字符串可读） |
| 组件级条件渲染（isDoctor && ...） | 医生端特有功能显示 | 医生端组件在患者端完全不挂载（而非CSS隐藏）；减少不必要的组件渲染开销；从逻辑上彻底隔离两端功能边界 | CSS display:none（组件仍挂载，可能触发不必要的useEffect和API调用，有状态泄漏风险）；动态import（代码分割对于小组件收益不大，增加复杂度） |

## 10. 开放问题

- **患者上下文切换的会话管理**：点击医生端患者列表中的患者时，是新建一个关联该患者的会话，还是在当前会话中切换上下文？如果是新建会话，会话如何命名（以患者姓名命名？）；如果是切换上下文，历史消息如何处理？建议：点击患者新建一个以患者姓名命名的会话，会话元数据中存储patientId，方便后续筛选。
- **CGA工作区MVP范围**：CGA工作区在MVP阶段需要做到什么程度？是仅展示一个入口占位页面（"CGA工作区开发中"），还是展示已生成的评估报告列表+基本汇总？建议：MVP实现基础版本——展示所有已完成的CGA评估列表，点击可查看原报告，暂不做历史对比和数据可视化。
- **处方编辑MVP交互**：医生端处方编辑是在右侧面板内联编辑（contentEditable或textarea），还是弹出一个编辑模态框？编辑是Markdown源码编辑还是所见即所得？建议：MVP采用右侧面板内联Markdown编辑（简单textarea或基础Markdown编辑器），避免引入复杂富文本编辑器。
- **切换角色时会话保留策略**：PRD说"当前会话将保留"，但医生端和患者端是否共享同一会话列表？医生在医生端创建的会话在患者端能看到吗？建议：MVP会话列表全局共享，但医生端关联了patientId的会话在患者端可能不太适用，二阶段账号模式再做严格的数据隔离。
- **模拟患者数据是否需要头像**：mockPatients是否需要包含头像？是用默认首字母头像还是内置几个简单的头像图片？建议：MVP使用首字母背景色头像，不引入额外图片资源，简单可靠。
- **角色标识的位置和视觉风格**：RoleIndicator是放在侧边栏顶部Logo旁边好，还是放在底部切换按钮上方？颜色是否做区分（如患者端暖色、医生端蓝色）？建议：参考适老化文档设计，侧边栏顶部显示当前角色，颜色做轻量区分但不夸张。
- **二阶段无刷新切换的演进路径**：现在采用刷新式切换，二阶段要实现无刷新切换时，当前设计是否能平滑演进？答：当前RoleProvider的状态集中管理设计已预留演进空间——二阶段只需去掉switchRole中的reload()调用，改为重置各模块状态（对话重置、面板关闭、组件重新挂载）即可，Context接口不变。
- **自动TTS与角色模块的边界**：AI回复自动TTS是属于角色切换模块、适老化模块还是语音交互模块？建议：自动TTS开关状态和触发逻辑由适老化模块管理（因为是老年模式特性），角色模块负责在切换到患者端时通知适老化模块读取/启用autoTTS设置，切换到医生端时不自动播放。
