## 1. 模块映射总览

GerClaw 医疗智能体系统借鉴业界"星座模式（Constellation）"和"MDT 多学科会诊模式"，在 AgentScope 上落地为以下四层结构：

```
┌─────────────────────────────────────────────────────────────┐
│              交互层（适老化 UI / 语音 ASR-TTS）                │
├─────────────────────────────────────────────────────────────┤
│     主对话 Orchestrator Agent（共情引导 + 任务调度）           │
│     system_prompt: 老年友好沟通 + 风险识别 + 任务路由          │
├──────────┬──────────┬──────────┬──────────┬─────────────────┤
│老年科医生 │  药师     │ 护理师    │急症识别   │营养/随访         │
│Agent     │ Agent    │ Agent    │Agent     │Agent            │
│(ReAct+   │(用药审查  │(护理评估  │(120触发) │(慢病管理)        │
│ 鉴别诊断) │ DDI检测) │ 跌倒/认知)│          │                 │
├──────────┴──────────┴──────────┴──────────┴─────────────────┤
│          Toolkit：医学工具集（检验查询/DDI/指南检索）           │
├─────────────────────────────────────────────────────────────┤
│  知识层（RAG 老年医学指南 + 药品说明书 + 老年综合征 KG）        │
├─────────────────────────────────────────────────────────────┤
│  安全护栏层（免责声明 / 风险分级 / 人工接管触发器）              │
└─────────────────────────────────────────────────────────────┘
```

**AgentScope 能力到 GerClaw 角色的映射**：

| GerClaw 角色 | AgentScope 构造方式 | 核心能力 |
|-------------|-------------------|---------|
| 主对话 Orchestrator | 单 Agent + Toolkit（team 工具） | 用户交互、共情、路由分发 |
| 老年科医生 Agent | 单 Agent（ReAct 模式） + 医学工具 | 老年综合评估（CGA）、鉴别诊断、检查建议 |
| 药师 Agent | 单 Agent（ReAct 模式） + DDI 工具 | 多重用药审查、药物交互检测、用药指导 |
| 护理师 Agent | 单 Agent（ReAct 模式） + 评估工具 | 跌倒风险、营养筛查、认知评估、护理计划 |
| 多学科会诊（MDT） | Agent Team（TeamCreate/AgentCreate/TeamSay）或顺序编排 | 复杂病例的多角色会诊、意见汇总 |

## 2. 核心 API 参考

### 2.1 Agent 的 ReAct 模式配置

AgentScope Agent 内置 Reasoning-Acting（ReAct）循环，通过 `ReActConfig` 控制迭代行为。

**构造签名**：

```python
from agentscope.agent import Agent, ReActConfig, ContextConfig, ModelConfig
from agentscope.model import DashScopeChatModel
from agentscope.credential import DashScopeCredential
from agentscope.tool import Toolkit
from agentscope.message import UserMsg

model = DashScopeChatModel(
    credential=DashScopeCredential(api_key=os.environ["DASHSCOPE_API_KEY"]),
    model="qwen-plus",  # 或 qwen3.5-plus / qwen-max
    stream=False,
    context_size=131072,
)

agent = Agent(
    name="geriatrician",
    system_prompt="你是一名老年科医生...",
    model=model,
    toolkit=Toolkit(),                    # 工具集（可以为空）
    react_config=ReActConfig(
        max_iters=10,                    # 单次 reply 最大 ReAct 迭代次数
        stop_on_reject=False,            # 工具被拒绝时是否停止（医疗场景建议 False 以继续推理）
    ),
    context_config=ContextConfig(
        trigger_ratio=0.8,               # 上下文压缩触发阈值
        reserve_ratio=0.1,               # 压缩时保留比例
    ),
    model_config=ModelConfig(
        max_retries=2,                   # 模型调用失败重试次数
    ),
)

# 非流式调用
reply = await agent.reply(UserMsg(name="patient", content="医生，我最近头晕..."))
print(reply.get_text_content())
```

**ReActConfig 关键参数说明**：

| 参数 | 类型 | 默认值 | GerClaw 建议值 | 说明 |
|------|------|--------|---------------|------|
| `max_iters` | `int` | 20 | 8-15 | 医疗场景不宜过大，避免无限工具调用循环；建议设置合理上限 |
| `stop_on_reject` | `bool` | False | False | 若为 True，工具被用户/权限系统拒绝后停止 reply；医疗场景建议 False 让 Agent 继续输出安全建议 |

**ReAct 循环工作机制**：
1. Agent 接收用户消息 → 进入 reasoning 阶段（调用 LLM 推理）
2. LLM 输出文本回复或工具调用（ToolCallBlock）
3. 若有工具调用 → 执行工具 → 将结果作为 ToolResultBlock 注入上下文 → 再次 reasoning
4. 终止条件：LLM 不再输出 tool_call / 达到 max_iters / 被外部中断

### 2.2 单 Agent 系统提示词（system_prompt）设计

`system_prompt` 是 Agent 行为的核心控制手段，在 GerClaw 中应包含以下要素：

```python
DOCTOR_SYSTEM_PROMPT = """你是张医生，一名资深老年科副主任医师，在GerClaw老年医疗AI平台工作。

## 你的专业能力
- 老年综合评估（CGA）：躯体功能、认知、情绪、营养、社会支持多维评估
- 老年综合征管理：跌倒、痴呆、尿失禁、谵妄、晕厥、睡眠障碍
- 慢病共病管理：高血压、糖尿病、冠心病、脑卒中等老年常见慢病
- 合理用药：老年人多重用药（polypharmacy）风险识别

## 沟通原则（面向老年患者）
1. 使用温和、耐心的语气，语速宜慢
2. 避免专业术语，用通俗语言解释
3. 主动追问关键信息（症状持续时间、诱因、伴随症状、用药史）
4. 每次对话后给出明确的下一步建议
5. 对于急症征兆（胸痛、呼吸困难、意识障碍、大出血）立即建议拨打120

## 输出规范
- 所有建议末尾必须附加免责声明：
  "本建议由GerClaw老年医疗AI助手生成，仅供参考，不能替代医生的专业诊断和治疗建议。"
- 不做出确诊结论，使用"建议进一步检查"、"需排除"等表述
- 用药建议仅限OTC药品信息，处方药必须建议咨询医生
"""
```

**system_prompt 设计原则**：
1. **角色锚定**：明确身份（职称/科室/平台）
2. **能力边界**：列出专业能力，同时声明不做什么（不开处方、不确诊）
3. **沟通规范**：针对老年用户的语气/用词/追问策略
4. **安全红线**：急症识别规则 + 强制免责声明
5. **输出格式**：结构化输出要求（结论+依据+建议）

### 2.3 多 Agent Team 协作模式

AgentScope 提供服务级 Agent Team 机制（基于 Redis + MessageBus），也支持在库模式下手动编排多 Agent 协作。

#### 2.3.1 Agent Service Team（完整 Team API，需部署 Agent Service）

**Team 工具集**（位于 `src/agentscope/app/_tool/`）：

| 工具 | 文件 | 功能 |
|------|------|------|
| `TeamCreate` | `_team_create.py` | 创建团队，当前 session 成为 leader。参数：`name`（团队名）、`description`（团队章程，注入每个成员 prompt） |
| `AgentCreate` | `_agent_create.py` | 创建子 Agent（member），自动注册到团队。参数：`name`、`description`、`prompt`（初始任务）、`subagent_type`（可选，匹配 SubAgentTemplate） |
| `TeamSay` | `_team_say.py` | 团队内消息传递。参数：`content`（消息内容）、`to`（目标成员名，None 为广播） |
| `TeamDelete` | `_team_delete.py` | 解散团队，级联删除所有 member agents |

**Team 工作流程**：

```
1. Leader Agent 调用 TeamCreate(name, description) 创建团队
2. Leader 调用 AgentCreate 逐个添加成员（医生/药师/护理师），每个成员通过 prompt 接收初始任务
3. 成员自动开始工作，完成后通过 TeamSay 向 Leader 汇报
4. Leader 通过 TeamSay 向特定成员追问或广播协调
5. Leader 汇总意见后回复用户
6. 调用 TeamDelete 解散团队
```

**SubAgentTemplate（子 Agent 模板）**：

```python
from agentscope.app import SubAgentTemplate
from agentscope.agent import ReActConfig
from agentscope.permission import PermissionContext, PermissionMode

doctor_template = SubAgentTemplate(
    type="geriatrician",
    description="老年科专科医生，负责诊断评估和检查建议",
    system_prompt_template="""你是{member_name}，老年科医生，在团队'{team_name}'中工作，
团队负责人是{leader_name}。
团队目标：{team_description}
你的职责：{member_description}
完成任务后必须使用 TeamSay 工具向{leader_name}汇报结果。""",
    react_config=ReActConfig(max_iters=10),
    permission_context=PermissionContext(mode=PermissionMode.EXPLORE),  # 只读模式
)
```

#### 2.3.2 简化版多 Agent 协作（库模式，无需 Redis/Service）

对于不需要持久化和分布式部署的场景，可以在同一进程中顺序调用多个 Agent 实例，通过消息传递实现协作（见 `care_coordinator_team.py` 示例）。

```python
# 简化模式：顺序编排
async def mdt_consultation(patient_info: str):
    # 1. 医生初诊
    doctor_reply = await doctor.reply(UserMsg("患者", patient_info))
    # 2. 药师审查
    pharmacist_input = f"医生诊断：{doctor_reply.get_text_content()}\n请审查用药方案"
    pharmacist_reply = await pharmacist.reply(UserMsg("协调员", pharmacist_input))
    # 3. 护理师评估
    nurse_input = f"医生意见：{doctor_reply.get_text_content()}\n药师意见：{pharmacist_reply.get_text_content()}\n请制定护理计划"
    nurse_reply = await nurse.reply(UserMsg("协调员", nurse_input))
    # 4. 汇总
    return await coordinator.reply(UserMsg("系统", f"汇总三方意见..."))
```

### 2.4 各角色 Agent 差异化 system_prompt 设计

| 角色 | system_prompt 核心要点 | 专属工具 | ReActConfig 建议 |
|------|----------------------|---------|-----------------|
| 老年科医生 | CGA能力、鉴别诊断思维、老年综合征识别、检查建议规范 | `query_lab_result`、`search_guideline`、`assess_cga` | max_iters=10 |
| 药师 | 药理学知识、DDI检测流程、老年人剂量调整原则、用药教育 | `check_drug_interaction`、`query_drug_info`、`review_prescription` | max_iters=8 |
| 护理师 | 护理评估量表（Morse跌倒、MNA营养、MMSE认知）、护理计划制定、健康教育 | `fall_risk_assessment`、`nutrition_screen`、`care_plan_template` | max_iters=6 |
| 主对话Agent | 共情沟通、风险分诊、任务路由、免责声明追加 | Team工具 + 急症识别工具 | max_iters=15 |

### 2.5 Agent 间消息流转机制

**库模式（进程内）**：
- 通过 `agent.observe(msgs)` 向 Agent 注入观察消息（不触发推理）
- 通过 `await agent.reply(msg)` 触发一轮推理并获取回复
- 消息使用 `UserMsg`/`AssistantMsg`/`SystemMsg` 工厂函数构造
- 内容块使用 `TextBlock`/`HintBlock` 等

**服务模式（Agent Service Team）**：
- Leader 通过 `TeamSay` 工具发送 `HintBlock`（包装在 `<team-message from="...">` 标签中）到 member 的 inbox
- Member 收到 wakeup 信号后自动开始处理
- Member 完成后调用 `TeamSay(role="worker")` 将结果投回 Leader inbox
- `SubagentHitlProjector` 将 member 事件投影到 Leader session，实现前端统一展示

**消息内容块类型**：

```python
from agentscope.message import TextBlock, HintBlock, ToolCallBlock, ToolResultBlock

# 文本消息
text = TextBlock(text="患者主诉头晕3天...")

# 提示消息（用于 Agent 间指令传递，在 Team 中使用）
hint = HintBlock(hint="<team-message from=\"leader\">请审查用药方案</team-message>", source="leader")
```

## 6. GerClaw 适配要点

### 6.1 老年科医生/药师/护理师角色 Prompt 设计

**老年科医生 Agent system_prompt 关键要素**：
- 明确"老年科副主任医师"身份，锚定专业级别
- 强调老年综合评估（CGA）思维：不只看单一疾病，要评估功能状态、认知、营养、社会支持
- 老年综合征筛查清单：跌倒、痴呆、尿失禁、谵妄、晕厥、睡眠障碍、慢性疼痛
- 共病用药思维：老年人常服用5-7种药物，需考虑药物相互作用和不良反应
- 鉴别诊断路径：症状→常见老年病因→危险病因（必须排除）→建议检查
- 沟通适配：语言通俗、语速慢、关键信息重复确认
- 安全红线：识别急症（胸痛/呼吸困难/意识障碍/大出血）→立即建议120

**药师 Agent system_prompt 关键要素**：
- 明确"临床药师"身份，专注老年人用药安全
- 核心能力：多重用药审查（polypharmacy review）、药物-药物交互（DDI）、老年人剂量调整、Beers Criteria 老年人潜在不适当用药
- 审查流程：核对用药清单→检查适应症→检查剂量（肝肾功能调整）→检查 DDI→检查重复用药→提出优化建议
- 输出规范：每条建议给出依据（药品说明书/指南），处方药调整必须建议咨询医生

**护理师 Agent system_prompt 关键要素**：
- 明确"老年专科护理师"身份
- 核心能力：跌倒风险评估（Morse量表）、营养筛查（MNA-SF）、认知筛查（MMSE/MoCA）、ADL/IADL评估、压疮风险、护理计划制定
- 输出：护理诊断+护理目标+护理措施+健康教育要点
- 适老化：居家环境改造建议、辅助器具推荐、家属照护指导

### 6.2 推荐配置：多 Agent Team 的 ReActConfig

```python
from agentscope.agent import ReActConfig, ContextConfig
from agentscope.permission import PermissionContext, PermissionMode

# 主对话 Orchestrator（协调者）
ORCHESTRATOR_CONFIG = {
    "react_config": ReActConfig(max_iters=15, stop_on_reject=False),
    "context_config": ContextConfig(trigger_ratio=0.8, reserve_ratio=0.1),
    "permission": PermissionContext(mode=PermissionMode.ACCEPT_EDITS),
}

# 专科 Agent（医生/药师/护理师）
SPECIALIST_CONFIG = {
    "react_config": ReActConfig(max_iters=8, stop_on_reject=False),
    "context_config": ContextConfig(trigger_ratio=0.7, reserve_ratio=0.15),
    "permission": PermissionContext(mode=PermissionMode.EXPLORE),  # 只读模式
}

# 模型选择建议
MODEL_CONFIG = {
    "orchestrator": "qwen-max",         # 主对话使用最强模型
    "specialist": "qwen-plus",          # 专科 Agent 使用性价比模型
    "embedding": "text-embedding-v4",   # RAG 向量化
}
```

**Team 创建建议**：
1. 每次会诊创建独立 Team，会诊结束后 TeamDelete
2. 每个专科 Agent 使用独立 SubAgentTemplate，预定义 system_prompt_template
3. Leader（主对话Agent）负责：分诊→分发任务→收集意见→整合输出→免责声明
4. 设置 max_iters 防止 Agent 间消息循环（见风险6.3）
5. 所有专科 Agent 使用 EXPLORE 只读权限，防止误操作

### 6.3 风险：Agent 间消息循环、角色越权

**风险1：Agent 间消息循环（Message Loop）**
- 表现：医生Agent和药师Agent互相追问，无限循环不收敛
- 原因：ReAct max_iters 未生效；TeamSay 没有终止条件；Agent 反复调用 TeamSay 追问
- 防护：
  - 设置合理的 `ReActConfig.max_iters`（建议 Team leader 设为15，member 设为8）
  - 在 system_prompt 中明确"最多追问2轮"、"信息不足时基于现有信息给出最佳判断"
  - TeamSay 后等待回复，不轮询（Team 工具描述中已提醒 "DO NOT repeatedly call this to check on a member's progress"）
  - 设置总轮次上限（如 MDT 会诊不超过3轮讨论）

**风险2：角色越权（Role Overstepping）**
- 表现：药师给出诊断结论；护理师推荐处方药；医生给出药学专业建议但未请药师确认
- 原因：system_prompt 角色边界不清晰；Agent 自行扩展能力范围
- 防护：
  - system_prompt 中明确"你不得..."的禁止事项（如"你不得做出诊断结论"、"你不得推荐处方药"）
  - 使用 PermissionContext 限制工具访问（药师不授予诊断工具，护理师不授予处方工具）
  - Leader 最终汇总时校验各角色意见是否在其专业范围内
  - 高风险输出（用药/诊断）必须经过对应专科 Agent 确认

**风险3：医疗安全事故**
- 表现：AI 给出错误的用药/诊断建议导致患者受损
- 防护：
  - 所有输出强制附加免责声明
  - 急症关键词（胸痛/呼吸困难/意识障碍/大出血）→立即建议120并终止对话
  - 处方药建议必须标注"请咨询医生"，不得给出具体处方
  - 设置人工接管触发器：高风险场景自动转交真人医生
  - 符合《互联网诊疗管理办法》：AI 不得独立诊断和处方

**风险4：老年用户特殊性风险**
- 表现：老年人认知能力有限，被误导或无法理解复杂建议
- 防护：
  - 语言简化：避免专业术语，短句为主
  - 关键信息重复：重要建议重复至少1次
  - 支持语音交互（TTS/ASR）
  - 家属通知机制：重要健康建议同步给家属
  - 防欺诈：药品广告/保健品营销识别和拦截

## 7. 可运行示例指引

本模块配套 3 个可运行 Python 示例，位于 `agentscope-examples/02_medical_agents/` 目录：

| 示例文件 | 功能 | 运行前提 |
|---------|------|---------|
| `doctor_agent.py` | 老年科医生 Agent（ReAct模式），模拟初诊对话（张大爷主诉头晕） | `pip install agentscope && export DASHSCOPE_API_KEY=xxx` |
| `pharmacist_agent.py` | 药师 Agent，专注用药审查和用药指导，模拟审查张大爷的用药方案 | 同上 |
| `care_coordinator_team.py` | 多 Agent 协作（医生+药师+护理师），顺序编排模式模拟多学科会诊 | 同上 |

**运行方式**：

```bash
# 设置 API Key
export DASHSCOPE_API_KEY="your-dashscope-api-key"

# 运行单个示例
cd agentscope-examples/02_medical_agents
python doctor_agent.py
python pharmacist_agent.py
python care_coordinator_team.py
```

**示例覆盖的 AgentScope API**：
- `Agent` 类构造（name/system_prompt/model/toolkit/react_config）
- `DashScopeChatModel` + `DashScopeCredential` 模型初始化
- `Toolkit` + `FunctionTool` 自定义工具注册
- `ReActConfig`（max_iters/stop_on_reject）配置
- `UserMsg`/`AssistantMsg`/`SystemMsg` 消息构造
- `agent.reply()` 非流式调用
- `agent.observe()` 观察消息注入
- `reply.get_text_content()` 提取回复文本
- 多 Agent 顺序编排模式（库模式 Team 协作）
- 老年医疗模拟数据（张大爷病例、用药方案、评估数据）

4. 参考 `examples/agent_service/main.py` 学习 Agent Service + Redis Team 部署
