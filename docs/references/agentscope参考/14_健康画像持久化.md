# 14_健康画像持久化 — AgentScope 开发参考

> **模块定位**：GerClaw老年医疗AI平台中，基于 AgentScope 的状态持久化机制实现老年用户健康画像（基础信息/慢病/过敏/用药/评估标签）的跨会话持久化、状态变更监听与自动标签更新。
>
> **AgentScope 版本**：v2.0.3
>
> **关联调研文档**：`gerclaw前期调研/各部分调研/14_健康画像持久化.md`

---

## 1. 模块映射总览

健康画像持久化在 GerClaw 中承担三个核心职责：

| 职责 | AgentScope 对应机制 | GerClaw 落地 |
|------|---------------------|-------------|
| **会话状态持久化** | `RedisStorage`（`app/storage/_redis_storage.py`）— 基于 Redis 的异步 KV/List/Set 存储，承载 Agent/Session/Message 等运行时状态 | 老年用户（张大爷/李奶奶）对话历史跨进程/跨实例恢复 |
| **状态变更监听** | `StateChangeMiddleware`（`app/middleware/_state_change_middleware.py`）— 应用级 Middleware，通过 MD5 哈希对比 `tasks_context`/`permission_context` 变更并发布事件 | 自定义 `HealthProfileMiddleware`，在每轮 `on_reply` 后检测对话中新增的慢病/用药/过敏信息，自动写入健康画像 |
| **跨会话长期记忆** | `middle_context`（`state._state.AgentState.middle_context: dict[str, Any]`）— Middleware 专属的状态槽位，随 `AgentState` 序列化；或 `Mem0Middleware`（长期记忆后端） | 健康画像结构化数据（基础信息/慢病/过敏/用药/评估标签）存放在 `middle_context["health_profile"]` 中，随会话恢复自动反序列化 |
| **AgentState 序列化** | `AgentState`（Pydantic `BaseModel`）— `model_dump()`/`model_validate_json()` 原生支持 JSON 序列化；`RedisStorage.update_session_state()`/`get_session()` 负责存取 | 会话结束时 `AgentState` 序列化为 JSON 写入 Redis，新实例启动时反序列化恢复 context/middle_context |
| **会话恢复流程** | Agent Service 中 `ChatService` 在 chat 触发时通过 `storage.get_session()` 读取 `SessionRecord`，从中恢复 `AgentState`（含 `context` 消息历史和 `middle_context`） | 用户重新打开对话或后端重启后，从 Redis 读取 `AgentState`，健康画像和对话历史一并恢复 |

## 2. 核心 API 参考

### 2.1 StateChangeMiddleware：监听 Agent 状态变更

**文件**：`src/agentscope/app/middleware/_state_change_middleware.py`
**基类**：`agentscope.middleware.MiddlewareBase`

```python
class StateChangeMiddleware(MiddlewareBase):
    def __init__(self, message_bus: MessageBus, session_id: str) -> None: ...
```

**工作原理**：

1. 在 `on_reply` 开始时计算初始状态哈希（`tasks_context` + `permission_context` 的 MD5）。
2. 在 `on_acting`（每个工具调用后）重新计算哈希，与旧哈希对比；若变更则通过 `message_bus` 发布 `CustomEvent(name="state_updated")`。
3. 检测到团队工具（`TeamCreate`/`AgentCreate`/`TeamDelete`）执行后发布 `CustomEvent(name="team_updated")`。

**Hook 覆盖**：`on_reply`、`on_acting`。

**GerClaw 借鉴点**：自定义 Middleware 可在 `on_reply` 洋葱模型中，在 `await anext(next_handler)` 之后（即 reply 完成后）读取最新 `agent.state.context` 消息，从中提取健康信息并更新 `middle_context["health_profile"]`。

### 2.2 RedisStorage：会话持久化 API

**文件**：`src/agentscope/app/storage/_redis_storage.py`
**测试参考**：`tests/storage_redis_test.py`（使用 `fakeredis.aioredis.FakeRedis`）

**构造函数**：

```python
from agentscope.app.storage import RedisStorage

storage = RedisStorage(
    host="localhost",
    port=6379,
    db=0,
    password=None,
    connection_pool=None,   # 外部连接池
    key_ttl=None,          # 滑动 TTL（秒），None 永不过期
    key_config=None,       # RedisStorage.KeyConfig() 自定义键前缀
)
```

**核心持久化 API**（与健康画像相关）：

| 方法 | 说明 |
|------|------|
| `async upsert_session(user_id, agent_id, config, session_id=None, source=SessionSource.USER, source_schedule_id=None) -> SessionRecord` | 创建/更新会话；相同 `(user_id, agent_id, workspace_id)` 三元组复用同一会话 |
| `async get_session(user_id, agent_id, session_id) -> SessionRecord | None` | 获取会话记录，包含 `state: AgentState | None` |
| `async update_session_state(user_id, agent_id, session_id, state: AgentState) -> None` | 持久化 AgentState（序列化为 JSON 存入 Redis String） |
| `async delete_session(user_id, agent_id, session_id) -> bool` | 删除会话及关联消息 |
| `async upsert_message(user_id, session_id, msg: Msg) -> None` | 追加消息（RPUSH 到 List，相同 id 替换最后一条支持流式覆盖） |
| `async list_messages(user_id, session_id, offset=0, limit=50) -> list[Msg]` | 分页拉取历史消息 |
| `async list_sessions(user_id, agent_id) -> list[SessionRecord]` | 列出某 Agent 的所有会话 |

**Key 命名规则**（`KeyConfig` 默认模板）：

| 资源 | Key 模式 |
|------|---------|
| Session | `agentscope:user:{user_id}:session:{session_id}` |
| Session 索引 | `agentscope:user:{user_id}:agent:{agent_id}:sessions` (Set) |
| 消息列表 | `agentscope:user:{user_id}:session:{session_id}:messages` (List) |

### 2.3 AgentState 序列化 / 反序列化

**文件**：`src/agentscope/state/_state.py`
**公开 API**：`from agentscope.state import AgentState`

`AgentState` 继承 Pydantic `BaseModel`，原生支持 JSON 序列化：

```python
class AgentState(BaseModel):
    session_id: str = Field(default_factory=_generate_id)
    summary: str | list[TextBlock | DataBlock] = ""
    context: list[Msg] = Field(default_factory=list)       # 对话历史
    reply_id: str = Field(default_factory=_generate_id)
    cur_iter: int = 0
    permission_context: PermissionContext = Field(default_factory=PermissionContext)
    tool_context: ToolContext = Field(default_factory=ToolContext)
    tasks_context: TaskContext = Field(default_factory=TaskContext)
    middle_context: dict[str, Any] = Field(default_factory=dict)  # Middleware 状态槽
```

**序列化/恢复**：

```python
# 保存
state = agent.state  # AgentState
state_json = state.model_dump_json()
await storage.update_session_state(user_id, agent_id, session_id, state)

# 恢复（Agent Service 内部 ChatService 自动完成）
record = await storage.get_session(user_id, agent_id, session_id)
if record.state is not None:
    # record.state 已是 AgentState 对象（Pydantic 自动反序列化）
    agent.load_state(record.state)  # 或通过 Agent 构造恢复
```

> **重要**：`middle_context` 是 `dict[str, Any]`，Pydantic 会原样保留可 JSON 序列化的 Python 对象（dict/list/str/int/float/bool/None）。健康画像数据（纯 JSON 结构）可以直接放入。

### 2.4 Agent.state.middle_context：存储健康标签

`middle_context` 是各 Middleware 跨 reply 持久化私有状态的官方槽位。约定每个 Middleware 使用 `get_middleware_key()` 返回的字符串作为键（默认是类名），避免冲突。

```python
class HealthProfileMiddleware(MiddlewareBase):
    async def get_middleware_key(self) -> str:
        return "health_profile"

    async def on_reply(self, agent, input_kwargs, next_handler):
        # 读取现有健康画像（首次为 None）
        profile = agent.state.middle_context.get("health_profile")
        if profile is None:
            profile = self._empty_profile()
            agent.state.middle_context["health_profile"] = profile

        # 放行 reply
        async for event in next_handler(input_kwargs):
            yield event

        # reply 完成后：从最新对话提取健康信息，更新 profile
        self._extract_health_facts(agent.state.context, profile)
```

**健康画像推荐结构**（映射自调研文档 §1.2.2 JSONB 字段）：

```python
profile = {
    "basic_info": {           # 基础信息
        "name": "张大爷",
        "age": 72,
        "gender": "male",
        "height_cm": 170.0,
        "weight_kg": 68.5,
        "blood_type": "A+",
    },
    "conditions": [           # 慢病（ICD-10 编码）
        {"code": "I10", "display": "原发性高血压", "onset_date": "2018-03-15",
         "severity": "moderate", "status": "active"}
    ],
    "allergies": [            # 过敏
        {"substance": "青霉素", "type": "drug", "severity": "severe",
         "reaction": "皮疹"}
    ],
    "medications": [          # 用药
        {"name": "苯磺酸氨氯地平片", "dose": "5mg", "frequency": "每日1次",
         "start_date": "2018-04-01", "route": "oral"}
    ],
    "assessments": {          # CGA 评估标签
        "adl": {"score": 90, "level": "mildly_impaired"},
        "fall_risk": {"level": "moderate_risk"},
        "mmse": {"score": 24, "level": "mild_cognitive_impairment"},
    },
    "conversation_summary": { # 对话摘要
        "key_facts": ["独居", "子女每周探望"],
        "active_priorities": ["血压控制", "跌倒预防"],
        "last_interaction": "2026-06-30",

}
```

### 2.5 自定义 Middleware：在 on_reply 时更新健康画像

Middleware 洋葱模型的 `on_reply` 签名：

```python
async def on_reply(
    self,
    agent: "Agent",
    input_kwargs: dict,       # 包含 "inputs" 字段（用户消息）
    next_handler: Callable[..., AsyncGenerator],
) -> AsyncGenerator:
    # ---- before ----
    # 可在此处注入上下文、读取 middle_context

    async for event in next_handler(input_kwargs):
        # 可在此处拦截/转换 event
        yield event

    # ---- after ----
    # reply 完成，agent.state.context 已更新，可提取信息
```

**健康画像提取的典型逻辑**：

1. 在 `before` 段从 `middle_context` 读取画像（若无则初始化空画像）。
2. `yield event` 透传所有事件（保持流式输出不中断）。
3. 在 `after` 段遍历 `agent.state.context[-N:]`（最近 N 条消息），用关键词/规则匹配提取慢病、药物、过敏信息。
4. 合并到画像（去重：同一 ICD-10 编码不重复添加；同一药物名更新剂量）。
5. 更新 `profile_version` 和 `updated_at`。
6. （可选）将画像持久化到 Redis/PostgreSQL；在 Agent Service 模式下 `update_session_state` 会在 ChatService 中自动触发。

### 2.6 会话恢复流程

**Agent Service 模式（服务端）**：

```
1. 客户端 POST /chat {agent_id, session_id, input}
2. ChatService.trigger_chat()
   ├─ storage.get_session(user_id, agent_id, session_id)
   │    └─ 从 Redis 读取 SessionRecord，反序列化 AgentState（含 middle_context）
   ├─ 构造 Agent 实例，传入恢复的 state（或通过 agent.load_state()）
   ├─ 组装 middlewares（含 HealthProfileMiddleware）
   ├─ 执行 agent.reply_stream()
   └─ reply 结束后 storage.update_session_state() 保存最新 AgentState
3. 事件通过 SSE 流 /sessions/{session_id}/stream 推送给客户端
```

**库模式（非服务端，自管状态）**：

```python
# 第一次会话
agent = Agent(name="elder-assistant", model=model, middlewares=[profile_mw], ...)
await agent.reply(UserMsg(name="张大爷", content="我有高血压，吃氨氯地平"))
save_state = agent.state.model_dump_json()   # 序列化保存到 Redis/文件

# ---- 进程重启 / 新实例 ----
from agentscope.state import AgentState
recovered = AgentState.model_validate_json(save_state)
agent2 = Agent(name="elder-assistant", model=model, middlewares=[profile_mw], ...)
agent2.load_state(recovered)                 # 恢复对话历史和健康画像
await agent2.reply(UserMsg(name="张大爷", content="我最近脚踝有点肿"))
# agent2.state.middle_context["health_profile"] 已包含高血压+氨氯地平信息
```

### 4.2 Mem0Middleware 跨会话记忆（健康画像可借鉴的模式）

## 6. GerClaw 适配要点

### 6.1 健康画像数据结构

健康画像在 AgentScope 侧存放在 `AgentState.middle_context["health_profile"]`，为纯 JSON dict，结构对应调研文档 §1.2.2 的 JSONB 字段，但精简为对话态高频使用字段：

| 顶层键 | 类型 | 说明 | 写入时机 |
|--------|------|------|---------|
| `basic_info` | `dict` | 姓名、年龄、性别、身高、体重、血型、联系方式 | 用户主动告知/家属录入 |
| `conditions` | `list[dict]` | 慢病列表（ICD-10 `code`/`display`/`onset_date`/`severity`/`status`） | 对话中提及新诊断、病情变化 |
| `allergies` | `list[dict]` | 过敏列表（`substance`/`type`=drug/food/other/`severity`/`reaction`） | 对话中提及药物/食物过敏 |
| `medications` | `list[dict]` | 当前用药（`name`/`dose`/`frequency`/`route`/`start_date`/`adherence`） | 新处方、停药、剂量调整 |
| `assessments` | `dict` | CGA 评估标签（`adl`/`mmse`/`mna`/`fall_risk` 等量表分数与等级） | 量表评估完成后 |
| `conversation_summary` | `dict` | 对话摘要（`key_facts`/`active_priorities`/`pending_questions`/`recent_alerts`） | 每轮 reply 后增量更新 |
| `profile_version` | `int` | 乐观锁版本号，每次修改 +1 | 每次写入自增 |
| `updated_at` | `str` | ISO 时间戳 | 每次写入更新 |

**去重规则**：
- `conditions` 按 `code`（ICD-10）去重，相同 code 更新 `status`/`severity`。
- `allergies` 按 `substance` 去重。
- `medications` 按 `name`（通用名）去重，新药追加，旧药更新剂量/频次。
- `assessments` 按量表名覆盖（同一量表保留最新分数）。

### 6.2 推荐配置：Redis + StateChangeMiddleware 风格自动持久化

**最小可行配置（库模式，适合原型/单机）**：

```python
from agentscope.agent import Agent
from agentscope.model import DashScopeChatModel
from agentscope.credential import DashScopeCredential
from agentscope.tool import Toolkit
from agentscope.app.storage import RedisStorage
# 或使用 fakeredis/dict 模拟

profile_mw = HealthProfileMiddleware()  # 自定义
agent = Agent(
    name="gerclaw-elder-assistant",
    system_prompt=SYSTEM_PROMPT,
    model=chat_model,
    toolkit=Toolkit(),
    middlewares=[profile_mw],
)

# 会话循环中：每轮 reply 后手动持久化
await agent.reply(user_input)
state_json = agent.state.model_dump_json()
await redis.set(f"gerclaw:user:{user_id}:state", state_json)
```

**生产级配置（Agent Service 模式）**：

```python
from agentscope.app import create_app
from agentscope.app.storage import RedisStorage
from agentscope.app.message_bus import RedisMessageBus
from agentscope.app.workspace_manager import LocalWorkspaceManager

storage = RedisStorage(host="redis-host", port=6379, key_ttl=86400*30)  # 30天滑动TTL
message_bus = RedisMessageBus(host="redis-host", port=6379)

async def health_profile_mw_factory(user_id, agent_id, session_id):
    return [HealthProfileMiddleware(user_id=user_id)]

app = create_app(
    storage=storage,
    message_bus=message_bus,
    workspace_manager=LocalWorkspaceManager(basedir="/data/workspaces"),
    extra_agent_middlewares=health_profile_mw_factory,  # 每个会话注入
)
```

`ChatService` 会在每轮 reply 结束后自动调用 `storage.update_session_state()`，`middle_context["health_profile"]` 随 AgentState 一并持久化，无需手动 save。

**长期持久化建议**：
- Redis 作为**热缓存**（会话态，TTL 30 天），加速会话恢复。
- PostgreSQL（`health_profiles` 表，见调研文档 §1.2.1）作为**冷存储**，用户级别画像长期保存。
- 自定义 Middleware 在 `on_reply` after 段异步双写：先更新 `middle_context`（Redis 自动持久），再异步写入 PostgreSQL（通过业务 API）。
- 会话恢复时优先从 Redis 读，Redis 未命中则从 PostgreSQL 加载画像填充到 `middle_context`。

### 6.3 风险：画像版本冲突与数据一致性

| 风险 | 场景 | 缓解策略 |
|------|------|---------|
| **画像版本冲突** | 多设备/多端同时对话，`profile_version` 不一致 | 使用乐观锁：Middleware 更新前比对 `profile_version`，冲突时以 LWW（`updated_at` 最新者胜）合并，冲突字段进入 `pending_questions` 由用户/家属确认 |
| **Redis 数据丢失** | Redis 重启未持久化（AOF 关闭）或 TTL 过期 | 开启 Redis AOF everysec；PostgreSQL 作为权威存储；Middleware after 段异步写 PG 做兜底 |
| **对话中误提取** | 规则匹配误将普通对话识别为诊断/用药 | 健康标签提取分两级：(1) 规则匹配高置信度（明确的"我对XX过敏""我在吃XX药"）自动写入；(2) 模糊匹配放入 `pending_tags`，由 Agent 追问确认后升级为正式标签 |
| **敏感数据泄露** | 身份证号、手机号等 S3/S4 级数据进入 Redis/日志 | `middle_context` 中不存储绝密字段（身份证号、医保卡号）；S3 字段（姓名、手机号）在 Redis 中加密或仅存 hash；敏感字段走 PostgreSQL 加密列，Middleware 只持有脱敏副本 |
| **Middleware 状态膨胀** | `middle_context` 无限制增长导致 AgentState 过大 | 健康画像固定 schema（控制在 <10KB）；`context` 由 Agent 内置 `ContextConfig.compress` 自动压缩摘要；对话历史走消息 List（`upsert_message`），不堆在画像里 |
| **跨会话类型不一致** | 画像 schema 升级后，旧 AgentState 反序列化失败 | 使用 `schema_version` 字段（见调研文档）；Middleware 初始化时做 schema 迁移（补齐缺失字段、重命名字段） |
| **异步写入竞态** | after 段异步写 PG 时，用户快速发送下一条消息导致重复写 | 使用 asyncio.Lock（per user_id）或 Redis 分布式锁（`RedisStorage` 可扩展），保证同一用户画像串行更新 |
