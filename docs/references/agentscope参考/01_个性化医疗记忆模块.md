# 01 个性化医疗记忆模块 — AgentScope 开发参考索引

> **模块定位**: GerClaw老年医疗AI平台的长期记忆层，负责跨会话持久化患者过敏史/慢病/用药/就诊记录，支持上下文压缩以适应长对话场景。
> **AgentScope版本**: v2.0.3
> **核心组件**: `Mem0Middleware`（长期记忆中间件）+ `ContextConfig`（上下文压缩配置）+ `Agent.state.summary`（压缩摘要）
> **最后更新**: 2026-07-02

---

## 1. 模块映射总览

GerClaw"个性化医疗记忆"模块在AgentScope框架中的实现由两条技术路径协同完成：

| GerClaw需求 | AgentScope对应能力 | 关键类/配置 | 作用层级 |
|-------------|-------------------|-------------|----------|
| 跨会话记忆（过敏史/慢病/用药持久化） | 长期记忆中间件 | `Mem0Middleware` | Middleware层（on_reply钩子） |
| 记忆自动提取与语义检索 | mem0 OSS/Platform后端 | `mem0.AsyncMemory` / `AsyncMemoryClient` | 外部存储层 |
| 长对话上下文压缩（多轮就诊记录压缩） | 内置上下文压缩 | `ContextConfig` + `Agent.compress_context()` | Agent核心引擎 |
| 压缩摘要保留医疗关键信息 | 自定义summary模板 | `ContextConfig.summary_template` / `summary_schema` | Agent配置 |
| 医疗记忆按用户隔离 | mem0命名空间 + metadata过滤 | `Mem0Middleware(user_id=..., agent_id=...)` | 中间件参数 |
| 压缩后原始内容可追溯 | Offloader协议 | `LocalWorkspace`（实现Offloader） | Workspace层 |
| Agent状态中保留压缩摘要 | AgentState | `Agent.state.summary: str` | State层 |

**记忆分层架构对应关系**：

```
GerClaw记忆分层          AgentScope实现
─────────────────────────────────────────────────────
L0 Core Memory（核心档案） → system_prompt中硬编码患者基本信息/过敏红警
L1 Session Memory（会话）  → Agent.state.context（当前对话上下文）
L2 Episodic Memory（情景） → Mem0Middleware + Qdrant向量库（语义检索）
L3 Structured Memory（结构化）→ 外部PostgreSQL（独立于AgentScope）
L4 Knowledge Memory（知识）→ RAGMiddleware + KnowledgeBase（医学指南）
上下文压缩层              → ContextConfig自动压缩 + state.summary
```

## 2. 核心API参考

### 2.1 Mem0Middleware（长期记忆中间件）

**导入路径**: `from agentscope.middleware import Mem0Middleware`

**类签名**:
```python
class Mem0Middleware(MiddlewareBase):
    def __init__(
        self,
        user_id: str,                                    # 必填：mem0命名空间
        client: AsyncMemory | AsyncMemoryClient | None = None,  # 预构建客户端（优先级最高）
        chat_model: ChatModelBase | None = None,         # 聊天模型（记忆提取用）
        embedding_model: EmbeddingModelBase | None = None,  # Embedding模型（维度须匹配vector store）
        mem0_config: MemoryConfig | None = None,         # 自定义mem0配置
        mode: Literal["static_control", "agent_control", "both"] = "both",
        agent_id: str | None = None,                     # 更细粒度命名空间
        top_k: int = 5,                                  # 检索条数
        threshold: float | None = None,                  # 相似度阈值
        scope_search_by_agent: bool = True,              # True=按user+agent过滤
        await_write: bool = True,                        # True=同步等待写入
        memory_section_header: str = "## Relevant memories from past conversations",
        memory_section_intro: str = ...,                  # static注入的介绍文本
        tool_instructions: str = ...,                     # agent模式追加到system prompt的说明
    )
```

**三种控制模式（mode参数）**:

| mode | 自动检索 | 自动写回 | 暴露工具 | 适用场景 |
|------|---------|---------|---------|---------|
| `"static_control"` | reply前自动检索并注入HintBlock | reply后自动写回 | 否 | Agent无感知，最简单集成 |
| `"agent_control"` | 否 | 否 | search_memory + add_memory | Agent自主决定读写时机 |
| `"both"`（默认） | 是 | 是 | 是 | 兼顾自动与手动 |

**三种构造路径**:

```python
# 路径1：传AgentScope模型（最简单，中间件内部构建OSS AsyncMemory）
mw = Mem0Middleware(
    user_id="patient_zhang_001",
    chat_model=chat_model,
    embedding_model=embedding_model,
)

# 路径2：模型 + 自定义mem0_config（自定义Qdrant路径/维度等）
from mem0.configs.base import MemoryConfig
from mem0.vector_stores.configs import VectorStoreConfig
mw = Mem0Middleware(
    user_id="patient_zhang_001",
    chat_model=chat_model,
    embedding_model=embedding_model,
    mem0_config=MemoryConfig(
        vector_store=VectorStoreConfig(
            provider="qdrant",
            config={"collection_name": "gerclaw_mem0", "path": "/data/qdrant",
                    "embedding_model_dims": 1536, "on_disk": True},
        ),
    ),
)

# 路径3：预构建client（托管Platform或多agent共享）
from mem0 import AsyncMemoryClient
mw = Mem0Middleware(
    user_id="patient_zhang_001",
    client=AsyncMemoryClient(api_key=os.environ["MEM0_API_KEY"]),
)
```

**供Agent调用的工具（agent_control/both模式）**:

- `search_memory(keywords: list[str], limit: int = 5)` — 用简短精准关键词检索记忆，每个keyword独立并行搜索，结果去重合并
- `add_memory(thinking: str, content: list[str])` — 记录持久信息；`thinking`仅审计不写入，`content`为独立完整句子列表，写入mem0

**工具注册方式**（必须手动传入Toolkit）:
```python
agent = Agent(
    ...,
    toolkit=Toolkit(tools=await mw.list_tools()),
    middlewares=[mw],
)
```

### 2.2 LongTermMemoryMiddleware（通用长期记忆中间件基类）

`Mem0Middleware`是`LongTermMemoryMiddleware`的具体实现。`LongTermMemoryMiddleware`定义在middleware基类中，提供统一的生命周期hook接口：`on_reply`（检索+写回）、`on_system_prompt`（注入工具说明）、`list_tools`（暴露记忆工具）。

### 2.3 ContextConfig（上下文压缩配置）

**导入路径**: `from agentscope.agent import Agent, ContextConfig`

**类定义**（pydantic.BaseModel）:

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `trigger_ratio` | `float` | `0.8` | 触发压缩的token比例阈值（0 < x < 0.9），超过 `trigger_ratio × context_size` 时压缩 |
| `reserve_ratio` | `float` | `0.1` | 压缩后保留的最近消息token比例（必须 < trigger_ratio） |
| `compression_prompt` | `str` | 内置系统提示 | 引导模型生成结构化摘要的prompt |
| `summary_template` | `str` | 内置模板 | 将摘要拼回上下文时的字符串模板，包含 `{task_overview}`/`{current_state}`/`{important_discoveries}`/`{next_steps}`/`{context_to_preserve}` 占位符 |
| `summary_schema` | `dict` | `SummarySchema.model_json_schema()` | 约束模型结构化摘要输出的JSON Schema |
| `tool_result_limit` | `int` | `50000` | 单条工具结果最大token数，超限截断 |

**GerClaw医疗场景推荐配置**:
```python
medical_context_config = ContextConfig(
    trigger_ratio=0.7,           # 较早触发压缩，给医疗推理留更多空间
    reserve_ratio=0.15,          # 保留15%最近消息（当前就诊对话）
    tool_result_limit=3000,      # 检验结果/病历文本较易超长，适当降低
    summary_template="""<system-info>以下是之前就诊对话的医疗摘要：
# 就诊概述
{task_overview}

# 当前健康状态
{current_state}

# 重要医疗发现（过敏/用药/异常指标）
{important_discoveries}

# 后续诊疗计划
{next_steps}

# 需要保留的患者信息
{context_to_preserve}</system-info>""",
)
```

### 2.4 Agent.state.summary（压缩摘要）

压缩完成后，结构化摘要存储在 `Agent.state.summary` 字段中，类型为 `str | list[TextBlock | DataBlock]`。注入LLM时位于system message之后、context之前，格式为summary_template渲染结果，包裹在`<system-info>...</system-info>`标签中。

默认SummarySchema包含五个字段：
- `task_overview`: 用户核心请求和成功标准
- `current_state`: 已完成的工作、关键产出
- `important_discoveries`: 技术约束、决策理由、错误及解决方案
- `next_steps`: 需要完成的具体行动、阻塞项
- `context_to_preserve`: 用户偏好、领域细节、对用户的承诺

**医疗适配**: 可通过自定义`summary_schema`增加`allergies`、`current_medications`、`vital_signs`、`pending_followups`等医疗字段。

### 2.5 手动触发压缩

```python
# 使用agent默认配置
await agent.compress_context()

# 传入自定义配置（更激进压缩）
await agent.compress_context(
    context_config=ContextConfig(trigger_ratio=0.5, reserve_ratio=0.1),
    instructions=HintBlock(hint="务必保留患者的过敏史和当前用药信息", source="system"),
)
```

压缩流程：
1. 计算token总数（system_prompt + summary + context + tool schema）
2. 若超过 `trigger_ratio × context_size`，触发压缩
3. 切分消息：较早消息待压缩，`reserve_ratio × context_size`内的最近消息保留
4. 工具调用/结果对保持成对不拆分
5. 调用模型生成结构化摘要
6. 摘要替换被压缩消息，保留最近消息为新context

### 2.6 mem0配置项说明

**mem0 OSS (AsyncMemory) 关键配置**:

| 配置项 | 说明 | GerClaw推荐 |
|--------|------|-------------|
| `vector_store.provider` | 向量存储后端 | `"qdrant"`（Mem0官方推荐默认） |
| `vector_store.config.path` | 本地Qdrant存储路径 | `"/data/gerclaw/qdrant"`（生产）/`":memory:"`（测试） |
| `vector_store.config.embedding_model_dims` | Embedding维度 | 须与embedding_model一致；默认Qdrant期望1536 |
| `vector_store.config.on_disk` | 是否持久化到磁盘 | `True`（生产） |

| `embedder.config` | Embedding配置 | 通过AgentScope adapter自动填充 |

**Mem0 Platform (AsyncMemoryClient) 配置**:
- `api_key`: mem0平台API Key（`m0-...`开头）
- 托管版无需配置vector store，由mem0平台管理

### 2.7 跨会话记忆检索机制

```
新会话reply触发
    ↓
Mem0Middleware.on_reply（ReplyStartEvent时）
    ↓
提取用户query → mem0.search(query, filters={user_id, agent_id?}, top_k=5)
    ↓
mem0内部：query embedding → 向量相似度+BM25+实体匹配（v3多信号融合）
    ↓
返回top_k记忆条目 → 包装为AssistantMsg(name="memory", content=[HintBlock(...)])
    ↓
注入到agent.state.context → LLM可见历史记忆
    ↓
reply结束 → on_reply finally块 → user+assistant消息对 → mem0.add(messages, user_id, agent_id)
    ↓
mem0自动LLM提取facts → 去重 → 存储到向量库
```

**写回降级策略** (`_async_add_with_fallback`):
1. 先尝试 `infer=True`（LLM提取facts）
2. 若返回空（mem0判定无记忆价值），降级为 `infer=False`（直接存原始文本）
3. 保证"add总是能保存点什么"

### 4.1 oss_demo.py 核心流程

`examples/long_term_memory/mem0/oss_demo.py` 演示了Mem0Middleware的标准用法：

1. **环境准备**: 从环境变量读取`DASHSCOPE_API_KEY`，重置本地Qdrant和history DB保证可复现
2. **模型初始化**: `DashScopeChatModel`（stream=False）+ `DashScopeEmbeddingModel`（dimensions=1536，必须匹配mem0 Qdrant默认）
3. **mem0配置**: `MemoryConfig(vector_store=VectorStoreConfig(provider="qdrant", config={...}))`
4. **Middleware创建**: `Mem0Middleware(user_id=..., chat_model=..., embedding_model=..., mem0_config=..., mode="both", top_k=5)`
5. **SESSION 1**: 创建Agent，发送包含用户偏好的消息，mem0自动提取并存储
6. **SESSION 2**: 创建全新Agent实例（空context），发送新问题，mem0自动检索之前存储的偏好

**关键观察点**:
- static路径：`[mem0 → context (static)]` 在ReplyStartEvent时注入记忆
- agent路径：模型可主动调用`search_memory`/`add_memory`
- SESSION 2是全新Agent、无对话历史，但mem0桥接跨session记忆

## 6. GerClaw适配要点

### 6.1 医疗记忆分层设计

基于GerClaw调研文档的L0-L4分层，在AgentScope中的落地方式：

| 层级 | 名称 | AgentScope实现 | 存储内容 | 生命周期 |
|------|------|---------------|---------|---------|
| L0 | 核心档案 | `system_prompt`中动态注入 | 患者基本信息、过敏红警、当前诊断、当前用药列表（~2K tokens） | 每次会话构建时注入 |
| L1 | 会话记忆 | `Agent.state.context` | 当前对话轮次上下文 | 会话内，受context window限制 |
| L2 | 情景记忆 | `Mem0Middleware` + Qdrant | 历史对话、症状描述、就诊摘要（语义检索） | 跨会话持久化 |
| L3 | 结构化记忆 | 外部PostgreSQL（独立于AgentScope） | 用药记录(FHIR)、检验结果、生命体征时序 | 永久，SQL精确查询 |
| L4 | 知识记忆 | `RAGMiddleware` + KnowledgeBase | 医学指南、药品说明书 | 永久，语义检索 |
| 压缩层 | 上下文摘要 | `ContextConfig` + `Agent.state.summary` | 多轮就诊压缩摘要 | 会话内，压缩后注入 |

**记忆分类（临床一致性三分法）**:
- **稳定事实（Stable Facts）**: 过敏史、确诊慢病、手术史 → mem0中独立存储，永不折叠，检索时最高权重
- **演变状态（Evolving State）**: 当前用药方案、症状严重程度、生命体征趋势 → 同一主题合并为一条，使用add_memory更新
- **重大事件（Significant Events）**: 急诊就诊、跌倒、药物调整 → 带日期独立存储，mem0 metadata标记event_type

### 6.2 推荐配置：Mem0Middleware + ContextConfig组合

```python
from agentscope.agent import Agent, ContextConfig
from agentscope.middleware import Mem0Middleware
from agentscope.tool import Toolkit
from mem0.configs.base import MemoryConfig
from mem0.vector_stores.configs import VectorStoreConfig

# 1. Mem0长期记忆配置
mem0_mw = Mem0Middleware(
    user_id=patient_id,                       # 患者唯一ID
    agent_id="gerclaw_medical_assistant",     # 医疗助手agent命名空间
    chat_model=chat_model,
    embedding_model=embedding_model,          # dimensions=1536 匹配Qdrant默认
    mem0_config=MemoryConfig(
        vector_store=VectorStoreConfig(
            provider="qdrant",
            config={
                "collection_name": "gerclaw_medical_memory",
                "path": "/data/gerclaw/qdrant",
                "embedding_model_dims": 1536,
                "on_disk": True,
            },
        ),
    ),
    mode="both",                              # 自动检索+Agent可主动调用工具
    top_k=5,
    scope_search_by_agent=False,              # 同一患者记忆在多个agent间共享
    await_write=True,                         # 医疗场景同步等待写入，确保不丢失
)

# 2. 上下文压缩配置（医疗定制）
medical_context_config = ContextConfig(
    trigger_ratio=0.7,
    reserve_ratio=0.15,
    tool_result_limit=3000,
    summary_template="""<system-info>以下是之前对话的医疗摘要：
# 就诊概述
{task_overview}
# 当前健康状态
{current_state}
# 重要医疗发现（过敏/用药/异常指标）
{important_discoveries}
# 后续诊疗计划
{next_steps}
# 需要保留的患者信息
{context_to_preserve}</system-info>""",
)

# 3. 组装Agent
agent = Agent(
    name="gerclaw_medical_assistant",
    system_prompt=build_medical_system_prompt(patient_profile),  # L0核心档案注入
    model=chat_model,
    toolkit=Toolkit(tools=await mem0_mw.list_tools()),
    middlewares=[mem0_mw],
    context_config=medical_context_config,
    offloader=LocalWorkspace(workdir="/data/gerclaw/workspace"),  # 压缩内容可追溯
)
```

### 6.3 风险与注意事项

**(1) 记忆错误风险**:
- mem0自动提取facts可能产生幻觉或错误关联
- **缓解**: 医疗关键信息（过敏、用药）在写入前由Agent通过`add_memory`工具主动确认，而非完全依赖static_control自动写回；对高危信息使用`agent_control`模式，由Agent推理后显式调用`add_memory`
- 建议在`add_memory`的`thinking`字段中记录推理过程以便审计

**(2) 过期用药信息需时间戳**:
- 用药方案会随时间变化，旧用药信息可能导致危险建议
- **缓解**: 所有写入mem0的医疗记忆必须包含时间戳metadata；检索时优先返回近期记忆；旧用药标记为"已停用"而非删除
- 示例：`add_memory(thinking="患者确认停用氨氯地平，改服缬沙坦", content=["2026-07-01: 患者停用氨氯地平（因脚踝水肿），开始服用缬沙坦80mg qd"])`

**(3) 过敏史安全红线**:
- 过敏信息检索优先级最高，不可因压缩丢失
- **缓解**: 过敏史作为L0 Core Memory始终注入system_prompt，不依赖mem0检索；同时在mem0中冗余存储作为双重保险
- 压缩时通过`instructions`参数（HintBlock）强制保留过敏信息

**(4) 上下文压缩丢失医疗细节**:
- compress_context生成的摘要可能遗漏关键检验数值
- **缓解**: 使用自定义`summary_template`增加医疗字段；配置`offloader`（LocalWorkspace）持久化压缩前的原始消息，Agent可通过Read工具回查；`trigger_ratio=0.7`较早触发压缩，避免极端情况下的信息丢失

**(5) 多角色信息输入冲突**:
- 患者自述、家属补充、医生记录可能矛盾
- **缓解**: mem0 metadata中标记`source`字段（patient/family/doctor），检索时医生记录权重最高；后续可实现FHIR对账引擎检测矛盾

**(6) Embedding维度匹配**:
- Mem0Middleware传入embedding_model时，dimensions必须与mem0 vector store配置一致（默认Qdrant期望1536）
- DashScope text-embedding-v4支持1024/1536等维度，创建embedding_model时指定`dimensions=1536`

## 7. 可运行示例指引

本模块提供2个可运行Python示例，位于 `agentscope-examples/01_medical_memory/` 目录：

### 7.1 mem0_longterm_memory.py

**功能**: 演示`LongTermMemoryMiddleware(Mem0)`集成，模拟张大爷的医疗问诊场景：
- 第一轮对话：张大爷告知过敏史（青霉素过敏）、慢病（高血压/糖尿病）、用药（氨氯地平/二甲双胍）
- 第二轮对话：新会话中询问"我平时吃什么药？需要注意什么？"，Agent通过mem0记住之前的信息

**运行前置条件**:
```bash
# 安装依赖（可选，无mem0时自动使用mock模式）
pip install agentscope[mem0]
# 设置环境变量
export DASHSCOPE_API_KEY="your-api-key"
```

**运行命令**:
```bash
cd agentscope-examples/01_medical_memory
python mem0_longterm_memory.py
```

**无mem0依赖时**: 脚本内置MockMem0Middleware，模拟mem0的存储和检索接口，无需安装mem0ai即可运行验证流程逻辑。

### 7.2 context_config_demo.py

**功能**: 演示`ContextConfig`配置和上下文压缩功能，不依赖mem0：
- 配置`trigger_ratio=0.7, reserve_ratio=0.15`，自定义医疗summary_template
- 模拟多轮就诊记录（高血压随访、糖尿病复查、感冒就诊等）
- 使用MockModel触发压缩，展示医疗历史压缩为结构化摘要的过程
- 验证压缩后`agent.state.summary`包含医疗关键字段，最近消息保留在context中

**运行前置条件**:
```bash
# 仅需agentscope核心包
pip install agentscope
export DASHSCOPE_API_KEY="your-api-key"
```

**运行命令**:
```bash
cd agentscope-examples/01_medical_memory
python context_config_demo.py
```

**注意**: 此示例使用MockModel模拟LLM响应，无需实际API调用即可观察压缩流程和数据结构。如需使用真实模型，将MockModel替换为DashScopeChatModel即可。
