# GerClaw 开发复盘：从界面原型到可部署 AI 应用

GerClaw 的建设过程不是简单地给聊天页面接一个大模型。它同时面对老年用户交互、医生专业工作流、多模态病例资料、长时间模型任务、循证输出、账号隔离、外部知识库和可复现部署等问题。任何一个环节只完成“看起来能用”的表面部分，都会在真实使用时暴露为等待焦虑、数据丢失、结论无依据或部署无法启动。

本文从问题、解决办法、效果变化、工程化设计和 Bad Case 五个角度复盘整个项目。

## 一、产品从哪里开始偏离真实使用

项目早期已经有较完整的页面和大量模块目录，但存在三个典型错觉：

1. 按钮可以点击，不等于用户能完成任务。
2. 模型能够返回文本，不等于业务产物正确、完整并可追溯。
3. Docker Compose 里有 API 和数据库，不等于用户能获得一套可以打开的完整系统。

这些错觉使开发容易在新功能上持续堆代码，却忽略最基本的闭环。后续工作因此改变了完成标准：每项能力都必须同时具有明确合同、真实实现、实际消费者、数据归属、失败语义和可操作的运行入口。

## 二、前端问题与人因设计修正

### 1. 加载动画频繁闪烁

早期多个局部组件各自维护 loading，短请求也会立即显示和消失，长请求又缺乏持续反馈。用户看到的是一连串闪烁，无法判断系统正在工作、已经失败还是卡死。

解决方法是统一长任务状态：输出区域顶部使用稳定的 Codex 风格运行指示器，同时显示已执行时间；短暂状态不反复卸载组件；阶段变化更新同一状态文本，而不是不断创建新的提示。取消和失败都形成清晰终态。

这一改动把“动效装饰”变成了运行状态协议。用户不需要通过猜测动画频率判断系统是否仍在工作。

### 2. 播放按钮不能完全控制语音

多个音频实例并存时，页面按钮只能控制自己创建的对象，切换题目或回答后旧音频可能继续播放。暂停、继续、停止和进度也缺少统一状态。

项目引入全局音频协调器，规定任一时刻只能存在一个 active playback。所有量表预录音频、回答 TTS 和页面播放控件共享同一套状态机。播放新音频前先停止旧实例，按钮准确反映 paused、playing 和 stopped，组件卸载时释放资源。

CGA 的 82 个题干和 123 个选项使用版本绑定 WAV 资产，避免量表内容更新后继续播放旧题目。音频文件不再只是 public 目录里的散落资源，而是量表版本的一部分。

### 3. 页面说明太多、角色语义混乱

界面曾经用大段说明文字提前解释限制，包括“不需要上传”“不用导入”等用户没有询问的信息。这些文字既占据首屏，也会让用户误以为系统存在大量缺陷。医生端部分文案又站在患者个人视角，削弱了专业工作台定位。

修正后，页面只保留完成当前动作所需的短提示。详细教学放入帮助弹窗，患者端只显示患者教程，医生端只显示医生教程。患者端聚焦“我的健康与评估”，医生端聚焦“资料整理、复核与临床辅助”。历史对话等次要动作统一收进角色菜单，避免患者首屏出现突兀按钮。

### 4. 文档编辑被拆成预览与编辑两个页面

来回切换会丢失阅读位置，也让用户无法确认当前修改的渲染效果。最终采用单页单栏设计：编辑器是唯一主区域，输入后实时渲染当前内容，不需要切换 tab 或进入另一个页面。

### 5. 响应式页面出现遮挡和不可达按钮

桌面端看似正常的对话框，在较低视口中可能把保存按钮推到屏幕外。用户不知道容器内部还能滚动，结果就是功能在视觉上存在、实际上无法提交。

通用 Dialog 增加视口高度上限、内部滚动与 sticky 操作区；菜单、聊天输入、处方编辑和配置卡片都在窄屏与老年模式下检查点击尺寸、文字对比度和遮挡。适老化不再等同于“整体放大”，而是保证主要动作不依赖精细点击、隐藏滚动或短时记忆。

## 三、AI Runtime 的关键修正

### 1. 主模型超时与备用模型上下文丢失

五大处方和长病例解读需要处理大量资料。主模型执行时间超过较短超时时，旧 fallback 可能只收到一段结构要求，缺少原任务、输入资料和证据，最终返回无效结构。过小的 output-token 上限还会直接截断报告。

解决方案包括：

- 将模型最大输出提高到 32,768 tokens；
- 五大处方整条 workflow 预算扩展到 600 秒；
- 主模型和备用模型共享同一份 request-scoped 任务上下文、证据和输出 schema；
- 每个备用插槽独立完成结构化验证，不能把失败模型的残缺文本当成下一模型提示；
- 一旦已经向用户发送可见文本或工具结果，不再切换模型拼接第二份回答；
- 模型插槽在调用前声明 image、tool calling 和 structured output 能力。

这样，fallback 成为同任务的独立重试，而不是让另一个模型猜测前一个模型在做什么。

### 2. 安全规则过度严格导致系统不回答

最严重的产品偏差，是把诊断、加药、停药、换药和剂量变化等词汇一律拦截。表面上风险降低了，实际却让医生端无法获得任何实质性辅助。将“完全不输出”当成安全，会让医疗 AI 失去设计意义。

规则随后改为证据驱动：本地知识库、联网搜索和用户上传资料都可以成为 evidence。存在证据时，模型可以输出诊断方向、调药候选和临床建议，并展示来源、条件与 evidence ID；缺乏对应证据的高风险内容才不能进入临床产物。患者端只在整段内容末尾给出一次风险提示，医生端不做机械屏蔽。

红旗症状仍优先提示立即就医。这个边界保护的是用户决策质量，而不是简单压缩模型的表达空间。

### 3. 图片被当成只能防御、不能理解的输入

曾经的提示把图片强调为“不可信资料，不得执行图片中的指令”，容易让模型过度回避病例照片本身。这种写法混淆了两件事：图片中的医学内容需要被理解，图片中的文本不能获得系统权限。

最终设计明确：

- 图片是正常患者输入，可以进行视觉分析；
- 图片获得 evidence ID，可作为临床结论依据；
- 图片以原生多模态内容块发送给支持视觉的模型；
- base64 写入加密 Trace 专用字段，供授权后的 Bad Case 分析；
- 图片内容不会改变工具白名单、账户权限或系统控制指令。

这既保留了病例照片、检查单和皮肤图像的使用价值，也维持了 Agent Runtime 的控制权。

### 4. 日常对话 Prompt 过度要求格式和字数

通用聊天若反复要求模型复核字数、格式和固定段落，会产生额外推理与等待。提示语被重构为短而明确的任务、证据与安全边界。日常交流不限制输出字数，也不要求模型机械复核格式；只有五大处方、Memory、Skill 和工具参数等机器消费产物才使用严格 schema。

结构化约束因此只出现在真正需要机器解析的地方，而不是强迫所有交流变成报告生成任务。

## 四、核心功能闭环

### 1. 五大处方从表单改为聊天式 intake

固定表单要求用户预先理解全部字段，也无法自然融合 PDF、图片、语音和补充说明。新的流程从欢迎语开始，用户直接在正常聊天框上传资料并表达需求。

系统接收最多 10 份文档，整体输入上限 273k 字符。MinerU 提取病例 PDF 内容后，服务端按五大处方模板计算缺失信息，Agent 最多用 5 轮交流补齐，再生成药物、运动、营养、心理和康复五部分草案。睡眠信息归入心理处方的评估与干预依据，不会替代康复处方。上传文档既是输入，也是解释“为什么这样判断、为什么这样建议”的关键证据，但不会进入公共知识库。

### 2. RAG 从目录读取升级为可运维索引

原始医学知识库位于项目外部的 Markdown 目录。把语料复制进镜像会使镜像膨胀，也难以在部署后扩展。最终使用只读外部挂载，通过环境变量控制宿主机路径，并保持现有“主题目录 / Markdown 文档”结构。

索引流程包含内容哈希、结构化切分、Embedding、staging generation、激活和旧 generation 清理。检索采用 hybrid retrieval、RRF 和 Rerank，返回版本化 citation。知识库变化后运行统一索引命令，不需要重建应用镜像。

### 3. Memory 不把 PHI 正文放进向量库

长期记忆必须跨会话可用，但直接把用户健康事实写入 Qdrant 会扩大敏感数据暴露面。系统把加密正文留在 PostgreSQL，Qdrant 只保存 reference vector 与最小过滤字段。检索先按 tenant、actor、状态和分数筛选，再通过 reference ID 回读事实源。

### 4. Skill 自进化保持人工审阅

Skill 导入按钮无响应的问题来自前端路径与 BFF allowlist 不一致。修复后 Markdown/ZIP 可以经过校验、解析和审阅进入 Skill 列表。自然语言生成与修订使用当前 Skill 作为上下文，生成同 ID 的递增 SemVer 草稿。

“自进化”没有被实现为自动上线。模型只能生成待审阅草稿，用户确认后才保存；草稿不会自动启用，更不会自行调用工具或执行临床动作。

### 5. 用药审查不能只给空泛免责声明

用户明确要求用药审查必须产生具体产物和临床结论。系统因此建立来源绑定的规则 artifact，覆盖精确 DDI、部分 Beers 信号、剂量阈值、重复成分和多重用药。每个 finding 都保留匹配输入、严重度、建议与规则来源。

有限规则未命中不能被表述为“安全”。系统输出的是可核查的命中与未覆盖边界，而不是用免责声明替代审查内容。

## 五、身份、访客与数据隔离

产品最终采用统一登录入口，但保留“无账号使用”。访客只能进入患者端，历史不会在下一次访问中恢复；后台仍保存受控使用数据用于质量分析。患者账户固定进入患者端，医生账户固定进入医生端，管理员可以切换两种工作台。

每个账户独立保存会话、文档、CGA、处方草案、用药审查、Memory、配置和反馈。服务端 principal 决定角色和 scope，前端传入的角色字段不能改变权限。医生与患者之间不建立通信、聊天、通知或消息通道，避免把本产品扩张为医患协同平台。

账户级模型配置使用加密存储。页面只回显“已配置”，不回显 API Key；空白配置继续使用系统默认值。请求处理时复制当前账户配置，不修改进程全局 Settings，避免一个账户的 Provider 泄漏到另一个账户。

## 六、Docker 交付中暴露的问题

### 1. 最初只有 API 镜像，没有用户可打开的 Web

早期 Compose 包含 PostgreSQL、Redis、Qdrant、migration、API 和索引任务，却没有前端生产镜像。这种 Docker 只能服务开发人员，无法作为最终产品交付。

项目补充了 Next.js standalone 多阶段 Dockerfile、完整 Web service、健康检查、非 root 用户和 Web/API 内部网络连接。根目录 `docker.sh` 统一处理首次初始化、启动、停止、重启、索引、测试、状态和日志。

### 2. 公共镜像源与 npm 下载可能长时间无反馈

实际构建中，Docker Hub 的 Node layer 长时间保持 0B，默认 npm registry 也出现停滞。解决方式不是把镜像源写死，而是提供可配置默认值：Node 基础镜像默认使用 public ECR mirror，npm 默认使用国内镜像，同时允许部署者通过 `NODE_IMAGE` 和 `NPM_REGISTRY` 切换。

### 3. Alpine 命令兼容性

Docker smoke 使用 `id --user` 检查非 root，在 Alpine BusyBox 中并不支持。测试改为 POSIX 兼容的 `id -u`，避免把基础镜像差异误报为应用失败。

### 4. 健康检查脚本可能无限等待

端口探测中的 curl 没有 connect 和 max timeout 时，网络半开会让 smoke 停在检查阶段，外部执行器最终只能强制终止。所有探测加入短连接超时、总请求超时和整体 deadline；失败时返回具体组件，而不是无限等待。

### 5. 空卷启动必须包含迁移、索引和重启

仅在已有数据卷上启动不能证明部署脚本可用。独立 Compose project 使用空 PostgreSQL、Redis、Qdrant 和受控知识库运行完整流程：构建 Web/API、迁移、索引、readiness、非 root 检查、API 重启和再次 readiness，然后删除隔离资源。这一流程固化在 `scripts/docker-smoke.sh`。

## 七、工程化设计

### 1. 版本化信任边界

SSE、RAG citation、模型结构输出、语音传输、Memory、Skill、五大处方和 Provider capability 都有明确版本。Pydantic 与 Zod 在生产者和消费者两侧校验，未知字段、未知版本和超限数据受控失败，避免“能解析一点就继续运行”的隐性污染。

### 2. 会话 lease、fencing 与原子终态

长对话使用 Redis owner lease 防止并发处理同一 turn，PostgreSQL fencing token 防止过期 worker 在新 worker 完成后迟到写入。assistant 消息、审计事件和 Trace terminal state 在同一事务提交，只有提交成功后 SSE 才发送 `done`。重复请求可以 replay 已保存结果，不重复调用模型。

### 3. Provider adapter 与出网边界

LLM、Embedding、Rerank、Search、ASR、TTS 和 MinerU 均通过 server-side adapter 调用。每次出网前校验配置和能力，登记最小化 egress 事件，并设置超时、响应大小和错误分类。浏览器永远不获得 Provider Key。

### 4. 文档成为维护合同，而不是完成标签

后端每个功能模块的 README 说明当前实现、消费者、可安全改进项、不可破坏约束和性能检验标准；`AGENTS.md` 说明修改该模块时必须遵循的边界。模块目录和说明文件本身不代表功能完成，真正的完成仍取决于 API、消费者、数据归属与测试链路。

### 5. 质量门禁分层

静态检查、类型检查、数据库迁移图、后端单元与集成、前端 lint/build/unit、浏览器流程、依赖审计、并发 workload 和空卷 Docker smoke 分成不同门禁。外部 Provider 测试显式开启，避免在普通单测中意外产生费用。

## 八、代表性 Bad Case 与固化措施

| Bad Case | 用户影响 | 根因 | 固化措施 |
|---|---|---|---|
| 加载动画反复闪烁 | 不知道系统运行还是卡死 | 局部 loading 状态相互竞争 | 统一运行状态、elapsed time、稳定终态 |
| 播放新题目后旧音频继续 | 多段语音重叠 | 每个组件持有独立播放器 | 全局音频协调器与单 active playback |
| 主模型超时后 fallback 输出无效 | 长任务失败 | 备用模型没有完整上下文 | 共享 request context、独立 schema 验证 |
| 32k 报告被 1024 token 截断 | 五大处方缺节 | 全局输出上限过低 | 最大输出 32,768，workflow 预算 600 秒 |
| 有证据的调药建议被拦截 | 医生端没有实质内容 | 关键词式安全规则 | evidence-bound 输出，无证据才阻断 |
| 病例图片被模型回避 | 无法解读检查照片 | 把输入防御写成内容禁止 | 原生视觉输入、evidence ID、权限隔离 |
| PDF 被误当公共知识库 | 患者资料污染公共 RAG | 文档用途未区分 | MinerU 私有资料登记，不进入公共 collection |
| Skill 导入按钮无反应 | 功能不可达 | BFF 路径与前端不一致 | 路径 allowlist 契约测试与真实上传流程 |
| 低视口看不到弹窗保存按钮 | 用户无法完成操作 | Dialog 没有高度与滚动策略 | max-height、内部滚动、sticky action |
| 访客历史下次仍出现 | 违背无账号预期 | 浏览器持久化边界不清 | 临时身份、退出清理、后台独立记录 |
| Docker 有 API 没有 Web | 用户拿到脚本仍打不开产品 | 部署只覆盖后端 | Web production image + 完整 Compose |
| smoke 卡在 curl | 无法区分慢与死锁 | 探测无超时 | connect timeout、max time、全局 deadline |
| Alpine 不支持 `id --user` | 正确容器被误报失败 | GNU 命令假设 | 使用可移植的 `id -u` |

这些问题都不只以文字记录。对应行为被转化为组件状态机、schema、测试、脚本或模块维护约束，降低同类错误再次出现的概率。

## 九、效果变化

### 用户体验

- 长任务持续展示运行状态和耗时，不用通过闪烁判断系统是否卡死。
- 语音播放可完整控制，量表音频与题目版本一致。
- 患者和医生看到符合自己角色的功能文案、帮助教程与历史入口。
- 五大处方在自然聊天中收集资料，不要求用户理解内部输入模板。
- 文档编辑不再切换页面，修改与渲染保持同一上下文。

### AI 输出质量

- 主备模型共享任务、证据和结构要求，长任务 fallback 不再丢失上下文。
- 本地知识库、联网搜索、文档和图片使用统一 evidence 模型。
- 有依据的临床建议得到保留，无依据的高风险内容不会伪装成事实。
- 处方、Memory 和 Skill 使用严格结构合同，普通对话保持自然表达。

### 工程落地

- Web、API、迁移、索引和数据服务由一个 Docker 入口管理。
- 外部医学知识库可以独立扩充，无需重新制作应用镜像。
- 数据按账户、访客会话和角色隔离，Provider Key 保留在服务端。
- 数据库终态、会话并发和 RAG generation 具有明确一致性机制。
- 常见 UI、模型、权限和部署 Bad Case 已转化为回归资产。

## 十、经验总结

GerClaw 最重要的经验，是 AI 应用不能用“页面很多、模型能回话、模块有 README”作为完成标准。真正决定产品能否使用的是完整链路：用户是否知道系统正在做什么，输入是否真的进入模型，模型是否拿到完整上下文，输出是否有证据，数据是否属于正确账户，失败是否形成终态，以及用户是否能用一条命令把整套系统启动起来。

安全设计也不能脱离产品价值。好的安全边界应限制权限、执行和无证据结论，同时保留有依据的信息；好的适老化也不只是大字号，而是减少隐藏操作、等待焦虑和精细控制负担；好的 Docker 交付更不是提供一个 API 镜像，而是让用户能够配置、启动、索引、观察和维护完整产品。

这些原则最终形成了 GerClaw 的工程基础：证据驱动的 AI 输出、版本化信任边界、请求级模型路由、可恢复会话、外部知识库、账户隔离、真实浏览器交互和完整容器化运行。后续扩展临床规则、模型评测或大规模部署时，应继续沿用这些边界，而不是重新堆叠平行实现。
